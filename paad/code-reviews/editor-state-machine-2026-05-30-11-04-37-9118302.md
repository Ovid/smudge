## Suggestions

- **[S1] Imperative-disable / declarative-re-enable asymmetry is load-bearing but invisible** (`packages/client/src/hooks/useEditorMutation.ts:140-142` + `packages/client/src/pages/EditorPage.tsx:448-450`). Lock-down `setEditable(false)` is synchronous; re-enable only fires if the reconcile effect's `state.editable` dep changes value across two committed renders. Safe today only because every proceed-past-lockdown path has an `await` between `MUTATION_STARTED` and the terminal `finally` dispatch, forcing an intermediate `editable:false` commit. A future same-tick terminal dispatch would let React batch both, skip the reconcile render, and leave TipTap synchronously disabled with no banner. Consider re-enabling imperatively at the settle site (symmetry) or adding a real EditorPage-level no-await regression test; the hook's event-log-reducing unit tests cannot catch a missing reconcile render. *(Error Handling, `claude-opus-4-8[1m]`, Medium)*
- **[S2] Transient `editable:false` pushed into a freshly-mounted editor on a reload-key bump** (`packages/client/src/pages/EditorPage.tsx:448-450` vs `:435-443`). When `chapterReloadKey` bumps before the hook's `finally` dispatches `RELOADED`, the reconcile effect reads the still-`false` `state.editable` and disables the brand-new editor for one commit, self-healing on the next. Only ever disables (never wrongly enables), no data-loss â€” a coupling foot-gun, not a defect. Optionally treat a `chapterReloadKey`/`activeChapter.id` change as "remount â†’ assert true" so a stale `false` can't be applied on the remount commit. *(Concurrency, `claude-opus-4-8[1m]`, Medium)*
- **[S3] Terminal-save-error lock path no longer locks the editor synchronously** (`packages/client/src/pages/EditorPage.tsx:295-306`). On `main`, `applyReloadFailedLock` ran `safeSetEditable(false)` synchronously; now it only dispatches `COMMITTED_UNRELOADED`, so the `useProjectEditor.onRequestEditorLock` path (which does not pass through `useEditorMutation.run()`, so no prior lock-down ran) leaves the editor TipTap-writable for one render tick. This is **ratified by the plan** (`docs/plans/2026-05-29-editor-state-machine-plan.md:634` routes this path through the effect deliberately) and well-backstopped (1.5s save debounce + `handleSaveLockGated` no-op once `isLocked()` flips in the same commit), so it does not violate Decided Q3 (which scopes synchronous lock-down to the mutation paths only). Worth an inline comment noting the path is intentionally effect-driven. *(Spec Compliance, `claude-opus-4-8[1m]`, Medium)*
- **[S4] `machine.busy` is unconsumed dead state; `isBusy()` / `getState()` exports have no production callers** (`packages/client/src/hooks/useEditorMutationMachine.ts:12-22, 90-92, 108`). Every production busy gate reads `mutation.isBusy()` â†’ `inFlightRef`. This is documented as intentional (testable mirror, future-phase wiring) and ratified by the plan, so it is not a bug â€” but the unused exports and the desync-capable field are latent foot-guns. Either drop the unused `getState()`/`isBusy()` or leave them per the documented future-phase plan; the design-doc "single source of truth" prose is forward-looking, not a contract this PR breaks. *(Spec Compliance, `claude-opus-4-8[1m]`, Medium)*
