## Executive Summary

This branch lands Tasks 1‚Äì3 of the 15-task Phase 4c.1 inline-notes plan: the pure
`stripNoteMarks` / `extractNotes` JSON walkers, the `note` TipTap mark, and its
highlight CSS. **It must not merge to `main` as-is.** Commit `5cbbc25` registers
`NoteMark` in the shared `editorExtensions` array ‚Äî the array consumed by the
**server export renderer**, the client preview, and snapshot view ‚Äî while
`stripNoteMarks()` is wired into **zero production call sites**. The result is
that a chapter carrying a note mark exports its private note text verbatim into
HTML (`<span data-note="SECRET">`) and its note *positions* into EPUB.

The aggravating detail, verified by repro: before this branch, a stored `note`
mark made `generateHTML` **throw** ("no mark type note in this schema"), which
`export.renderers.ts` catches and turns into an empty chapter. Registering the
mark is precisely what converts that **fail-closed throw into a successful leak**.
Task 3 did not merely stage a leak ‚Äî it opened one. Four separate comments in the
diff assert the strip invariant as accomplished fact, and one cites a CLAUDE.md
section that does not exist.

Confidence is high: every finding below was reproduced by executing the real
renderers, walkers, and schema ‚Äî not inferred from reading. Six of seven
specialists independently reported the export leak.

The good news: the leak is **latent, not live**. No UI can create a note yet
(Tasks 8/9/12 are unlanded), so no note exists in any document today. The fix is
small and the branch is otherwise clean, well-tested, TDD-shaped, and scope-
compliant.

## Critical Issues

### [C1] Note text leaks into HTML export; note positions leak into EPUB
- **File:** `packages/server/src/export/export.renderers.ts:60` (and `packages/server/src/export/epub.renderer.ts:67`, `packages/shared/src/editorExtensions.ts:31`)
- **Bug:** `chapterContentToHtml` calls `generateHTML(content, editorExtensions)` with no
  `stripNoteMarks()`. `NoteMark` is now in `editorExtensions`, so the mark serializes
  straight through. Verified output:
  `<p><span data-note="SECRET" class="note-highlight">Marcus drew his sword</span></p>`
- **Impact:** The entire security property of this feature is that notes are private
  authorial commentary ("cut this, the publisher will hate it"). HTML is the format you
  hand to a beta reader. Verified per-format, by running all five renderers:

  | Format | Note **text** | Note **position** (`note-highlight` class) |
  |---|---|---|
  | **HTML** | **LEAKS** | **LEAKS** |
  | **EPUB** | clean (`epub-gen-memory` strips `data-*`) | **LEAKS** into `OEBPS/0_Ch1.xhtml` |
  | Markdown | clean (turndown drops the span) | clean |
  | Plain text | clean (`stripHtmlTags`) | clean |
  | DOCX | clean (`marksToProps` ignores unknown marks) | clean |

  The EPUB case still discloses *which ranges the author flagged as problematic* ‚Äî most
  of the confidentiality value, even without the text.
- **Suggested fix:** Land plan Tasks 5 & 6 in this branch. Better: introduce one chokepoint
  rather than three hand-placed calls (see [I3] ‚Äî there are **three** `generateHTML` sites,
  not the two the plan names). If the phase boundary forbids that, then **do not add
  `NoteMark` to `editorExtensions` yet** ‚Äî Tasks 1‚Äì2 (the pure helpers) are perfectly safe
  to land alone.
- **Confidence:** High (100 ‚Äî reproduced)
- **Found by:** Logic & Correctness, Error Handling, Contract & Integration, Concurrency & State, Security, Plan Alignment (6 of 7)

### [C2] Registering the mark converts an existing fail-closed throw into a successful leak
- **File:** `packages/shared/src/editorExtensions.ts:31`
- **Bug:** Verified by repro: `generateHTML(docWithNoteMark, [StarterKit, Heading, Image])` ‚Äî
  the pre-branch extension list ‚Äî **throws** `There is no mark type note in this schema`.
  `export.renderers.ts:61-64` catches that, logs a warning, and returns `""`.
- **Impact:** A note-bearing doc used to fail closed (empty chapter + a logged warning).
  After this commit it renders successfully, private text included. This is why [C1] is
  Critical rather than merely "an unlanded task": the branch actively removed a safety net
  that was there by accident. It also means the leak is reachable without any UI ‚Äî
  `TipTapDocSchema` (`packages/shared/src/schemas.ts:48-56`) is `.passthrough()` with no
  mark-type allowlist, so a hand-crafted `PATCH /api/chapters/{id}` persists a note mark,
  and `NoteMark.parseHTML` matching `span[data-note]` means a crafted clipboard paste
  creates one in the live editor.
- **Suggested fix:** Same as [C1]. Sequence the strip *before or with* the registration,
  never after.
- **Confidence:** High (95 ‚Äî reproduced)
- **Found by:** Plan Alignment (originating), confirmed by Verifier

## Important Issues

### [I1] Find-and-replace across a note boundary silently erases the note and its text
- **File:** `packages/shared/src/tiptap-text.ts:745`
- **Bug:** The replacement node inherits marks via `marksAtOffset(segments, match.start)` ‚Äî
  marks at the match's **start offset only**. A match that starts *outside* a noted range and
  extends *into* it drops the note mark entirely. Reproduced:
  ```
  in : "He drew his " + note("check the weapon")["sword"] + " again."
  op : replace "his sword" ‚Üí "her blade"
  out: [{"type":"text","text":"He drew her blade again."}]      ‚Üê no marks at all
  ```
  The note and its user-authored body are gone; `extractNotes` returns `[]`.
- **Impact:** Qualitatively different from the same behavior on `bold`, because a note's
  `attrs.text` is **user content, not formatting**. Project-wide replace is a server-committed
  mutation followed by a reload, so there is no editor undo and the draft cache is cleared
  after server success (save-pipeline invariant 3). The note is unrecoverable except by
  snapshot restore.
- **Suggested fix:** Union the note marks spanning `[match.start, match.end)` onto the
  replacement node, or have the find-replace service refuse/warn on a replace that partially
  overlaps a `note` mark. **Also rewrite plan Task 4** ‚Äî as specified
  (`docs/plans/2026-07-12-notes-plan.md:212`) it pins only the match-starts-*inside* case,
  which passes. Task 4 would go green while certifying the one case that was never at risk.
- **Confidence:** High (95 ‚Äî reproduced)
- **Found by:** Concurrency & State

### [I2] `strip()` has no null/type guard ‚Äî throws on malformed docs that pass Zod
- **File:** `packages/shared/src/tiptap-notes.ts:44`
- **Bug:** `strip()` lacks the `!node || typeof node !== "object"` guard that its own sibling
  `collect()` has (line 68), and that `collectLeafBlocks` (`tiptap-text.ts:79`) and
  `validateTipTapDepth` have. `tiptap-text.ts:70-76` documents exactly why the guard exists:
  *"walkers can't rely on the TipTap schema because chapters read from the DB bypass Zod."*
  Verified ‚Äî these all pass `TipTapDocSchema.safeParse` (nested children are only
  `z.record(z.unknown())`, so nested shape is unvalidated) and then throw:

  | doc (all pass Zod) | `stripNoteMarks` |
  |---|---|
  | `content:[{paragraph, content:[null]}]` | throws `Cannot read properties of null (reading 'marks')` |
  | nested `marks: {}` (non-array) | throws `node.marks.filter is not a function` |
  | nested `content: "oops"` | throws `node.content.map is not a function` |
  | nested `content: ["oops"]` | **silently corrupts** ‚Üí `{"0":"o","1":"o","2":"p","3":"s"}` |

  `collect()` guards `content` but **not** `marks`, so `extractNotes` throws on a non-array
  `marks` too.
- **Impact:** Once Task 5 wires `stripNoteMarks` into `chapterContentToHtml`, the throw lands
  inside the existing `try/catch` (`export.renderers.ts:61-64`) ‚Üí returns `""` ‚Üí **the whole
  chapter exports empty**, from one corrupt node. Silent content loss.
- **Suggested fix:** One-line guard in `strip()`; `Array.isArray(node.marks)` in both walkers.
  Add a malformed-doc test ‚Äî none exists.
- **Confidence:** High (95 ‚Äî reproduced)
- **Found by:** Logic & Correctness, Error Handling

### [I3] `extractNotes` returns N duplicate entries for one note spanning N text nodes
- **File:** `packages/shared/src/tiptap-notes.ts:67-77`
- **Bug:** `collect()` pushes one `ExtractedNote` per **text node**, not per contiguous marked
  range. A note spanning mixed formatting ‚Äî which is how ProseMirror actually stores
  "Marcus **drew** his sword" ‚Äî yields:
  ```json
  [{"note":"rename?","excerpt":"Marcus "},
   {"note":"rename?","excerpt":"drew"},
   {"note":"rename?","excerpt":" his sword"}]
  ```
- **Impact:** The Task-11 Notes panel would render the same note three times with fragmented
  excerpts. `extractNotes` cannot answer "how many notes are in this doc" ‚Äî the exact question
  the panel asks. Invisible in the suite because every fixture puts a note on a single whole
  text node.
- **Suggested fix:** Coalesce adjacent text nodes carrying an equal note mark. Fix this
  **before the panel is built** ‚Äî the panel's data model rests on it.
- **Confidence:** High (90 ‚Äî reproduced)
- **Found by:** Logic & Correctness

### [I4] The depth cap fails *open* in `strip()` and *closed* in `extractNotes()`
- **File:** `packages/shared/src/tiptap-notes.ts:45`
- **Bug:** `if (depth > MAX_TIPTAP_DEPTH) return node;` returns the over-deep subtree
  **verbatim, note marks intact**. Every sibling walker fails closed (`collectLeafBlocks` ‚Üí `[]`,
  `validateTipTapDepth` ‚Üí `false`, `canonicalJSON` ‚Üí `"null"`, `content-hash` canonicalize
  *throws*). Verified on a 70-deep doc: the note text **survives** `stripNoteMarks` while
  `extractNotes` reports **0 notes**.
- **Impact:** The exact inverse of the stated invariant ‚Äî the panel shows the writer *no notes*,
  so they believe there are none, while the export *contains* them. `strip()` is the one walker
  whose failing open is a confidentiality leak. Not reachable via the API (Zod rejects >64), but
  reachable via a hand-edited DB or a restored backup ‚Äî a threat model this repo already defends
  against elsewhere (`sanitizer.ts`, `stripDisallowedImages`).
- **Suggested fix:** Strip the node's own marks at the cap, or drop the subtree ‚Äî fail closed.
  Both existing depth tests assert only `not.toThrow()` (`__tests__/tiptap-notes.test.ts:70`,
  `:126`); neither asserts the note was actually **removed**, which is why this passes.
- **Confidence:** High (88 ‚Äî reproduced)
- **Found by:** Logic & Correctness, Error Handling, Security, Contract & Integration

### [I5] Preview and snapshot view are safe only by accident, and the sanitizer's comment is now false
- **File:** `packages/client/src/sanitizer.ts:14-47` (also `packages/client/src/hooks/useSnapshotController.ts:49`)
- **Bug:** Two parts.
  1. **A third `generateHTML` call site the plan never mentions.** The census is
     `export.renderers.ts:60`, `PreviewMode.tsx:38`, and `useSnapshotController.ts:49`. Plan
     Tasks 5 & 6 name only the first two.
  2. Preview and snapshot view do not leak **today** ‚Äî but only because `span` is absent from
     `ALLOWED_TAGS` and DOMPurify's `KEEP_CONTENT` preserves the child text. That allowlist was
     written for a different threat model. Its own comment now reads false: *"generateHTML always
     produces HTML bounded by editorExtensions‚Ä¶ any tag outside that set is either a future
     extension we'll add deliberately or an attacker-controlled payload."* `editorExtensions` now
     legitimately emits a `<span>` that the sanitizer classifies as hostile.
- **Impact:** Verified: DOMPurify's `ALLOW_DATA_ATTR` defaults to `true`
  (`node_modules/dompurify/dist/purify.cjs.js:430`) and `sanitizeEditorHtml` does not set
  `SAFE_FOR_TEMPLATES`. So the day anyone adds `span` to `ALLOWED_TAGS` ‚Äî Phase 4c.3 tags, a
  Highlight extension, a Link extension ‚Äî `data-note` rides in for free and preview leaks
  instantly, **with no test to catch it**.
- **Suggested fix:** Strip at all three sites. Add a line to the `ALLOWED_TAGS` comment naming
  `span[data-note]` as deliberately excluded ‚Äî the exclusion is currently load-bearing and
  undocumented.
- **Confidence:** High (92 ‚Äî verified against the DOMPurify source)
- **Found by:** Security, Contract & Integration, Concurrency & State

## Suggestions

- **No extension-parity test.** `packages/shared/src/__tests__/editorExtensions.test.ts:49` asserts only rendered substrings ‚Äî no assertion on the array's length or membership, so `NoteMark` slid into the server-consumed extension list silently. A snapshot over `editorExtensions.map(e => e.name)` would have turned red at commit `5cbbc25` and forced the question "does this render into export?". This is the countermeasure CLAUDE.md already established as `editorEntryPointSurface.test.ts`. **Recommend adding it in this branch** ‚Äî it is what makes [C1] un-repeatable when Phase 4c.3's tag mark arrives. (Contract & Integration, confidence 100)
- **Four comments assert a strip invariant that does not hold**, and `packages/shared/src/noteMark.ts:10` cites *"CLAUDE.md ¬ßKey Architecture Decisions"* as its authority ‚Äî a section that contains nothing about notes (`grep` returns zero hits). The code cites documentation that does not exist to vouch for behavior that does not exist. Also at `tiptap-notes.ts:5`, `editorExtensions.ts:30`, `index.css:116`. Fix the comments (do not merely delete them) or land plan Task 15. (Security, Error Handling, Contract & Integration, confidence 100)
- **`NOTE_MARK_NAME` vs. hardcoded `"note"` ‚Äî adjudicated.** Logic wants `noteMark.ts:15` to import the constant; the over-engineering pass wants the constant deleted as unused. **The import is the right half:** the constant has zero importers *precisely because* `noteMark.ts` hardcodes the name. Importing it kills the drift by construction ‚Äî and a rename currently makes `stripNoteMarks` a silent no-op that ships notes into every export with **no failing test**. No cycle risk (`tiptap-notes.ts` imports only `tiptap-safety`). (confidence 85)
- **An empty-text note vanishes on any HTML round trip.** `noteMark.ts:25` omits `data-note` when the text is falsy, but `parseHTML` matches only `span[data-note]` ‚Äî so the mark is not re-parsed. The round trip is real: ProseMirror's clipboard serialization is HTML-based, so copy/paste *within the editor* loses the note. Fix is one character: emit the attribute unconditionally. Low severity only because no UI can create an empty note yet. (Logic, Error Handling, confidence 80)
- **Two adjacent notes with identical body text fuse into one** after find-and-replace creates adjacency (`cleanupTextNodes`/`marksEqual` compare structurally, and the design deliberately has no `id`). Verified. **Recommend documenting, not fixing** ‚Äî the information lost is a duplicate, and adding an `id` would reintroduce the identity model the design rejected. (Concurrency, confidence 90)
- **`strip()`'s "returns a new doc" is shallow** ‚Äî leaf `attrs` and surviving mark objects are shared by reference (`out.content[0].attrs === input.content[0].attrs`). The input genuinely is untouched, so this is a docblock fix at `tiptap-notes.ts:36`, not a code fix. (Logic, confidence 90)
- **`noteMark.test.ts:29`** re-renders the identical doc as the test above it; fold the one assertion up and delete the test.
- `index.css:125` adds `cursor: pointer`, advertising a click affordance that does not exist until Task 11.
