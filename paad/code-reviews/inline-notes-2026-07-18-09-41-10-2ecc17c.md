## Executive Summary

Phase 4c.1 lands the foundation for inline editor notes: an editor-only `note`
mark, the `stripNoteMarks`/`extractNotes` JSON helpers, a find-and-replace fix
that preserves notes overlapping a match, and a `renderEditorHtml()` chokepoint
that all rendered surfaces route through so private note text never reaches a
beta reader. The implementation is careful and heavily tested. One **Important**
finding: the DOCX export path uses its own independent TipTap walker that does
**not** go through the chokepoint, so the "all five export formats funnel
through `chapterContentToHtml`" invariant asserted in three places (CLAUDE.md,
`editorExtensions.ts`, `tiptap-notes.ts`) is false — DOCX is safe today only by
accident and is guarded by no test. Everything else is Suggestion-level: latent,
pre-existing, by-design, or an accepted idiom.

## Important Issues

### [I1] DOCX export bypasses the note-strip chokepoint; the "single route" invariant is false and untested
- **File:** `packages/server/src/export/docx.renderer.ts` (`renderDocx` → `tipTapToParagraphs` → `blockToParagraphs` → `inlineToRuns` → `marksToProps`, ~lines 44-64, 70-107, 409-421); false invariant asserted at `CLAUDE.md:154-155`, `packages/shared/src/editorExtensions.ts:38`, `packages/shared/src/tiptap-notes.ts:6-8`
- **Bug:** The confidentiality design rests on one load-bearing claim — that every rendered/exported surface routes through `renderEditorHtml()` (which strips note marks), and specifically that "all five export formats funnel through the server's `chapterContentToHtml`". This is false for DOCX. `export.service.ts` routes `docx` straight to `renderDocx`, which walks the raw TipTap JSON with its own visitor and never calls `renderEditorHtml`/`stripNoteMarks`. (EPUB, markdown, plaintext, and HTML *do* go through the chokepoint; DOCX is the lone exception.)
- **Impact:** Note confidentiality on the DOCX path holds **only by accident**: `marksToProps` switches on `bold`/`italic`/`strike`/`code` and silently ignores the `note` mark, and `inlineToRuns` reads only `node.text`, never `mark.attrs.text`. There is no DOCX note-leak regression test — `export.renderers.test.ts` covers `chapterContentToHtml`, `renderHtml`, and `renderEpub` only — and the extension-set pin in `editorExtensions.test.ts` cannot protect DOCX because DOCX doesn't use `renderEditorHtml`. A future edit to `marksToProps`/`inlineToRuns` (e.g. rendering a mark's attrs), or a future editor-only mark (Phase 4c.3 tags) whose payload the DOCX walker *does* read, would ship the writer's private commentary into the `.docx` handed to a beta reader — with no test turning red and no chokepoint to catch it.
- **Suggested fix:** Preferably make the chokepoint real for DOCX too — strip at the DOCX entry (`tipTapToParagraphs(stripNoteMarks(content), …)`) so all five formats share one guard. At minimum, add a DOCX note-leak regression test and correct the three "all five funnel through `chapterContentToHtml`" claims to name DOCX as a separate walker-based path.
- **Confidence:** High (verified: 90)

## Suggestions

- **[S1] Server export note-confidentiality is single-layer.** `packages/server/src/export/export.renderers.ts:49-63` — the client has two layers (upstream `renderEditorHtml` strip **and** `span` excluded from DOMPurify `ALLOWED_TAGS`); the server export path has only the strip. `stripDisallowedImages` matches `<img>` only, so a note-bearing `<span data-note>` that ever survived the strip would land in exported HTML/EPUB. Acceptable-by-design for a single-user app, but a real defense-in-depth asymmetry. *(Security + Error Handling)*
- **[S2] `extractNotes` double-counts a note spanning a hardBreak.** `packages/shared/src/tiptap-notes.ts:96-122` (`collect()` run loop) — a note applied across a line break is stored as `[text(note), hardBreak, text(note)]`; the `hardBreak` yields `noteTextOf === null`, resets `run` to null, and the two halves become two `ExtractedNote` entries for one contiguous note. Coalescing handles mark-change splits but not structural non-text splits. Latent (no production consumer yet), cosmetic until the panel lands. Fix: don't reset `run` on a content-less inline leaf (e.g. `hardBreak`). *(Logic)*
- **[S3] `extractNotes` is exported and tested but has zero production consumers.** `packages/shared/src/tiptap-notes.ts` / `index.ts:37` — export-ahead-of-use for the future Notes panel (plan Task 10). Its docstring forward-references a `collectNotes` client helper that does not yet exist anywhere in source. Acceptable staged development, but flags a drift risk: the panel needs editor positions `extractNotes` deliberately omits, so it may end up a second, subtly-different walker. Wire the panel to `extractNotes` when Task 10 lands, or delete it then. *(Contract & Integration)*
- **[S4] Depth-cap walk idiom duplicated.** `packages/shared/src/tiptap-notes.ts` (`strip`, `collect`) repeat the `if (depth > MAX_TIPTAP_DEPTH) return <fail-closed>` recursion also in `tiptap-safety.ts`, `wordcount.ts`, `tiptap-text.ts`. Mitigated: all share the single imported `MAX_TIPTAP_DEPTH` constant, so the load-bearing value can't drift; only the ~1-line guard shape repeats. This is the codebase's accepted idiom, not harmful duplication — noted for completeness, not worth acting on. *(Contract & Integration)*
- **[S5] `chapterContentToHtml` silently returns `""` on render throw.** `packages/server/src/export/export.renderers.ts:56-64` — a chapter whose stored JSON makes rendering throw exports as a titled-but-empty chapter across all formats, signalled only by a `logger.warn` the writer never sees. Pre-existing (the same `try/catch { return "" }` wrapped `generateHTML` before this branch); the branch only adds `stripNoteMarks` as a second, rarely-throwing source inside the same swallow. Fail-safe for confidentiality, but a silent data-integrity gap for the writer's own backup/export. Consider a visible `[content could not be rendered]` placeholder. *(Error Handling)*
