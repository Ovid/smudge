## Executive Summary

This branch addresses 4 architecture findings (F-1, F-2, F-3, F-6) from the prior architecture review, adds comprehensive test coverage with enforced thresholds, and is generally well-executed. The primary concerns are: (1) a pattern where post-mutation re-fetches omit the `whereNull("deleted_at")` guard that exists on read endpoints, creating race condition windows; (2) corrupt JSON in the database silently becomes `null` content in API responses, which could allow client auto-save to permanently destroy the original data; and (3) the e2e test for chapter-switch persistence doesn't actually exercise the immediate-save-on-switch code path.

## Important Issues

### [I1] PATCH re-fetch omits soft-delete guard — race condition returns deleted chapter as 200
- **File:** `packages/server/src/routes/chapters.ts:96`
- **Bug:** After the PATCH update transaction, the re-fetch query `queryChapter(db("chapters").where({ id: req.params.id }))` omits `.whereNull("deleted_at")`. The GET handler at line 16 correctly includes this filter. If a concurrent DELETE request soft-deletes the chapter between the transaction commit and the re-fetch, the endpoint returns a 200 with the deleted chapter's data (including a non-null `deleted_at` field).
- **Impact:** Client receives a "successful save" response for a chapter that has been deleted. In a multi-tab scenario, auto-save races with manual delete are plausible.
- **Suggested fix:** Add `.whereNull("deleted_at")` to the re-fetch:
  ```typescript
  const updated = await queryChapter(
    db("chapters").where({ id: req.params.id }).whereNull("deleted_at"),
  );
  ```
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling, Contract & Integration, Security

### [I2] Corrupt JSON in DB silently becomes `null` content — auto-save can destroy original data
- **File:** `packages/server/src/routes/parseChapterContent.ts:9`
- **Bug:** When chapter content stored in SQLite is corrupt JSON, `parseChapterContent` logs an error but returns `{ ...chapter, content: null }`. The route handler sends a 200 response with `content: null`. The client initializes TipTap with null content (empty editor), and the auto-save pipeline may then overwrite the corrupt-but-present original with an empty document — permanently destroying the data.
- **Impact:** Silent, irreversible data loss for any chapter whose content becomes corrupt. This directly undermines the "core trust promise" of the save pipeline described in CLAUDE.md.
- **Suggested fix:** Either return a 500 with a `CORRUPT_CONTENT` error code so the client does not auto-save, or return the raw string in a `content_raw` field with a `content_corrupt: true` flag so the client can display a non-editable warning.
- **Confidence:** High
- **Found by:** Logic & Correctness

### [I3] Chapter-switch e2e test doesn't exercise immediate-save-on-switch code path
- **File:** `e2e/editor-save.spec.ts:115-119`
- **Bug:** The test waits for `"Saved"` (auto-save completes via debounce) *before* clicking the second chapter. This means the immediate-save-bypass-debounce code path that fires on chapter switch (documented in CLAUDE.md: "On chapter switch, immediate save bypasses debounce") is never exercised. The most critical save edge case — switching chapters while content is unsaved — has no e2e coverage.
- **Impact:** A regression in the immediate-save-on-switch path would not be caught by e2e tests.
- **Suggested fix:** Add a test variant that switches chapters *before* the debounce timer fires (i.e., immediately after typing, without waiting for "Saved"), then verifies content persists after switching back.
- **Confidence:** Medium
- **Found by:** Logic & Correctness
