## Executive Summary

This branch adds architecture hardening (corrupt content guards, encapsulated JSON parsing, coverage thresholds) and dashboard/status features on top of an active MVP. The most significant issues cluster around **inconsistent `content_corrupt` handling** across endpoints -- the single-chapter endpoints correctly return 500, but the project-level endpoint silently passes corrupt chapters through with `content: null`, risking silent data loss via auto-save. The restore and PATCH endpoints also have transaction-boundary issues where the corruption check runs after a successful write, creating inconsistent server/client state. No critical issues found; 4 important issues and 13 suggestions.

## Important Issues

### [I1] ~~Corrupt chapter content silently served as `null` in project response~~ FIXED (5dc2efd)
- **File:** `packages/server/src/routes/projects.ts:203-218`
- **Bug:** `GET /api/projects/:slug` uses `queryChapters` which calls `parseChapterContent`. If any chapter has corrupt JSON, it returns `{ content: null, content_corrupt: true }` in the 200 response. The `content_corrupt` flag is not in the shared `Chapter` type (`packages/shared/src/types.ts`), so the client ignores it. Meanwhile, `GET /api/chapters/:id` correctly returns 500 for corrupt content -- the two endpoints are inconsistent.
- **Impact:** A corrupt chapter appears blank to the user. If they start typing, auto-save overwrites the potentially recoverable corrupt data with new content. Silent, irreversible data loss.
- **Suggested fix:** Either (a) check for `content_corrupt` in the project-get handler and filter/flag affected chapters, (b) add `content_corrupt?: boolean` to the shared `Chapter` type and handle it client-side, or (c) strip content from the project-level response entirely (the client already fetches individual chapters via GET when selecting them).
- **Confidence:** High
- **Found by:** Logic, Error Handling, Contract, Concurrency, Security (5 agents)

### [I2] ~~Restore endpoint commits transaction then returns 500 for corrupt content~~ FIXED (5dc2efd)
- **File:** `packages/server/src/routes/chapters.ts:182-233`
- **Bug:** `POST /:id/restore` executes the restore transaction (lines 182-213), which commits successfully. Then the read-back at lines 216-233 checks `content_corrupt` and returns 500 if true. The chapter IS restored in the database, but the client receives an error and doesn't update its UI. A second restore attempt returns 404 because the chapter now has `deleted_at: null`.
- **Impact:** The user sees the chapter still in trash but it's actually restored. They cannot interact with it through either the trash view or the project view until a page reload.
- **Suggested fix:** Either (a) check for corruption before the restore transaction, or (b) allow the restore to succeed regardless and let the `GET /chapters/:id` endpoint catch corruption when the user opens the chapter, or (c) return 200 with a warning flag instead of 500.
- **Confidence:** High
- **Found by:** Error Handling

### [I3] ~~PATCH post-update corruption check blocks non-content updates~~ FIXED (5dc2efd)
- **File:** `packages/server/src/routes/chapters.ts:99-127`
- **Bug:** After any PATCH (even title-only or status-only updates), the read-back goes through `queryChapter` which calls `parseChapterContent`. If the chapter had pre-existing corrupt content, `content_corrupt` is true and the endpoint returns 500 -- even though the title/status update succeeded in the database (line 99-104 committed).
- **Impact:** A chapter with corrupt content becomes completely uneditable via the API. Title renames return 500. Status changes trigger the optimistic-update-then-revert cycle repeatedly. The user cannot fix metadata on a corrupt chapter.
- **Suggested fix:** Only check `content_corrupt` when the update included content changes, or move the corruption check to only apply when content is being loaded for editing.
- **Confidence:** High
- **Found by:** Error Handling

### [I4] ~~Missing max-length on chapter title in UpdateChapterSchema~~ FIXED (eb45abd)
- **File:** `packages/shared/src/schemas.ts:29`
- **Bug:** `UpdateChapterSchema` defines `title: z.string().trim().min(1).optional()` with no `.max()` constraint. `CreateProjectSchema` and `UpdateProjectSchema` both cap titles at 500 characters.
- **Impact:** A malformed client request can PATCH a chapter with an arbitrarily long title (up to the 5MB `express.json` limit). This bloats the database, inflates API response sizes (dashboard fetches all chapters), and could degrade performance.
- **Suggested fix:** Add `.max(500, "Title is too long")` to the chapter title in `UpdateChapterSchema`.
- **Confidence:** High
- **Found by:** Security
