## Important Issues

### [I1] Inconsistent `content_corrupt` handling -- only GET endpoint is protected
- **Files:** `packages/server/src/routes/chapters.ts:106-117` (PATCH), `chapters.ts:204-219` (restore), `packages/server/src/routes/projects.ts:203-218` (project detail), `projects.ts:418-427` (trash)
- **Bug:** `GET /api/chapters/:id` correctly checks `content_corrupt` and returns 500 `CORRUPT_CONTENT` (lines 26-34). However, PATCH re-fetch, POST restore re-fetch, GET project detail (`queryChapters`), and GET trash (`queryChapters`) all use the same `queryChapter`/`queryChapters` helpers that set `content_corrupt: true` on corrupt JSON, but none of these endpoints check the flag. They return 200 responses containing `{ content: null, content_corrupt: true }`.
- **Impact:** If a chapter has corrupt JSON in the DB and the client receives it via any of these unprotected paths, `content: null` could be interpreted as empty content. The auto-save pipeline may then overwrite the corrupt-but-recoverable original data with null/empty content, causing irreversible data loss. This undermines the "core trust promise" of the save pipeline.
- **Suggested fix:** Add `content_corrupt` checks to PATCH and restore re-fetch paths (return 500 like GET). For array endpoints (project detail, trash), either filter out content from the response, or include the flag and handle it client-side.
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling, Contract & Integration, Security

### [I2] Corrupt chapter triggers misleading full-page "Project not found" overlay
- **File:** `packages/client/src/pages/EditorPage.tsx:324-342`, `packages/client/src/hooks/useProjectEditor.ts:143`
- **Bug:** When a user clicks a chapter with corrupt content, `api.chapters.get()` returns 500 `CORRUPT_CONTENT`. The catch block in `handleSelectChapter` sets the `error` state. In EditorPage, any truthy `error` renders a full-page "Project not found" overlay that replaces the entire UI including the sidebar. The user cannot navigate to other (healthy) chapters and the error message is misleading -- the project exists, only one chapter is corrupt.
- **Impact:** A single corrupt chapter makes the entire project inaccessible until the DB is manually repaired.
- **Suggested fix:** Distinguish between fatal errors (project not found) and per-chapter errors (corrupt content). For `CORRUPT_CONTENT`, show an inline error in the editor area while keeping the sidebar navigable.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [I3] Chapter creation re-fetch omits `whereNull("deleted_at")`
- **File:** `packages/server/src/routes/projects.ts:259`
- **Bug:** After inserting a new chapter, the re-fetch is `queryChapter(db("chapters").where({ id: chapterId }))` without `.whereNull("deleted_at")`. Every other chapter re-fetch in the codebase includes this filter (chapters.ts:16, chapters.ts:107, chapters.ts:204-205). CLAUDE.md explicitly states: "All queries must filter `deleted_at IS NULL`."
- **Impact:** Functionally safe for a just-created chapter, but violates the project's architectural rule and is inconsistent with the pattern applied elsewhere. Makes code review harder -- a reader must verify whether the omission is intentional or a bug.
- **Suggested fix:** Add `.whereNull("deleted_at")` to match the pattern:
  ```typescript
  const chapter = await queryChapter(db("chapters").where({ id: chapterId }).whereNull("deleted_at"));
  ```
- **Confidence:** High
- **Found by:** Logic & Correctness, Plan Alignment

### [I4] Chapter creation INSERT + UPDATE not in a transaction
- **File:** `packages/server/src/routes/projects.ts:237-257`
- **Bug:** The `max("sort_order")` read (line 237), chapter INSERT (line 246), and project `updated_at` UPDATE (line 257) are three separate awaits with no transaction wrapper. If the process crashes between INSERT and UPDATE, the chapter exists but `project.updated_at` is stale. Additionally, concurrent chapter creates could read the same max sort_order and produce duplicates. This is inconsistent with other multi-table operations in the same file (project creation at line 51, chapter reorder at line 321, project delete at line 447) which all use transactions.
- **Impact:** Low practical risk in a single-user app, but violates the transactional discipline established throughout the codebase.
- **Suggested fix:** Wrap the read-insert-update in a single transaction.
- **Confidence:** Medium
- **Found by:** Concurrency & State

### [I5] `handleSave` discards 4xx server error message
- **File:** `packages/client/src/hooks/useProjectEditor.ts:92-104`
- **Bug:** When a save attempt receives a 4xx response (e.g., 400 VALIDATION_ERROR), the retry loop correctly breaks immediately (line 94). However, the specific error message from the server is discarded. The code falls through to set `setSaveStatus("error")`, and the user sees only the generic "Unable to save" message. The actual validation error (e.g., "content must have root type 'doc'") is lost.
- **Impact:** If content becomes invalid (e.g., a bug in a TipTap extension), the user sees "Unable to save" with no indication of what's wrong. Retrying by typing more will keep failing. No actionable information is surfaced.
- **Suggested fix:** Capture the error message from the 4xx response and expose it via a separate state variable so the UI can display the specific reason.
- **Confidence:** Medium
- **Found by:** Error Handling & Edge Cases

## Suggestions

- `packages/server/src/routes/projects.ts:357-360` — Dashboard handler queries `chapter_statuses` table twice: once directly into `allStatuses` and once via `getStatusLabelMap(db)`. Build the label map from `allStatuses` to eliminate the redundant query. (Contract & Integration, confidence 90%)

- `packages/server/src/routes/chapters.ts:100` — PATCH UPDATE inside transaction omits `whereNull("deleted_at")`, creating a narrow TOCTOU window where a concurrent delete + PATCH could update a soft-deleted chapter. Add the filter for consistency. (Error Handling, confidence 60%)
- `packages/server/src/routes/chapters.ts` and `projects.ts` — Status label enrichment pattern (`{ ...chapter, status_label: ... }`) duplicated 6 times with two lookup strategies (`getStatusLabel` vs `getStatusLabelMap`). Consider an `enrichChapter` utility. (Contract & Integration, confidence 65%)
