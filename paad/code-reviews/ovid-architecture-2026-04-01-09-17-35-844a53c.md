## Important Issues

### [I1] `cacheWarning` not cleared on successful save or chapter switch
- **File:** `packages/client/src/hooks/useProjectEditor.ts:92-95`
- **Bug:** After a successful save, `handleSave` calls `clearCachedContent()` and `setSaveStatus("saved")` but never calls `setCacheWarning(false)`. Similarly, `handleSelectChapter` and `handleCreateChapter` don't clear it. Once `cacheWarning` is set to `true` (e.g., from a transient localStorage quota hit), it persists until the user types again and the next `handleContentChange` call succeeds at caching.
- **Impact:** User sees a persistent "Local backup unavailable" warning in the status bar even after content has been successfully saved and localStorage has recovered. This erodes trust in the save pipeline -- the core promise of the app.
- **Suggested fix:** Add `setCacheWarning(false)` after the successful `clearCachedContent` call in `handleSave` (~line 93). Also clear it in `handleSelectChapter` and `handleCreateChapter`.
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling & Edge Cases, Concurrency & State

### [I2] Test mocks for `setCachedContent` return `undefined` instead of `true`
- **File:** `packages/client/src/__tests__/App.test.tsx:8`, `ChapterTitle.test.tsx:10`, `EditorPageFeatures.test.tsx:10`, `KeyboardShortcuts.test.tsx:10`, `StatusBar.test.tsx:9`
- **Bug:** All five mock blocks use `setCachedContent: vi.fn()` with no return value configured. Since `useProjectEditor` now checks `const cached = setCachedContent(...)` and sets `setCacheWarning(!cached)`, any test triggering `handleContentChange` will have `cacheWarning = true`. Tests are running with a false cache-failure state that doesn't reflect production behavior.
- **Impact:** Tests may pass despite showing incorrect UI state. Any test asserting on the status bar area is testing against a spurious "Local backup unavailable" warning. Combined with I1, this masks the cacheWarning bug from test detection.
- **Suggested fix:** Change all five occurrences to `setCachedContent: vi.fn().mockReturnValue(true)`.
- **Confidence:** High
- **Found by:** Contract & Integration

### [I3] `handleCreateChapter` does not flush unsaved editor content
- **File:** `packages/client/src/hooks/useProjectEditor.ts:129-141` and `packages/client/src/pages/EditorPage.tsx:369,383,408`
- **Bug:** `handleSelectChapter` is wrapped in `handleSelectChapterWithFlush` which calls `editorRef.current?.flushSave()` before switching chapters. However, `handleCreateChapter` is passed directly to Sidebar and the empty-state button without any flush wrapper. It also increments `saveSeqRef.current` (line 132), cancelling in-flight save retries.
- **Impact:** If the user has unsaved content and clicks "Add Chapter", the debounced save is cancelled, in-flight retries are aborted, and the editor switches to a new empty chapter. The old chapter's unsaved content is lost (client-side cache may still hold it, but the save pipeline is broken for that content).
- **Suggested fix:** Create a `handleCreateChapterWithFlush` wrapper in EditorPage that calls `editorRef.current?.flushSave()` before `handleCreateChapter()`, mirroring the existing `handleSelectChapterWithFlush` pattern. Use it in all call sites.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [I4] `handleDeleteChapter` cancels save retries for ALL chapters, not just the deleted one
- **File:** `packages/client/src/hooks/useProjectEditor.ts:167`
- **Bug:** `++saveSeqRef.current` is a global counter. When deleting any chapter, it invalidates the sequence number for any in-flight save, including saves for the currently active (non-deleted) chapter. Example: Chapter 1 is being saved with retries. User deletes Chapter 3. The `++saveSeqRef.current` causes Chapter 1's retry loop to exit early (`seq !== saveSeqRef.current` on line 77).
- **Impact:** Silent content loss for the active chapter when a different chapter is deleted, if a save retry is in progress. The cached content would still be in localStorage, but the save is silently abandoned without error indication.
- **Suggested fix:** Only increment `saveSeqRef` when deleting the currently active chapter: `if (activeChapterRef.current?.id === chapter.id) ++saveSeqRef.current;`
- **Confidence:** Medium
- **Found by:** Error Handling & Edge Cases
