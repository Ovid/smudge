## Important Issues

### [I1] helmet() default CSP will block frontend when static serving is added
- **File:** `packages/server/src/app.ts:20`
- **Bug:** `helmet()` is called with no configuration. Helmet v8 defaults include `Content-Security-Policy` with `script-src 'self'` and `style-src 'self'`, which blocks inline scripts and styles. TipTap's ProseMirror editor relies on inline styles for cursor positioning, selection, and content rendering. Vite production builds may inject inline scripts. Currently Express only serves API routes (no `express.static`), so the bug is latent -- but CLAUDE.md states the production deployment model is "Express serves API + static frontend on port 3456."
- **Impact:** When static file serving is wired up for Docker deployment, the editor will be visually broken and the SPA may fail to load. The issue will not surface in development (separate Vite dev server).
- **Suggested fix:** Configure helmet with CSP appropriate for TipTap, or disable CSP until static serving is implemented:
  ```typescript
  app.use(helmet({ contentSecurityPolicy: false }));
  ```
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling & Edge Cases, Contract & Integration, Security

### [I2] Status change revert overwrites entire project state, losing concurrent optimistic updates
- **File:** `packages/client/src/hooks/useProjectEditor.ts:262`
- **Bug:** When `handleStatusChange` API call fails, the catch block calls `setProject(data)` with a full server reload response, replacing the entire project state. If the user made other optimistic changes between the API call and the server round-trip (reorder, rename, create chapter), those changes are silently lost. Unlike `handleSave` which uses functional updaters throughout, the revert path uses a non-functional setter.
- **Impact:** A status change failure could visually revert a concurrent chapter reorder or title rename with no explanation to the user.
- **Suggested fix:** Surgically revert only the `status` field on the affected chapter instead of replacing the whole project:
  ```typescript
  const revertedChapter = data.chapters.find((c) => c.id === chapterId);
  if (revertedChapter) {
    setProject((prev) => {
      if (!prev) return prev;
      return { ...prev, chapters: prev.chapters.map((c) =>
        c.id === chapterId ? { ...c, status: revertedChapter.status } : c
      )};
    });
  }
  ```
- **Confidence:** High
- **Found by:** Contract & Integration, Concurrency & State

### [I3] setCachedContent silently swallows QuotaExceededError -- caller cannot react
- **File:** `packages/client/src/hooks/useContentCache.ts:14-19`
- **Bug:** When `localStorage.setItem` throws (e.g., storage full), the function logs `console.warn` and returns silently. The caller (`handleContentChange` at useProjectEditor.ts:122) has no way to know caching failed. CLAUDE.md states: "client-side cache holds unsaved content until server confirms." If caching silently fails, the safety net for unsaved content is broken without any user indication.
- **Impact:** A user could lose unsaved content if both the cache write and subsequent server save fail, with no warning that the safety net was inactive.
- **Suggested fix:** Return a boolean from `setCachedContent` indicating success, allowing the caller to react (e.g., trigger an immediate save or show a subtle warning).
- **Confidence:** Medium
- **Found by:** Error Handling & Edge Cases
