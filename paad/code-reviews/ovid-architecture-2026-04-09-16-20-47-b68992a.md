## Executive Summary

This branch decomposes the EditorPage god component (859 lines, 20+ state variables) into 5 focused hooks and 1 new component. The decomposition is well-structured and successfully addresses architecture report finding F-01. However, the refactoring introduced two critical data integrity bugs where blur-triggered saves can target the wrong entity during chapter/project switches, plus several important concurrency and contract issues. Overall confidence in the findings is high -- the two critical bugs were independently identified by multiple specialist agents.

## Critical Issues

### [C1] Chapter title save to wrong chapter on blur during chapter switch — FIXED (adc3c88)
- **File:** `packages/client/src/hooks/useChapterTitleEditing.ts:21-24, 35-59`
- **Bug:** When a user is editing a chapter title and navigates away (e.g., Ctrl+Shift+Arrow), the `useEffect` at line 22 sets `editingTitle` to false, unmounting the input. The browser fires `onBlur`, which calls `saveTitle()`. Since `escapePressedRef` was never set to true, `saveTitle` proceeds and may call `handleRenameChapter` with the old draft against the new chapter's ID.
- **Impact:** Data corruption -- the wrong chapter gets renamed with a stale title draft.
- **Suggested fix:** Set `escapePressedRef.current = true` in the useEffect before `setEditingTitle(false)`:
  ```ts
  useEffect(() => {
    escapePressedRef.current = true;
    setEditingTitle(false);
    setTitleError(null);
  }, [activeChapter?.id]);
  ```
- **Confidence:** High
- **Found by:** Logic & Correctness, Concurrency & State

### [C2] Project title save to wrong entity on blur during project change — FIXED (adc3c88)
- **File:** `packages/client/src/hooks/useProjectTitleEditing.ts:17-20, 31-54`
- **Bug:** Identical pattern to C1. When `project?.id` changes, the useEffect sets `editingProjectTitle` to false without setting `projectEscapePressedRef`, allowing the blur handler to save a stale draft to the wrong project.
- **Impact:** Data corruption -- the wrong project gets renamed.
- **Suggested fix:** Set `projectEscapePressedRef.current = true` in the useEffect:
  ```ts
  useEffect(() => {
    projectEscapePressedRef.current = true;
    setEditingProjectTitle(false);
  }, [project?.id]);
  ```
- **Confidence:** High
- **Found by:** Logic & Correctness, Concurrency & State

## Important Issues

### [I1] Double-fire of chapter saveTitle via finally block timing — FIXED (adc3c88)
- **File:** `packages/client/src/hooks/useChapterTitleEditing.ts:35-59`
- **Bug:** When Enter is pressed, `saveTitle()` runs async. The `finally` block (line 57-58) resets `isSavingTitleRef.current = false` before React processes the `setEditingTitle(false)` unmount. When the input unmounts, blur fires and calls `saveTitle()` again. The `isSavingTitleRef` guard no longer blocks, so the rename API is called twice.
- **Impact:** Duplicate API call. The second call is usually a no-op (title already matches), but wastes a request and could cause issues if the rename has side effects.
- **Suggested fix:** Don't reset `isSavingTitleRef` in `finally`. Reset it in the `useEffect` that watches `activeChapter?.id` instead, or set `escapePressedRef.current = true` before `setEditingTitle(false)` inside `saveTitle`.
- **Confidence:** High
- **Found by:** Concurrency & State

### [I2] Double-fire of project saveProjectTitle (same pattern as I1) — FIXED (adc3c88)
- **File:** `packages/client/src/hooks/useProjectTitleEditing.ts:31-55`
- **Bug:** Identical double-fire pattern. Extra risk: project rename can change the slug, causing a `navigate()` call. A double-fire could trigger two navigations.
- **Impact:** Duplicate API call and potential double navigation.
- **Suggested fix:** Same approach as I1.
- **Confidence:** High
- **Found by:** Concurrency & State

### [I3] handleSave recreated on every successful save — FIXED (a578658)
- **File:** `packages/client/src/hooks/useProjectEditor.ts:60, 116-117`
- **Bug:** `handleSave` has `[activeChapter]` in its dependency array. `activeChapter` is updated on every successful save (line 79-82: `setActiveChapter` with new content/word_count), which recreates `handleSave`. The function already uses `activeChapterRef` for post-async checks (lines 78, 94, 108) but reads `activeChapter` directly at lines 62-63 for the initial guard and ID capture. These could use `activeChapterRef.current` instead, allowing `[]` deps.
- **Impact:** Unnecessary function recreation and re-render cascade on every auto-save. In a rich text editor with frequent saves, this could cause jank.
- **Suggested fix:** Replace lines 62-63 with `const current = activeChapterRef.current; if (!current) return false; const savingChapterId = current.id;` and change deps to `[]`.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases, Concurrency & State

### [I4] ProjectSettings onUpdate overwrites optimistic state — FIXED (15e46ca)
- **File:** `packages/client/src/pages/EditorPage.tsx:584-596`
- **Bug:** When `ProjectSettingsDialog` calls `onUpdate`, the handler fetches the full project via `api.projects.get(slug)` and replaces the entire project state with `setProject(data)`. Any optimistic updates (chapter reorder, status changes, renames) that haven't been reflected on the server are lost.
- **Impact:** Sidebar could snap back to stale data if the user triggered other actions while the settings dialog was open.
- **Suggested fix:** Merge only the settings fields (`target_word_count`, `target_deadline`, `completion_threshold`, `title`, `slug`) from the fetched data using a functional updater, rather than replacing the entire project.
- **Confidence:** Medium
- **Found by:** Error Handling & Edge Cases

### [I5] confirmDeleteChapter missing try/catch — FIXED (74373a6)
- **File:** `packages/client/src/hooks/useTrashManager.ts:56-69`
- **Bug:** `confirmDeleteChapter` calls `await handleDeleteChapter(deleteTarget)` without try/catch. While `handleDeleteChapter` has its own error handling and returns false, if an unexpected synchronous error occurs, `deleteTarget` is never cleared and the confirmation dialog becomes stuck (no dismiss path).
- **Impact:** A stuck confirmation dialog requiring a page refresh.
- **Suggested fix:** Wrap in try/catch and clear `deleteTarget` in the catch block.
- **Confidence:** Medium
- **Found by:** Error Handling & Edge Cases, Logic & Correctness

### [I6] Sidebar magic numbers not imported from useSidebarState — FIXED (a2c0cf4)
- **File:** `packages/client/src/components/Sidebar.tsx:473-474, 482, 500, 504`
- **Bug:** `useSidebarState.ts` exports `SIDEBAR_MIN_WIDTH` (180) and `SIDEBAR_MAX_WIDTH` (480), but `Sidebar.tsx` hardcodes these values as inline literals instead of importing the constants. This was flagged as F-13 in the architecture report and has not been resolved -- the source of truth moved but the consumer doesn't use it.
- **Impact:** Future changes to min/max width would require updating both files, risking drift.
- **Suggested fix:** Import `SIDEBAR_MIN_WIDTH` and `SIDEBAR_MAX_WIDTH` from `useSidebarState` in `Sidebar.tsx`.
- **Confidence:** High
- **Found by:** Plan Alignment
