## Important Issues

### [I1] Error handler maps all non-404/409 4xx to VALIDATION_ERROR and leaks `err.message`
- **File:** `packages/server/src/app.ts:56-64`
- **Bug:** The ternary only branches on 404 and 409. Any other 4xx status (413 Payload Too Large from `express.json({ limit: "5mb" })`, 401, 403, etc.) gets the code `"VALIDATION_ERROR"`, which is semantically incorrect. Additionally, `err.message` is passed verbatim to the client for all 4xx errors — Express's `SyntaxError` for malformed JSON produces messages like `"Unexpected token } in JSON at position 42"`, leaking parser internals.
- **Impact:** Misleading machine-readable error codes for clients; minor information disclosure (OWASP A05).
- **Suggested fix:** Add explicit mappings for 400 and 413, and sanitize `err.message` for middleware-originated errors (e.g., check for `err instanceof SyntaxError` and substitute a generic message).
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling & Edge Cases, Security

### [I2] Nested-transaction guard relies on undocumented Knex internal `isTransaction`
- **File:** `packages/server/src/stores/sqlite-project-store.ts:167`
- **Bug:** `if (this.db.isTransaction)` checks an undocumented internal property of `Knex.Transaction`. This is the only protection against nested transactions, which under SQLite/better-sqlite3 would silently create savepoints or deadlock. If a future Knex upgrade renames or removes this property, the guard silently stops working.
- **Impact:** Nested transaction attempts would not throw and could cause data corruption or deadlocks.
- **Suggested fix:** Track transaction state explicitly with a private boolean field on `SqliteProjectStore` set during construction, rather than relying on the Knex internal. Alternatively, add a type-level constraint that removes `transaction` from the transactional store's interface.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases, Contract & Integration, Concurrency & State

## Suggestions

- **[S1] Dead re-export in `chapters.service.ts:30`** — `export { isCorruptChapter, stripCorruptFlag }` re-exports functions that were moved to `chapters.types`. No consumer uses this path (all callers import from `chapters.types` directly). Remove the re-export to eliminate the ambiguous dual import path. *(Found by: Logic, Contract)*

- **[S2] Redundant `const msPerDay = MS_PER_DAY` aliases in `velocity.service.ts:75,121`** — Two functions create pointless local aliases of the module-level `MS_PER_DAY` constant. Use `MS_PER_DAY` directly. *(Found by: Contract)*

- **[S3] `console.error(err)` survives in `EditorPage.tsx:117`** — The statuses fetch retry handler uses a raw `console.error(err)` while the server migration to pino (F-01) replaced all `console.*` calls. This is client-side so pino doesn't apply, but it's inconsistent with the project's logging conventions and was in a file actively modified by this branch. *(Found by: Contract, Concurrency)*

- **[S4] `ViewModeNav.tsx` `.catch(() => {})` swallows flush errors silently** — All three view-mode buttons use `.catch(() => {})`. While `flushSave` is best-effort and save failures surface via the status bar, the empty catch prevents any diagnostic logging in dev. Replace with `.catch((err) => console.warn("View switch flush failed:", err))`. *(Found by: Error Handling, Concurrency)*

- **[S5] TOCTOU: velocity snapshot reads word count outside chapter transaction** — `chapters.service.ts:86-106` commits the chapter update, then `velocity.service.ts:45-52` opens a separate transaction to read and snapshot the word count. Another write between the two transactions could produce an incorrect snapshot. Negligible risk under single-user SQLite serialized writes; documented as best-effort. *(Found by: Concurrency, Error Handling)*

- **[S6] `LOG_LEVEL` env var not validated** — `logger.ts:6` passes `process.env.LOG_LEVEL` to pino without checking it's a valid level. An invalid value may crash the server at startup with an unhelpful error. Consider validating against pino's known levels. *(Found by: Error Handling, Security)*

- **[S7] `getTodayDate` reads timezone setting outside snapshot transaction** — `velocity.service.ts:29-41` reads the timezone in a separate query before the snapshot transaction opens. A timezone change at midnight could cause the snapshot to be filed under the wrong date. Negligible practical risk for a single-user app. *(Found by: Concurrency)*
