## Important Issues

### [I1] `handleCreateChapter` doesn't reset `saveStatus` or `saveErrorMessage`
- **File:** `packages/client/src/hooks/useProjectEditor.ts:125-139`
- **Bug:** `handleCreateChapter` increments `saveSeqRef` (cancelling in-flight retries) and clears `cacheWarning`, but does not call `setSaveStatus("idle")` or `setSaveErrorMessage(null)`. Compare with `handleSelectChapter` at line 144 which resets both.
- **Impact:** If a save error was showing for the previous chapter, creating a new chapter will display a stale "Unable to save" banner for a chapter that has never been saved. Confusing UX.
- **Suggested fix:** Add `setSaveStatus("idle"); setSaveErrorMessage(null);` at the top of `handleCreateChapter`, mirroring `handleSelectChapter`.
- **Confidence:** High
- **Found by:** Logic & Correctness

### [I2] 4xx save failures silently break retry loop without logging
- **File:** `packages/client/src/hooks/useProjectEditor.ts:98-106`
- **Bug:** When a save attempt fails with a 4xx status (lines 99-101), the retry loop `break`s immediately. The error is never logged (no `console.warn`, unlike every other catch block in this file), and only the generic `STRINGS.editor.saveFailed` is shown. Server-provided validation details (e.g., "Invalid status: xyz") are discarded.
- **Impact:** 4xx errors indicate a permanent failure (won't resolve on retry). The user sees a generic "Unable to save" with no diagnostics, and there's nothing in the console to help debug. Every other error handler in this file logs with `console.warn`.
- **Suggested fix:** Add `console.warn("Save failed with 4xx:", err);` before the `break`, and consider surfacing the server's error message in `saveErrorMessage`.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [I3] Missing delegation test for `upsertDailySnapshot` in store test
- **File:** `packages/server/src/__tests__/sqlite-project-store.test.ts`
- **Bug:** The `SqliteProjectStore` test file has delegation tests for every store method (projects, chapters, statuses, transactions) except the newly-added `upsertDailySnapshot`. The method is implemented at `sqlite-project-store.ts:165` and is exercised indirectly via velocity integration tests, but the delegation layer is untested.
- **Impact:** A wiring bug in the store's delegation to `velocityRepo.upsertDailySnapshot` would go undetected at the unit level. Breaks the established pattern of per-method delegation tests.
- **Suggested fix:** Add a `describe("velocity delegation")` block with a test that calls `store.upsertDailySnapshot(projectId, date, count)` and verifies the row was inserted/updated in `daily_snapshots`.
- **Confidence:** High
- **Found by:** Contract & Integration

### [I4] Chapter status fetch silently fails after retries
- **File:** `packages/client/src/pages/EditorPage.tsx:105-129`
- **Bug:** The chapter status fetch retries twice (3 total attempts), then silently gives up. The `statuses` state remains an empty array with no error indication. The sidebar status dropdown would render with no options, making it impossible for the user to change chapter statuses.
- **Impact:** Silent feature degradation with no user feedback. The DashboardView has a fallback (synthesizes labels from `status_summary` keys), but the sidebar's status dropdown becomes unusable with no indication of why.
- **Suggested fix:** Set an error state or show a subtle banner when statuses fail to load. Alternatively, provide a retry mechanism or surface the issue in the UI.
- **Confidence:** Medium
- **Found by:** Error Handling & Edge Cases

## Suggestions

- **[S1]** `packages/server/src/app.ts:66-77` ‚Äî SyntaxError check in the error handler's message mapping takes priority over status-specific messages. A SyntaxError with non-400 status would say "Invalid JSON in request body" instead of the status-appropriate message. Fix: narrow to `status === 400 && err instanceof SyntaxError`. Theoretical only -- Express JSON parser always sets status 400. (Found by: Error Handling & Edge Cases)
- **[S2]** `packages/client/src/hooks/useProjectEditor.ts:170-176` ‚Äî `handleDeleteChapter` computes `remaining` chapters from `projectRef.current` before `setProject` runs. A concurrent state update between the ref read and the state setter could cause a stale fallback chapter selection. Safe in practice for a single-user app. (Found by: Concurrency & State)
- **[S3]** `packages/server/src/__tests__/error-handler.test.ts:13-65` ‚Äî Test duplicates the error handler from `app.ts` rather than importing it. Comment at line 13 acknowledges this as debt. Extract the handler into a named export to eliminate the duplication risk. (Found by: Contract & Integration)
- **[S4]** `packages/server/src/logger.ts:6-8` ‚Äî `LOG_LEVEL` env var is not trimmed. Trailing whitespace (e.g., from `.env` files) causes silent fallback to "info" with a warning only on raw stderr. Fix: `process.env.LOG_LEVEL?.trim()`. (Found by: Error Handling & Edge Cases)
- **[S5]** `packages/server/src/chapters/chapters.service.ts:59-88` ‚Äî Chapter read at line 59 happens outside the transaction starting at line 83. Accidentally safe due to SQLite's serialized writes and `project_id` being immutable. (Found by: Concurrency & State)
- **[S6]** `packages/server/src/velocity/velocity.service.ts:102-173` ‚Äî `getVelocityBySlug` performs 6+ non-transactional reads.
