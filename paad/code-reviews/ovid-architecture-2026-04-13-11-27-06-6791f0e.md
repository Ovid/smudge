## Suggestions

### [S1] Inconsistent contract: singular enricher strips content_corrupt, plural does not
- **File:** `packages/server/src/chapters/chapters.types.ts:84-99`
- **Bug:** `enrichChapterWithLabel` (singular) internally calls `stripCorruptFlag`, removing `content_corrupt` from the output. `enrichChaptersWithLabels` (plural) does NOT strip -- it passes through all input fields plus `status_label`. Current callers work around this correctly (`projects.service.ts:112` manually strips before calling the plural version, and `getDashboard` passes `ChapterMetadataRow` which lacks `content_corrupt`). A future caller passing `ChapterRow[]` directly to the plural version would silently retain `content_corrupt`.
- **Impact:** Latent inconsistency, not an active bug. Refactoring from a loop of singular calls to a single plural call would lose the stripping behavior.
- **Suggested fix:** Either make the plural version also strip `content_corrupt` when given `ChapterRow`, or document the contract difference clearly.
- **Confidence:** High
- **Found by:** Contract & Integration

### [S2] getDashboard calls getStatusLabelMap() twice after refactor
- **File:** `packages/server/src/projects/projects.service.ts:253-255`
- **Bug:** `enrichChaptersWithLabels(store, chapters)` on line 253 internally calls `store.getStatusLabelMap()`. Then line 255 calls `store.getStatusLabelMap()` again for the `statusSummary` computation. The pre-refactor code fetched the map once and used it for both purposes.
- **Impact:** Redundant DB query. Performance impact is negligible (chapter_statuses is a tiny seed-data table with synchronous SQLite), but it's wasteful and the kind of thing that accumulates.
- **Suggested fix:** Fetch the map once and pass it to both consumers, or add an overload to `enrichChaptersWithLabels` that accepts a pre-fetched map.
- **Confidence:** High
- **Found by:** Contract & Integration, Logic & Correctness

### [S3] resetProjectStore JSDoc contradicts barrel "test-only" comment
- **File:** `packages/server/src/stores/index.ts:4-5` and `packages/server/src/stores/project-store.injectable.ts:25-28`
- **Bug:** The barrel comment says "setProjectStore and resetProjectStore are @internal (test-only)" but the JSDoc on `resetProjectStore` says "Production code uses this only during graceful shutdown." Production code at `packages/server/src/index.ts:73` does call `resetProjectStore()` during shutdown, so the barrel comment is inaccurate.
- **Impact:** Misleading documentation could cause a developer to remove the shutdown call thinking it's test scaffolding that leaked into production.
- **Suggested fix:** Update the barrel comment to: "setProjectStore is @internal (test-only); resetProjectStore is @internal (test + graceful shutdown). Import directly from './project-store.injectable' when needed."
- **Confidence:** Medium
- **Found by:** Logic & Correctness, Contract & Integration
