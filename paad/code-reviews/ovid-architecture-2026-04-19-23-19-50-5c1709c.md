## Executive Summary

Phase 4b.1 cleanly extracts the server-mutating-editor shape into `useEditorMutation` and migrates the three intended call sites. Invariants 1–4 (markClean / setEditable / cache-clear ordering / seq bump) are correctly enforced by construction in the happy, flush-fail, mutate-throw, reload-fail, busy, and synchronous-entry-throw paths. Two **Important** findings survive verification: (a) the reload-failed read-only lock is silently lifted by any subsequent mutation that returns through the finally with `reloadFailed=false`, while the persistent "refresh" banner stays on screen — a direct contradiction of the I1 design; and (b) the `setEditable(true)` call in the hook's `finally` lacks the symmetric try/catch that the C1 fix applied to `setEditable(false)`, so a synchronous throw on exit permanently latches `inFlightRef` for the session. A third **Important** finding is a strings-externalization leak on the save-path 4xx error message. The remaining six verified findings are Suggestions.

## Important Issues

### [I1] Reload-failed editor lock silently lifted by subsequent mutation
- **File:** `packages/client/src/hooks/useEditorMutation.ts:115-120` paired with `packages/client/src/pages/EditorPage.tsx:636-638`
- **Bug:** When a mutation's reload stage fails, the hook intentionally leaves the editor `setEditable(false)` and EditorPage sets a persistent `editorLockedMessage` banner. That banner is only cleared by the effect at EditorPage.tsx:636-638, which keys on `[activeChapter?.id, chapterReloadKey]`. A subsequent `mutation.run()` call that ends at stages `flush`, `mutate`, or a successful mutation whose directive sets `reloadActiveChapter: false` (e.g. a project-wide replace that doesn't touch the active chapter) returns through the finally with `reloadFailed=false` and unconditionally calls `editor?.setEditable(true)` — re-enabling typing while the "Editor reload failed — refresh the page" banner still claims the editor is locked.
- **Impact:** The user, trusting the banner, refreshes; but between the reload-fail and the subsequent re-enable, any auto-save they triggered by typing (before noticing the banner was a lie) PATCHes pre-restore/pre-replace content over the server's authoritative committed content. This is precisely the data-loss scenario the reloadFailed branch was designed to prevent. The safety net is the clearCacheFor step already wiped the client cache, so the draft isn't persisted on refresh — but a PATCH emitted before the user refreshes still clobbers the server.
- **Suggested fix:** Thread the locked state into the hook so `setEditable(true)` is gated by `!locked`, or have EditorPage pass a `isLocked()` predicate to `useEditorMutation` that wraps the finally's re-enable. Alternative: clear `editorLockedMessage` inside the hook's finally whenever the run ends non-reload-failed — and make EditorPage set the banner only after confirming the hook hasn't already recovered.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases; corroborated by Concurrency & State

### [I2] `setEditable(true)` in hook finally can throw synchronously and permanently latch `inFlightRef`
- **File:** `packages/client/src/hooks/useEditorMutation.ts:115-120`
- **Bug:** The finally block runs `editor?.setEditable(true); inFlightRef.current = false;` in that order. The commit at 5db6867 (C1) explicitly fixed the symmetric bug on the entry side — `setEditable(false)` can throw synchronously during TipTap mid-remount, and the authors moved `inFlightRef.current = true` inside the try so the finally would still clear it. The exit side left the same hazard exposed: if `setEditable(true)` throws, the next line never runs, `inFlightRef` stays true, and every future `mutation.run()` returns `{ stage: "busy" }` for the rest of the session.
- **Impact:** One bad synchronous TipTap call (triggered by, e.g., the Editor being in the middle of unmounting when the finally runs — exactly the window the migration was built to protect against) permanently disables all editor mutations until page refresh. The test at `useEditorMutation.test.tsx:304` asserts the entry-side fix; no test covers the exit side.
- **Suggested fix:** Reorder the finally so `inFlightRef.current = false` runs first, or wrap the `setEditable(true)` call in its own try/catch. The change is a two-line edit; mirror the C1 test for the exit side.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [I3] Raw server English 4xx save-error message leaks to UI via `saveErrorMessage`
- **File:** `packages/client/src/hooks/useProjectEditor.ts:195-201, 229`
- **Bug:** On a 4xx response from `api.chapters.update`, the handler assigns `rejected4xx = { message: err.message || STRINGS.editor.saveFailed }`, and line 229 surfaces that raw string via `setSaveErrorMessage`. `err.message` on `ApiRequestError` comes from the server's `body.error?.message` (api/client.ts:64), i.e. raw server-authored English, bypassing the strings.ts externalization discipline CLAUDE.md requires.
- **Impact:** Blocks future i18n; lets server copy changes silently alter the UI; and contradicts the codebase's own discipline — `findReplaceErrors.ts:32-34` explicitly comments on not leaking raw server English for the replace path. The save path is the most prominent error surface and should follow the same pattern.
- **Suggested fix:** Map `err.code` (INVALID_CONTENT, TITLE_TOO_LONG, etc.) to specific strings.ts keys; fall back to `STRINGS.editor.saveFailed`. Mirror the structure of `mapReplaceErrorToMessage`. A 413 branch (content-too-large) also deserves its own copy.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

## Suggestions

- **[S1]** `MutationResult<T>` reload-stage `error?: string` field is dead: all three EditorPage callers render hardcoded STRINGS copy; `reloadActiveChapter` only ever sends `STRINGS.error.loadChapterFailed` — remove the field or document as reserved. `useEditorMutation.ts:20-24`, `:100-111`.
- **[S2]** Plan doc (`docs/plans/2026-04-19-editor-orchestration-helper-design.md:72-78`) specifies `error?: unknown` uniformly but implementation diverges (`string` on reload, required `unknown` on flush/mutate, none on busy). Update the plan doc to match or simplify the impl.
- **[S3]** `handleSelectChapter` (useProjectEditor.ts:266-295) and `handleDeleteChapter` (:347-398) do not resolve `saveBackoffRef` — unlike `cancelPendingSaves`. A backoff sleep can hang up to 8s before exiting via seq-check. Factor a shared `_cancelInFlightSave` helper.
- **[S4]** `saveBackoffRef` timer callback (useProjectEditor.ts:203-209) unconditionally nulls `saveBackoffRef.current`, which could clobber a newer attempt's handle in re-entrant save scenarios. Capture the timer/resolve in locals and clear only if `saveBackoffRef.current?.timer === timer`.
- **[S5]** `useEditorMutation.run` sets `inFlightRef.current = true` synchronously before any await — correct today, but a latent hazard if someone later inserts an await before the ref-set. Add a comment warning against this. `useEditorMutation.ts:73-75`.
- **[S6]** `reloadActiveChapter` returns `true` both for "refreshed" and "intentionally skipped" (expectedChapterId mismatch). Consider `{ok, skipped}` for clarity; today callers can't distinguish. `useProjectEditor.ts:297-342`.
- **[S7]** Comment at `utils/findReplaceErrors.ts:21-22` implies a non-2xx BAD_JSON path is possible; in `api/client.ts:59-77`, non-ok body-parse failures silently become `code: undefined`. Either fix the comment or propagate BAD_JSON on non-2xx parse failure too.
- **[S8]** `handleReplaceOne` re-searches after SCOPE_NOT_FOUND (EditorPage.tsx:563-571). If that re-search fails, `findReplace.error` is set concurrently with the `actionError` set immediately after. Intentional per "separate regions" but worth a comment; consider a final `findReplace.clearError()` to keep only the mutation error visible.
- **[S9]** Tests cover mutate-throw re-enables editor (test.tsx:186) but not "cache was NOT cleared" via an explicit positive assertion on the mutate-throw path. Actually present at test.tsx:184 — verified; no fix needed. (Listed here only as a note that the discipline is in place.)
