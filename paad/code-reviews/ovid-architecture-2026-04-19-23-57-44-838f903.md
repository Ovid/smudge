## Executive Summary

Phase 4b.1 (editor orchestration helper) is tightly engineered and follows the design closely — the core `useEditorMutation` hook encodes save-pipeline invariants 1–3 by construction, and its unit test file is thorough. Review surfaces one Critical issue (413 cache-wipe that violates invariant #3), six Important issues (mostly UX double-banner and unmount-race edges in `useProjectEditor`), and four Suggestions (code duplication and idiom normalization). No security findings.

## Critical Issues

### [C1] `handleSave` 4xx path wipes cache on 413 "content too large"
- **File:** `packages/client/src/hooks/useProjectEditor.ts:212-249`
- **Bug:** Any 4xx classified by `mapSaveError` falls through to `clearCachedContent(savingChapterId)` and `latestContentRef.current = null` at L246-248. The justifying comment claims "the server preserves the previous good content" — true for `VALIDATION_ERROR` after the handler parsed the body, but 413 is rejected by the body-size guard *before* the chapter handler runs. The server still holds the old content; the user's typed content exists only in the cache until this branch wipes it.
- **Impact:** On a single 413 (e.g. user embeds a large image or writes past the per-row limit), the local draft is deleted and the reload path will rehydrate the server's stale pre-413 content. Direct violation of CLAUDE.md save-pipeline invariant #3 ("Cache-clear happens after server success, never before") — 413 is not server success.
- **Suggested fix:** Narrow the cache-clear to the `VALIDATION_ERROR` branch (server truly cannot store a malformed payload, so wiping avoids a persistent error loop). For 413, preserve the cache so the user can trim and retry, and surface `STRINGS.editor.saveFailedTooLarge` without data destruction.
- **Confidence:** High
- **Found by:** Error Handling

## Important Issues

### [I1] Double-banner on `handleReplaceOne` error paths
- **File:** `packages/client/src/pages/EditorPage.tsx:534-542`, `:577-585`
- **Bug:** Both the SCOPE_NOT_FOUND (404) branch and the `replaced_count === 0` branch call `findReplace.search(slug)` as a side-effect refresh. If that re-search itself fails, `useFindReplaceState` sets a panel-local `error` alongside the already-set `actionError` from the replace — two banners for one user action, often with slightly different wording (e.g. `searchScopeNotFound` vs `replaceScopeNotFound`).
- **Impact:** Confusing UX; user cannot tell which message describes what actually happened.
- **Suggested fix:** Clear `findReplace.error` via the new `clearError()` method after the refetch, or gate `setActionError` on the refetch's success so only one banner speaks per click.
- **Confidence:** Medium
- **Found by:** Logic & Correctness, Error Handling

### [I2] 4xx break in `handleSave` clears cache without seq check
- **File:** `packages/client/src/hooks/useProjectEditor.ts:245-249`
- **Bug:** After breaking out of the save loop on 4xx, `clearCachedContent(savingChapterId)` and `latestContentRef.current = null` run unconditionally. The subsequent `setSaveStatus`/`setSaveErrorMessage` block at L251-254 IS gated by `activeChapterRef.current?.id === savingChapterId`, but the cache-clear is not. If the user switched chapters just before the 4xx response landed, the old chapter's cache is wiped based on a stale response.
- **Impact:** Corner-case draft loss when a chapter switch races a 4xx on the previous chapter.
- **Suggested fix:** Wrap the cache-clear in the same `seq === saveSeqRef.current` guard already used at L160/L171 — if seq changed, another path owns the cache for this chapter.
- **Confidence:** Medium
- **Found by:** Error Handling, Concurrency & State

### [I3] `handleDeleteChapter` secondary GET swallows errors silently
- **File:** `packages/client/src/hooks/useProjectEditor.ts:400-408`
- **Bug:** After a successful delete, `api.chapters.get(first.id)` fetches the next chapter to activate. Any failure (network, transient 500) is caught with `catch {}` and the handler falls through to `setActiveChapter(null)`, `setChapterWordCount(0)`.
- **Impact:** User sees the empty-state UI ("Add chapter") as if the project had no chapters, even though other chapters still exist in `remaining` and the sidebar. No banner, no log. The failure is invisible.
- **Suggested fix:** Call `onError?.(STRINGS.error.loadChapterFailed)` in the catch (and/or `console.warn`) so the user learns something went wrong. Ideally also keep `activeChapter` set to `remaining[0]` so the sidebar highlights it — the user can click to retry.
- **Confidence:** High
- **Found by:** Error Handling

### [I4] `useEditorMutation` finally swallows `setEditable(true)` errors without signal
- **File:** `packages/client/src/hooks/useEditorMutation.ts:151-158`
- **Bug:** The inner `try { editor?.setEditable(true) } catch {}` swallows any throw silently. A TipTap mid-remount throw here would leave the editor stuck non-editable with no dev or production signal.
- **Impact:** Silent failure state; CLAUDE.md's zero-warnings testing policy favors explicit spy-suppressed `console.warn` over silent drops so surprise states are noticed at least once during development.
- **Suggested fix:** Replace `catch {}` with `catch (err) { console.warn("useEditorMutation: failed to re-enable editor", err); }`. Tests that deliberately trigger the branch spy-suppress and assert, per CLAUDE.md §Testing Philosophy.
- **Confidence:** Medium
- **Found by:** Error Handling

### [I5] `reloadActiveChapter` has no unmount/abort guard
- **File:** `packages/client/src/hooks/useProjectEditor.ts:321-366`
- **Bug:** `api.chapters.get(current.id)` is guarded by `selectChapterSeqRef` but not by a mount flag or AbortController. The unmount cleanup effect only calls `cancelInFlightSave` (which bumps `saveSeqRef`), so a reload GET in flight during unmount still resolves and calls `setActiveChapter`/`setChapterWordCount`/`setChapterReloadKey`/`setError`.
- **Impact:** React "state update on unmounted component" warnings in dev; violates the isolation the save path already enforces.
- **Suggested fix:** Either add a `cancelled` ref updated in the unmount effect (mirroring the loadProject effect at L78) and check it after await, or bump `selectChapterSeqRef` from the unmount effect too.
- **Confidence:** Medium
- **Found by:** Concurrency & State

### [I6] Hand-composed snapshot flows don't latch `inFlightRef`
- **File:** `packages/client/src/pages/EditorPage.tsx:1128-1178` (SnapshotPanel `onView`), `:1179-1189` (`onBeforeCreate`)
- **Bug:** Both handlers probe `mutation.isBusy()` at entry but do not set any in-flight latch themselves. Between the `isBusy` check and the follow-up awaits (`flushSave`, `viewSnapshot`), nothing prevents a concurrent `mutation.run()` (e.g. from a keyboard shortcut) from starting — `run()`'s inFlightRef is only set inside the hook. React's serial event loop mitigates this in practice, but the asymmetry is a drift hazard.
- **Impact:** Theoretically a double-flush/abort race; low risk today. The invariant "external callers refuse while mutation is in-flight" isn't symmetrically enforced for the panel's own re-entry.
- **Suggested fix:** Expose `reserve()`/`release()` (or a shared busy token) from `useEditorMutation` so hand-composed paths can participate in the same latch.
- **Confidence:** Medium
- **Found by:** Concurrency & State

## Suggestions

- **[S1]** `executeReplace` (`EditorPage.tsx:341-432`) and `handleReplaceOne` (`:487-596`) duplicate banner-clear + `mutation.run` + stage-branching logic. Extract a shared `performReplace(scopePayload, options)` so fixes land once, not twice (Contract).
- **[S2]** Cache-clear happens in both `useEditorMutation.run()` (L112) and `reloadActiveChapter` (`useProjectEditor.ts:336`). Idempotent today; drift risk tomorrow. Centralize or add a prominent cross-reference comment (Contract, Concurrency).
- **[S3]** `useEditorMutation` treats `flushSave` returning `undefined` as success via `=== false` check; external callers use `?? true`. Normalize on one idiom for documentation value (Contract, Error Handling).
- **[S4]** `findReplaceErrors.ts:55, 93` maps NETWORK (status 0) and 5xx to the same `replaceFailed`/`searchFailed` strings. Consider branching `err.status === 0` to a "check your connection" variant so users can self-diagnose (Error Handling).
