## Executive Summary

The `useEditorMutation` hook itself is carefully structured and its unit test file comprehensively covers the contract. However, the branch goes well beyond the Phase 4b.1 design — it absorbs `handleSave` changes, a new `ReloadOutcome` return type, persistent-lock-banner wiring, 2xx BAD_JSON handling, and error-mapping additions that the plan explicitly assigned to later phases (4b.2/4b.3/4b.5). Two realistic critical issues emerged: the BAD_JSON / `possibly_committed` paths leave stale drafts in the localStorage cache that can silently revert the server-committed mutation on refresh, and `switchToView` can unconditionally re-enable an editor that the lock banner claims is read-only. Several important edge-case gaps around unprotected `setEditable(false)` calls, unwrapped `await findReplace.search`, and inconsistent snapshot-list refreshes round out the picture.

## Critical Issues

### [C1] BAD_JSON / possibly_committed paths leak stale draft caches
- **Files:** `packages/client/src/pages/EditorPage.tsx:316-328` (restore `possibly_committed`), `:344-352` (restore `unknown`), `:503-515` (`executeReplace` 2xx BAD_JSON), `:674-686` (`handleReplaceOne` 2xx BAD_JSON)
- **Bug:** In all four ambiguous-commit branches, the mutate callback throws (via `RestoreFailedError` or an `ApiRequestError(BAD_JSON, 2xx)`) before returning a `MutationDirective`, so the hook's `clearAllCachedContent(directive.clearCacheFor)` at `useEditorMutation.ts:138-140` never runs. The caller then sets a persistent lock banner and re-disables the editor, but never clears the draft cache for chapters the server probably committed against.
- **Impact:** The banner tells the user to refresh. On refresh, `useProjectEditor` prefers cached draft over server content (useProjectEditor.ts:131, :355), so the pre-mutation cached draft re-hydrates. The first keystroke or debounced auto-save can then PATCH the pre-mutation text back over the server-committed replace/restore — the exact data-loss mode the lock banner was introduced to prevent. Same concern applies if the user navigates to another affected chapter in the same session.
- **Suggested fix:** In each branch, clear the best-guess affected chapters' caches. For restore `possibly_committed`/`unknown`: `clearCachedContent(activeChapter.id)` (we know the target). For replace BAD_JSON (unreadable body means `affected_chapter_ids` is unavailable): conservatively clear at least the active chapter's cache, or call `clearAllCachedContent(project.chapters.map(c => c.id))` to cover the project-wide scope.
- **Confidence:** High
- **Found by:** Logic & Correctness (conf 85)

### [C2] `switchToView` can silently re-enable a locked editor
- **File:** `packages/client/src/pages/EditorPage.tsx:815-873`
- **Bug:** After a successful flush, line 869 calls `editorRef.current?.setEditable(true)` unconditionally. If `editorLockedMessage` is non-null (a prior `stage:"reload"` or `possibly_committed` outcome locked the editor), any view switch — Preview, Dashboard, Editor — re-enables typing while the "refresh the page" banner still claims read-only. Additionally, the `catch { setEditable(true); throw err }` at line 852 re-throws a flushSave rejection that `handleSelectChapterWithFlush` (line 918-923) doesn't catch, letting it surface as an unhandled promise rejection through the sidebar click path.
- **Impact:** User enters Preview while locked, returns to Editor (which re-mounts fresh with `editable=true`), the banner persists but the editor is now writable. A keystroke schedules auto-save, which PATCHes pre-mutation content over the server-committed change — the I1 invariant the hook was explicitly architected to enforce. The unhandled-rejection hazard is latent today (`Editor.tsx`'s `flushSave` catches internally) but one future change away from breaking.
- **Suggested fix:** Gate the re-enable at line 869 on `editorLockedMessageRef.current === null`; wrap `handleSelectChapterWithFlush` (or `switchToView`) in a try/catch that degrades to a banner rather than re-throwing. Ideally, expose the hook's `isLocked` predicate through a shared helper both sites consult.
- **Confidence:** High (Logic), Medium (Error Handling)
- **Found by:** Logic & Correctness (conf 70), Error Handling & Edge Cases (conf 65)

## Important Issues

### [I1] `cancelPendingSaves()` / `markClean()` not wrapped in per-stage try/catch
- **File:** `packages/client/src/hooks/useEditorMutation.ts:130-131`
- **Bug:** The hook wraps `setEditable(false)`, `flushSave`, and `await mutate()` in per-stage try/catches so synchronous throws produce typed `MutationResult` returns. `cancelPendingSaves()` and `markClean()` are bare — a synchronous throw propagates out as a rejected Promise. Every caller does `await mutation.run(...)` without a try/catch and relies on the discriminated-result shape; an uncaught rejection bypasses the entire stage-routing contract (no banner, no editor-state cleanup except the `finally`).
- **Impact:** Today it is safe by coincidence of the implementations — Editor's `markClean` is a ref/timer touch, `cancelPendingSaves` in `useProjectEditor.ts` is a ref+setState. A future TipTap upgrade or `useProjectEditor` refactor could silently reintroduce the very class of bug the hook was built to prevent.
- **Suggested fix:** Wrap both in try/catch and return `{ ok: false, stage: "flush", error }` (or a new dedicated stage) if either throws.
- **Confidence:** Medium
- **Found by:** Error Handling & Edge Cases (conf 70)

### [I2] Unprotected `setEditable(false)` calls outside the hook
- **File:** `packages/client/src/pages/EditorPage.tsx:326, 350, 397, 844`
- **Bug:** The hook deliberately wraps `editor.setEditable(false)` in try/catch (`useEditorMutation.ts:113-117`) because TipTap can throw synchronously during remount. Four call sites in `EditorPage.tsx` call `editorRef.current?.setEditable(false)` unwrapped — the BAD_JSON handler in `handleRestoreSnapshot` (326, 350), `finalizeReplaceSuccess` (397), and `switchToView` (844). The `onView` site at 1338 is already protected.
- **Impact:** A TipTap mid-remount throw at any of these sites rejects the calling async function. `handleRestoreSnapshot` and `finalizeReplaceSuccess` are both awaited by `void executeReplace(...)` and `void handleRestoreSnapshot(...)` invocations, producing unhandled rejections and skipping subsequent refresh calls.
- **Suggested fix:** Hoist into a `safeLock(editorRef)` util used by both caller and hook, or wrap each site in try/catch.
- **Confidence:** Medium
- **Found by:** Contract & Integration (conf 70)

### [I3] `await findReplace.search(slug)` failures silently suppress success banner
- **File:** `packages/client/src/pages/EditorPage.tsx:399-411` (`finalizeReplaceSuccess`)
- **Bug:** The awaited `findReplace.search(slug)` runs before the `snapshotPanelRef.refreshSnapshots()`, `refreshSnapshotCount()`, and `setActionInfo(replaceSuccess(N))` calls. If the search refresh rejects (network blip between replace success and refresh), none of those run.
- **Impact:** The user just committed a replace; the success banner is their primary signal it worked. A secondary search failure (unrelated to the replace) silently erases confirmation plus the snapshot-count bump, and any reload-failed path loses its banner context too.
- **Suggested fix:** Wrap `await findReplace.search(slug)` in try/catch (log or warn but continue), or fire-and-forget so the success signals don't depend on the search's outcome.
- **Confidence:** Medium
- **Found by:** Error Handling & Edge Cases (conf 70)

### [I4] `handleReplaceOne` 404 branch — unguarded `findReplace.search(slug)` before banner
- **File:** `packages/client/src/pages/EditorPage.tsx:696-701`
- **Bug:** `await findReplace.search(slug)` runs before `mapReplaceErrorToMessage(err)` / `setActionError(msg)`. If the search rejects, the subsequent banner-setting is skipped entirely — user sees no error feedback despite a 404.
- **Impact:** Silent failure on compound errors. Additionally, the post-run portion of the handler runs outside the hook's busy guard, admitting the race in [I5].
- **Suggested fix:** Same as I3 — wrap the search call.
- **Confidence:** Medium
- **Found by:** Error Handling & Edge Cases (conf 60)

### [I5] `handleReplaceOne` post-`run()` UI side effects race with a second click
- **File:** `packages/client/src/pages/EditorPage.tsx:630-642, 696-701`
- **Bug:** After `mutation.run()` resolves, the hook's `finally` clears `inFlightRef`. The caller's subsequent `await findReplace.search(slug)` + `setActionError(...)` runs with no busy guard. A rapid second Replace click enters a new `mutation.run()` uncontested; when its `finalizeReplaceSuccess` posts "Replaced N occurrences" while the first click's trailing `setActionError("matchNotFound")` / `setActionError(mapReplaceErrorToMessage(err))` lands, both banners display simultaneously — one claiming failure, one claiming success for the same logical operation.
- **Impact:** Confusing UX, persistent ghost error banner on a successful replace.
- **Suggested fix:** Keep a caller-level in-flight flag spanning the entire async callback (not just `run()`), or move the search+banner work inside the mutate callback.
- **Confidence:** Medium
- **Found by:** Concurrency & State (conf 70)

### [I6] Restore error branches inconsistent about refreshing the snapshot list
- **File:** `packages/client/src/pages/EditorPage.tsx:330-352`
- **Bug:** `possibly_committed` (line 327) and `unknown` (line 351) both call `snapshotPanelRef.current?.refreshSnapshots()`. The three dismissible-banner branches — `corrupt_snapshot`, `cross_project_image`, `not_found` — do not. The `not_found` case is the sharpest mismatch: the snapshot row is stale, the user can keep clicking it to loop the same error. The sibling `handleReplaceOne` 404 path does refresh its result set for exactly this reason.
- **Impact:** Stale snapshot row remains clickable, user loops through the same 404.
- **Suggested fix:** Refresh the snapshot panel in all error branches, or at minimum in `not_found`.
- **Confidence:** Medium
- **Found by:** Logic & Correctness (conf 65)

### [I7] `restoreSnapshot` missing ABORTED error handling
- **File:** `packages/client/src/hooks/useSnapshotState.ts:267-289`
- **Bug:** `viewSnapshot` handles `err.code === "ABORTED"` explicitly (line 207) as a silent no-op. `restoreSnapshot`'s error ladder does not — ABORTED falls through to the network branch and surfaces as `reason: "network"` → "check your connection" banner.
- **Impact:** Low today (no path aborts restore), but the asymmetry is a future-fragility concern and the banner copy would be actively misleading if ABORTED ever fired.
- **Suggested fix:** Mirror the ABORTED branch from viewSnapshot.
- **Confidence:** Medium
- **Found by:** Error Handling & Edge Cases (conf 60)

## Suggestions

- **[S1]** Stage-dispatch boilerplate (`if (result.stage === "busy") ... "flush" ...`) duplicated across all three migrated callers (`EditorPage.tsx:292-298, 474-482, 650-657`) — extract a helper to reduce future drift. (Contract & Integration, conf 70)
- **[S2]** `if (mutation.isBusy()) { setActionInfo(...); return; }` duplicated across ~7 panel-exclusivity handlers (`EditorPage.tsx:199-235, 817-826, 885-910`) — extract a `guardBusy` helper or `useBusyGuard` returning a wrapped callback. (Contract & Integration, conf 80)
- **[S3]** `cancelPendingSaves` has unstable function identity (inline arrow in `useProjectEditor.ts:672-680`'s return object). The hook compensates with the latest-ref pattern, but future consumers that close over it could latch a stale copy. Wrap in `useCallback`. (Contract & Integration, conf 75)
- **[S4]** `handleRestoreSnapshot`'s `possibly_committed` branch briefly re-enables the editor (hook's `finally`) before re-disabling it in the caller (`EditorPage.tsx:315-328`). The window is synchronous with no paint between, so the invariant is held in practice — but the code would be more robust if the hook could surface a "caller wants to lock on mutate-error" hook option that suppresses the `finally`'s re-enable. (Contract & Integration, conf 65)
- **[S5]** `onView`'s catch branch conflates all throws into `{ok: false, reason: "save_failed"}` (`EditorPage.tsx:1356-1369`). A TipTap remount throw would be misreported. Minor — contract shape preserved. (Error Handling, conf 60)
