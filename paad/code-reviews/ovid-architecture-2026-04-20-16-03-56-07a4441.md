## Executive Summary

The `useEditorMutation` hook correctly enforces save-pipeline invariants 1–4 for the three migrated mutation paths (snapshot restore, executeReplace, handleReplaceOne), and specialists uncovered no Critical data-loss paths that survived verification. The highest-value findings cluster around **integration gaps at the EditorPage boundary**: three sidebar/toolbar handlers (reorder, status, rename) were left unguarded while every sibling was gated; two panel-toggle paths exit snapshot view without re-enabling the editor; and the restore error branches don't refresh the snapshot count badge the way the replace equivalents do. One hardening finding on the hook itself (optional `reloadChapterId`) is worth addressing because I2 protection is currently enforced by caller discipline rather than by construction.

## Important Issues

### [I1] Snapshot count badge goes stale on `possibly_committed` / `unknown` restore branches

- **File:** `packages/client/src/pages/EditorPage.tsx:346-395`
- **Bug:** After a 2xx-BAD_JSON or `unknown`-error restore, the server has almost certainly committed a new restore + its auto-snapshot, but the handler only calls `snapshotPanelRef.current?.refreshSnapshots()`. That method is a no-op when the snapshot panel is closed (the typical state when restore is triggered from `SnapshotBanner`), and there is no fallback `refreshSnapshotCount()` like `finalizeReplaceSuccess` uses (line 464).
- **Impact:** Toolbar badge shows stale snapshot count after an opaque-but-likely-committed restore. Users can't see the new auto-snapshot exists.
- **Suggested fix:** Call `refreshSnapshotCount()` after `snapshotPanelRef.current?.refreshSnapshots()` in both the `possibly_committed` branch (~line 367) and the `unknown` fallback (~line 394). The `ok` and `reload` branches are fine — `useSnapshotState.restoreSnapshot`'s success path updates the count internally.
- **Confidence:** High (80)
- **Found by:** Error Handling & Edge Cases specialist

### [I2] `reloadActiveChapter: true` without `reloadChapterId` silently bypasses the I2 chapter-switch guard

- **File:** `packages/client/src/hooks/useEditorMutation.ts:10-18`; `packages/client/src/hooks/useProjectEditor.ts:388-390`
- **Bug:** `MutationDirective.reloadChapterId` is `string | undefined`. `reloadActiveChapter`'s mismatch guard (`current.id !== expectedChapterId`) is only armed when `expectedChapterId !== undefined`. A caller that sets `reloadActiveChapter: true` but omits `reloadChapterId` wipes whatever chapter happens to be active at response time — the exact draft-overwrite hazard the I2 plan cites.
- **Impact:** Today all three callers supply `reloadChapterId` when `reloadActiveChapter` is true, so no current exploit. Future callers will not notice the trap — and CLAUDE.md claims the hook enforces invariants 1–4 "by construction".
- **Suggested fix:** Replace the loose shape with a discriminated union: `{ reloadActiveChapter: false } | { reloadActiveChapter: true; reloadChapterId: string }`. The typechecker then forces every future caller to supply the chapter id.
- **Confidence:** High (70)
- **Found by:** Logic & Correctness, Contract & Integration (overlap — same root cause)

### [I3] Lock banner may not clear on a successful `run()` whose reload skipped the active chapter

- **File:** `packages/client/src/hooks/useEditorMutation.ts:182-223`; `packages/client/src/pages/EditorPage.tsx:875-877`
- **Bug:** The banner-clearing `useEffect` runs on `[activeChapter?.id, chapterReloadKey]`. If a later successful `run()` does not remount the Editor — e.g. project-scope replace whose `affected_chapter_ids` does not include the currently-active chapter, so `reloadActiveChapter` is false and `chapterReloadKey` isn't bumped — the banner (and the caller-enforced `isLocked` gate that keeps `setEditable(false)`) persists.
- **Impact:** User-visible editor stays read-only with a stale "refresh the page" banner even though a subsequent replace has completed successfully. No path to unlock except a chapter switch.
- **Suggested fix:** Clear `editorLockedMessage` at the end of every successful `run()` — either via a `clearLock` callback on the hook's args, or in each caller's `if (result.ok)` branch (`finalizeReplaceSuccess` with `reloadFailed=false` should also set `setEditorLockedMessage(null)`).
- **Confidence:** Medium (65)
- **Found by:** Logic & Correctness specialist

### [I4] Reorder, status-change, and rename handlers are unguarded against in-flight mutations

- **File:** `packages/client/src/pages/EditorPage.tsx:1254` (`onReorderChapters={handleReorderChapters}`) and `908-922` (`handleStatusChangeWithError`, `handleRenameChapterWithError`)
- **Bug:** Every other sidebar entry point was gated with `isActionBusy()` (create at 1013, delete at 1021, trash at 1032, select-with-flush via `switchToView`), but these three slip past. `handleReorderChapters` (useProjectEditor.ts:502-521) does `setProject(...)` that rebuilds the chapter list from pre-mutation word counts; status/rename PATCH the same chapter row the in-flight replace is writing.
- **Impact:** Reorder during an in-flight replace interleaves optimistic word-count state, pinning other chapters' counts to pre-replace values until a full project reload. Status/rename PATCH can race a replace's PATCH on the same chapter row.
- **Suggested fix:** Wrap all three in an `isActionBusy()` guard that sets `actionInfo = STRINGS.editor.mutationBusy`, same pattern as the other gated handlers (see `handleCreateChapterGuarded` at line 1003-1012).
- **Confidence:** High (72)
- **Found by:** Concurrency & State specialist

### [I5] Panel toggles that exit snapshot view don't restore `editable=true`

- **File:** `packages/client/src/pages/EditorPage.tsx:225-249`
- **Bug:** `handleToggleReferencePanel` (line 232) and `handleToggleFindReplace` (line 246) call `exitSnapshotView()` unconditionally when opening. The `onView` callback at line 1475 calls `editorRef.current?.setEditable(false)` on entry to snapshot view and only re-enables on failure paths. The success path leaves the editor disabled, relying on `SnapshotBanner`'s "Back to editing" action to bring the user back (via `safeSetEditable(editorRef, true)`). When the user is viewing a snapshot and clicks the reference/find-replace toolbar button, `exitSnapshotView()` runs without an Editor remount (same chapter, same `chapterReloadKey`), and nothing re-enables the editor.
- **Impact:** Editor stays `editable=false` silently. Typing does nothing with no visible explanation.
- **Suggested fix:** After `exitSnapshotView()` in both toggle handlers, call `safeSetEditable(editorRef, true)` (respecting `editorLockedMessageRef.current === null`, following the `switchToView` pattern at line 995).
- **Confidence:** High (72)
- **Found by:** Concurrency & State specialist

### [I6] Restore `not_found` leaves the stale "Viewing snapshot" banner onscreen

- **File:** `packages/client/src/pages/EditorPage.tsx:374-404`; `packages/client/src/hooks/useSnapshotState.ts:273-302`
- **Bug:** When restore fails with reason `not_found` (snapshot gone from server), the handler calls `setActionError(...)` and `snapshotPanelRef.current?.refreshSnapshots()` but never exits snapshot view. `viewingSnapshot` state stays truthy; `SnapshotBanner` with the (now-gone) snapshot's date/label remains on screen with a broken Restore button.
- **Impact:** User sees "Viewing snapshot from <date>" + "Restore" button, clicks Restore, gets the same 404 banner indefinitely.
- **Suggested fix:** In the `not_found` branch, call `exitSnapshotView()` before (or after) `setActionError(...)`.
- **Confidence:** High (75)
- **Found by:** Contract & Integration specialist
