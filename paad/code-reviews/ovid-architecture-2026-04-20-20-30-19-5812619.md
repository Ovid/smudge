## Executive Summary

Phase 4b.1 extracts a new `useEditorMutation` hook that encapsulates the save-pipeline invariants across `handleRestoreSnapshot`, `executeReplace`, and `handleReplaceOne`. The hook itself is rigorously constructed; invariants 1–4 are enforced by construction in the `run()` happy path. One Critical data-loss pathway remains: `safeSetEditable` silently swallows TipTap synchronous throws during lock-convergence paths, leaving the editor writable while the persistent "refresh the page" banner claims read-only — a subsequent keystroke's auto-save PATCH reverts the server-committed mutation. Four Important findings cover the same-class hole at the hook-side mid-remount re-lock, an ungated Ctrl+S flushSave external entry point, a missing `markClean` in `SnapshotPanel.onBeforeCreate`, and a project-scope BAD_JSON branch that over-wipes drafts in chapters the server never touched. Confidence in the refactor is high overall; the residual bugs cluster at lock-convergence edge cases TipTap's remount lifecycle can expose.

## Critical Issues

### [C1] `safeSetEditable` swallows throws during lock-convergence — banner says locked, editor is writable
- **File:** `packages/client/src/utils/editorSafeOps.ts:15-24`; caller sites `packages/client/src/pages/EditorPage.tsx:366`, `:405`, `:456`, `:528`
- **Bug:** `safeSetEditable(editorRef, false)` catches any TipTap synchronous throw and only `console.warn`s. Callers use it as the final step that converts the editor to read-only when raising the persistent `editorLockedMessage` banner. If the `setEditable(false)` throws during a mid-remount window, the editor remains editable while the banner claims "refresh the page — editor is locked."
- **Impact:** User keystrokes in this window flip `dirtyRef=true` and schedule auto-save PATCHes. The server-committed restore or replace is silently reverted — the exact save-pipeline invariant 3 violation the entire Phase 4b.1 refactor exists to prevent. This is the highest-severity bug on the branch: a keystroke after a "successful" replace-all can revert the whole operation with no user-visible signal.
- **Suggested fix:** Split the helper into `safeSetEditable` (swallow, for best-effort enable-after-success paths) and `requireSetEditable` (re-throw on failure, for lock-convergence). The four lock-convergence callers should use the strict variant and surface a distinct terminal-state banner on throw. Minimum: make `safeSetEditable` return a boolean so lock-convergence callers can detect failure and escalate.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

## Important Issues

### [I1] `useEditorMutation` mid-remount re-lock catches throws and proceeds to cache-clear + reload
- **File:** `packages/client/src/hooks/useEditorMutation.ts:171-186`
- **Bug:** When `editorAfterMutate !== editor`, the code calls `setEditable(false)`, `markClean()`, and `cancelPendingSaves()` on the newly-mounted editor inside a try/catch that only `console.warn`s on throw. Execution then proceeds to `clearAllCachedContent(directive.clearCacheFor)` and (if requested) `reloadActiveChapter`. If the re-lock threw, the fresh editor starts editable=true by TipTap default and the cache for affected chapters has already been wiped.
- **Impact:** Same invariant-3 failure shape as C1, internal to the hook. User keystrokes during the reload-GET window PATCH stale pre-mutation content back over the just-committed server change, and the localStorage safety net for those chapters was cleared by `clearAllCachedContent` on the success assumption. Data loss window is the duration of the reload GET (hundreds of ms to several seconds on slow network).
- **Suggested fix:** On catch, promote to `stage: "reload"` (server committed) and return early with the lock banner — the same treatment as a reload failure — rather than proceeding to cache-clear / reload as if the re-lock succeeded.
- **Confidence:** High
- **Found by:** Security, Error Handling & Edge Cases

### [I2] Ctrl+S flushSave is an external entry point NOT gated on `mutation.isBusy()`
- **File:** `packages/client/src/hooks/useKeyboardShortcuts.ts:107`; prop wiring `packages/client/src/pages/EditorPage.tsx:1247`
- **Bug:** The Ctrl+S handler calls `editorRef.current?.flushSave()` directly without consulting `mutation.isBusy()`. The design doc §API (lines 117-121) explicitly names `isBusy()` as the probe for "external flushSave entry points (chapter switch, snapshot view, snapshot create) to refuse hand-composed setEditable/flushSave/cancelPendingSaves sequences while a mutation is mid-flight." Ctrl+S is another such entry and it is ungated.
- **Impact:** A Ctrl+S during a mid-flight `mutation.run` aborts the mutation's in-flight save `AbortController` (via `handleSave`'s save-abort churn), flipping the hook's stage reporting from clean "flush"/"mutate" to an inappropriate branch, and potentially races two PATCHes for the same chapter. The migrated snapshot-view and snapshot-create paths correctly gate on `isActionBusy()`; Ctrl+S is the one ungated escape.
- **Suggested fix:** Wrap the prop at the EditorPage wiring: `flushSave: () => { if (mutation.isBusy()) return; return editorRef.current?.flushSave(); }`.
- **Confidence:** High
- **Found by:** Contract & Integration, Error Handling

### [I3] `SnapshotPanel.onBeforeCreate` flushes without `markClean()` — debounce timer survives
- **File:** `packages/client/src/pages/EditorPage.tsx:1679-1701`
- **Bug:** `onBeforeCreate` calls `flushSave()` then `cancelPendingSaves()` but neither calls `setEditable(false)` nor `markClean()`. `cancelPendingSaves` clears useProjectEditor-level state but does not clear the Editor component's internal `debounceTimerRef`. A keystroke between `flushSave` resolving and the snapshot create completing can set `dirtyRef=true` on the editor and schedule a new debounced PATCH that races the snapshot create.
- **Impact:** Save-pipeline invariant 1 violation: a server call (snapshot create) that depends on a clean editor fires while a concurrent keystroke schedules an auto-save. The snapshot captured server-side won't reflect post-save state; the auto-save may or may not land before the snapshot commit. Narrow race window but real. `onView` at line 1641 is protected by its prior `safeSetEditable(editorRef, false)`; `onBeforeCreate` has no analogous gate.
- **Suggested fix:** Add `editorRef.current?.markClean()` after `cancelPendingSaves()` in `onBeforeCreate`. Better: extract a shared `flushAndSettle(editorRef, cancelPendingSaves)` helper used by `onView`, `onBeforeCreate`, and `switchToView`.
- **Confidence:** Medium
- **Found by:** Concurrency & State, Error Handling

### [I4] Project-scope Replace-All BAD_JSON (2xx) wipes drafts in chapters the server never touched
- **File:** `packages/client/src/pages/EditorPage.tsx:678-682`
- **Bug:** On a 2xx response with BAD_JSON body for a project-scope replace, `clearAllCachedContent(project.chapters.map((c) => c.id))` clears localStorage drafts for every chapter. The response body was unreadable so `affected_chapter_ids` is unavailable, but the pessimistic wipe destroys drafts in chapters the server never wrote to.
- **Impact:** A user with unsaved work in chapter B who runs Replace-All project-wide (matching only chapter A) will lose chapter B's draft on BAD_JSON. The lock banner then directs the user to refresh, permanently destroying the work. Narrower than C1 (requires BAD_JSON, which is rare) but real.
- **Suggested fix:** On 2xx BAD_JSON for project-scope, clear only the active chapter's cache (the one the hook already flushed and whose content is known-consistent with server state post-flush), and rely on the lock-banner + refresh flow for reconciliation of the rest. Or scope to chapters whose cached draft timestamp predates the replace request.
- **Confidence:** Medium
- **Found by:** Security, Error Handling

### [I5] `restoreSnapshot` `reason === "unknown"` treats pre-send client throws as possibly-committed
- **File:** `packages/client/src/pages/EditorPage.tsx:446-456`
- **Bug:** The `unknown` reason branch runs `setEditorLockedMessage(...)`, `clearCachedContent(activeChapter.id)`, and `safeSetEditable(editorRef, false)` regardless of whether the throw was a client-side pre-send failure (e.g., TypeError in request construction) or a post-send ambiguity. `useSnapshotState.restoreSnapshot` maps any non-ApiRequestError throw to `reason: "unknown"` without distinguishing.
- **Impact:** A purely client-side failure (server never received the request) wipes the user's cached draft and permanently locks the editor pending a refresh. The opposite treatment in the `network` branch — which preserves the cache because the server is known not to have committed — is the correct policy; `unknown` should join it rather than `possibly_committed`.
- **Suggested fix:** In `useSnapshotState`, distinguish pre-send throws (TypeError before fetch left) from post-send throws. Route pre-send to `reason: "network"` / dismissible; keep `unknown` only for genuinely ambiguous cases.
- **Confidence:** Medium
- **Found by:** Error Handling, Security

## Suggestions

- **[S1]** `handleRestoreSnapshot` routes non-RestoreFailedError/non-RestoreAbortedError mutate-stage throws to the generic `STRINGS.snapshots.restoreFailed` banner; `executeReplace` / `handleReplaceOne` route non-ApiRequestError mutate-stage throws to `replaceFailed`. Both invite double-mutation if server committed. Add an "unknown-error" fallback mirroring C1 lock semantics. (`EditorPage.tsx:478, 690-691, 913-914`) — Error Handling
- **[S2]** `cancelPendingSaves` is an inline arrow in the `useProjectEditor` return object — fresh identity every render. Latest-ref in `useEditorMutation.ts:65-66` neutralizes the stale-closure risk, but wrap in `useCallback` for clarity and to reduce the ref-sync churn. (`useProjectEditor.ts:688`) — Concurrency & State
- **[S3]** `handleReorderChapters` has no sequence guard — concurrent reorders could race the setProject updates. Practically gated by the DnD UI but inconsistent with every other handler. (`useProjectEditor.ts:507-526`) — Concurrency & State
- **[S4]** `handleProjectSettingsUpdate` lacks an AbortController on its `api.projects.get(slug)` fetch — unmount-during-fetch leaks the connection and triggers a setState-on-unmounted warning. (`EditorPage.tsx:1212-1234`) — Concurrency & State
- **[S5]** Duplicated 2xx BAD_JSON lock-convergence shape across `handleRestoreSnapshot` (possibly_committed) and `finalizeReplaceSuccess` (replace callers). `finalizeReplaceSuccess` is already a shared helper; restore reimplements the pattern inline. Extract a `applyCommittedLock(chapterIds, lockMessage)` helper. (`EditorPage.tsx:385-416` vs `:499-547`) — Contract & Integration
- **[S6]** `handleReplaceOne` `replaced_count === 0` sets `matchNotFound` banner before `await findReplace.search(slug)` — user sees banner over stale rows until refresh completes. Swap ordering. (`EditorPage.tsx:847-850`) — Contract & Integration
- **[S7]** `reloadActiveChapter` bumps `saveSeqRef` directly rather than calling `cancelInFlightSave()`. Safe-by-call-graph today; fragile if a future caller invokes `reloadActiveChapter` without first calling `cancelPendingSaves`. (`useProjectEditor.ts:391`) — Concurrency & State
- **[S8]** Scope-creep: the `actionBusyRef` / `isActionBusy` latch is added across ~14 sibling handlers and a new `safeSetEditable` utility; none of this is in the design or plan. Each addition appears load-bearing (every I-number has a commit rationale), but the plan scope was three call sites. — Plan Alignment
- **[S9]** Scope-creep: the design's deliverable §5 states "running the existing EditorPageFeatures.test.tsx suite unmodified and passing"; the actual diff adds +1145 lines to that file. Documentation-vs-reality drift. — Plan Alignment
