The `useEditorMutation` refactor is tightly built and the save-pipeline invariants (markClean → lock → flushSave → mutate → cache-clear → reload) are enforced consistently at the three run()-routed callers. 19 rounds of prior review have scrubbed the obvious data-loss paths and `handleSaveLockGated` provides a strong defense-in-depth backstop. A secondary cluster covers closure-capture hazards during mid-flight project rename (E3) and one clear double-banner UX bug on busy snapshot creation (L6).

## Important Issues

### [I1] stage:"reload" contract violated by mid-remount re-lock bail
- **File:** `packages/client/src/hooks/useEditorMutation.ts:218-228`
- **Bug:** The bail returns `stage:"reload"` regardless of whether the caller's directive requested a reload, and regardless of whether the server commit actually needed one. Callers treat `stage:"reload"` as "server committed, reload GET failed" — that premise doesn't hold here.
- **Impact:** Root cause of issues I2 and I3 below. Callers end up running lock-banner / cache-wipe / editor-lock work for mutations that explicitly signaled "no reload needed".
- **Fix:** Either introduce a distinct stage (`stage:"relock_failed"`) or have the bail respect `directive.reloadActiveChapter === false` and return a softer result. A third option: check `directive.clearCacheFor.length === 0` as a proxy for "nothing to reconcile".
- **Confidence:** Medium
- **Found by:** Contract & Integration

### [I2] handleRestoreSnapshot stage:"reload" branch ignores staleChapterSwitch
- **File:** `packages/client/src/pages/EditorPage.tsx:368-402`
- **Bug:** If the mid-remount re-lock bail (I1) fires with `data.staleChapterSwitch: true`, this branch runs unconditionally. It sets the non-dismissible editor-lock banner, calls `clearCachedContent(activeChapter.id)` and `safeSetEditable(editorRef, false)` — but `activeChapter` is the chapter the user switched **to**, not the restored one. Locks and wipes an unrelated chapter.
- **Impact:** Narrow timing window (chapter switch during a restore whose new-editor re-lock throws), but the consequence is misleading UX and draft loss on an uninvolved chapter. `handleSaveLockGated` prevents a stale PATCH after the lock goes up, but the cache wipe happens first.
- **Fix:** Short-circuit the branch when `result.data.staleChapterSwitch === true`, OR fix at the hook layer per I1.
- **Confidence:** Medium
- **Found by:** Logic & Correctness

### [I3] 0-replace + mid-remount bail shows contradictory banners
- **File:** `packages/client/src/pages/EditorPage.tsx:904-912` + `packages/client/src/hooks/useEditorMutation.ts:218-228`
- **Bug:** `handleReplaceOne` returns `{clearCacheFor: [], reloadActiveChapter: false, data: resp}` when `replaced_count === 0`. If the mid-remount re-lock bail fires, hook returns `stage:"reload"`; the caller calls `finalizeReplaceSuccess({replacedCount: 0, reloadFailed: true})`, producing a persistent "refresh the page" lock banner alongside a "Replaced 0 occurrences" success banner for an operation that did nothing.
- **Fix:** Gate `finalizeReplaceSuccess`'s banner on `replacedCount > 0`, or fix at the hook layer per I1.
- **Confidence:** Medium
- **Found by:** Logic & Correctness

### [I4] isLocked predicate throw forces lockedByCaller=true, overriding reloadSucceeded
- **File:** `packages/client/src/hooks/useEditorMutation.ts:301-308`
- **Bug:** The conservative-on-throw fallback `lockedByCaller = true` unconditionally overrides `reloadSucceeded` / `reloadSuperseded`. On a reload-succeeded run whose `isLocked()` predicate throws, the editor stays `setEditable(false)` while `chapterReloadKey` clears the banner — reproducing the "looks editable but can't type" dead state the reloadSuperseded flag exists to prevent.
- **Impact:** Today's closure can't throw, but the code block's own comment warns about future flaky state. Cheap fix.
- **Fix:** `lockedByCaller = !reloadSucceeded && !reloadSuperseded;` in the catch branch.
- **Confidence:** Medium
- **Found by:** Logic & Correctness, Error Handling (both agents agreed)

### [I5] onBeforeCreate "busy" produces double contradictory banners
- **File:** `packages/client/src/pages/EditorPage.tsx:1724-1730` + `packages/client/src/components/SnapshotPanel.tsx:200-215`
- **Bug:** `onBeforeCreate` returns `false` when busy and sets `actionInfo = mutationBusy`. `SnapshotPanel.handleCreate` interprets `flushed === false` as failure and stamps `createError = "Unable to create snapshot. Save your unsaved changes and try again."`. User sees two contradictory banners at once.
- **Fix:** Distinguish busy-return from failure-return in the onBeforeCreate contract (e.g. discriminated `{ok, reason}`), and have the panel skip `createError` when `reason === "busy"`.
- **Confidence:** High
- **Found by:** Logic & Correctness

### [I6] finalizeReplaceSuccess uses stale closure slug after project rename during replace
- **File:** `packages/client/src/pages/EditorPage.tsx:540, 561` + `packages/client/src/hooks/useProjectTitleEditing.ts`
- **Bug:** `executeReplace`/`handleReplaceOne` close over `slug` via `useCallback`. `useProjectTitleEditing.saveProjectTitle` → `handleUpdateProjectTitle` is **not** gated by `isActionBusy`, so a user can rename the project during an in-flight replace. After the replace commits, `finalizeReplaceSuccess` calls `findReplace.search(oldSlug)` → 404 → `searchProjectNotFound` error banner pinned next to a successful replace.
- **Fix:** Gate `saveProjectTitle` behind `isActionBusy()` (consistent with every other chapter-modifying entry point), OR have `finalizeReplaceSuccess` read the live slug from a ref.
- **Confidence:** Medium
- **Found by:** Error Handling

### [I7] reloadActiveChapter uses bare saveSeqRef bump, leaks AbortController + backoff
- **File:** `packages/client/src/hooks/useProjectEditor.ts:391`
- **Bug:** `++saveSeqRef.current` short-circuits the retry loop but does not abort the in-flight AbortController nor clear the backoff timer. The sole current caller (`useEditorMutation`) runs `cancelPendingSaves` first, so it's covered today, but any future direct caller inherits a hazard.
- **Fix:** Replace with `cancelInFlightSave()` for consistency with `handleSelectChapter` / `handleDeleteChapter` / `handleCreateChapter`.
- **Confidence:** Medium
- **Found by:** Error Handling, Concurrency & State (both agents agreed)
