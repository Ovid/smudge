## Important Issues

### [I1] `useProjectEditor.ts` PR-scope deviation
- **File:** `packages/client/src/hooks/useProjectEditor.ts` (308 lines changed)
- **Bug:** The plan (`docs/plans/2026-04-19-editor-orchestration-helper-plan.md:1217-1225`, Task 16 Check 5) and design (`docs/plans/2026-04-19-editor-orchestration-helper-design.md:23`, Non-goals) both explicitly forbid behavioral changes to `useProjectEditor.ts` beyond non-behavioral comment changes. The branch nonetheless adds `cancelInFlightSave` (lines 59-70), `cancelInFlightSelect` (85-87), an unmount `useEffect` teardown (96-101), a `mapSaveError` helper (179-183), and changes the public return type of `reloadActiveChapter` to `Promise<ReloadOutcome>` where `ReloadOutcome = "reloaded" | "superseded" | "failed"`.
- **Impact:** CLAUDE.md one-feature rule violation. The branch now carries an editor-mutation extraction *and* a save-pipeline refactor, recreating exactly the kind of bundled PR that the snapshots/find-and-replace 16-round-review incident drove the rule for.
- **Suggested fix:** Either split the save-pipeline changes into a follow-up PR (preserving only the minimal `ReloadOutcome` change required to fix I5), or update plan/design retroactively to acknowledge the expanded scope and document the rationale.
- **Confidence:** High
- **Found by:** Plan Alignment

### [I2] Ctrl+S not gated on lock banner — flips save indicator to error
- **File:** `packages/client/src/pages/EditorPage.tsx:1305-1308` (Ctrl+S `flushSave`) and `:181-187` (`handleSaveLockGated`)
- **Bug:** While the editor lock banner is up, `handleSaveLockGated` returns `false`. `Editor.flushSave` interprets `false` as "save failed" and keeps `dirtyRef = true`. The Ctrl+S handler is gated on `isActionBusy()` only, NOT on `editorLockedMessageRef.current`. Pressing Ctrl+S during the lock window triggers `flushSave` → returns false → save banner flips to "error" without any actual server attempt.
- **Impact:** UX regression. No data loss (PATCH is correctly skipped), but the save indicator misrepresents state and can scare the user into a refresh that loses the lock-banner context.
- **Suggested fix:** Either gate Ctrl+S on `editorLockedMessageRef.current === null` (skip silently while locked), or have `handleSaveLockGated` `markClean()` before returning false so dirtyRef matches the "intentionally not saving" semantic.
- **Confidence:** High
- **Found by:** Logic & Correctness

### [I3] `useFindReplaceState` unmount leaks debounce timer
- **File:** `packages/client/src/hooks/useFindReplaceState.ts:79-84`
- **Bug:** The unmount `useEffect` cleanup aborts the in-flight search controller and nulls `searchAbortRef`, but does NOT `clearTimeout(debounceRef.current)`. The same hook's `closePanel` correctly clears the debounce timer at lines 137-140 — proving the omission is unintentional.
- **Impact:** When the editor unmounts (route change) while a debounced search is pending, the 300ms timer fires `search(slug)` after the hook is dead: bumps `searchSeqRef`, creates an uncleaned `AbortController`, fires the fetch (the new controller is never aborted by the unmounted hook), and the `finally` block calls `setLoading(false)` on an unmounted component. Dev-mode "state update on unmounted component" warning, prod-mode silent leak of the request.
- **Suggested fix:** In the unmount cleanup add `if (debounceRef.current) { clearTimeout(debounceRef.current); debounceRef.current = null; }`.
- **Confidence:** High
- **Found by:** Concurrency & State

### [I4] `handleUpdateProjectTitle` not gated by `isActionBusy()`
- **File:** `packages/client/src/hooks/useProjectEditor.ts:537-556`
- **Bug:** Every other editor-affecting entry point (create/delete/rename chapter, status change, reorder, settings update) is gated on `isActionBusy()`. `handleUpdateProjectTitle` is the lone unguarded entry point that mutates project routing state. If the user renames the project mid-replace, `projectSlugRef.current` and `project.slug` update immediately, while in-flight `executeReplace`/`handleReplaceOne` mutate callbacks at `EditorPage.tsx:632, 859` still hold the old `slug` closure for `api.search.replace(slug, …)`. Meanwhile `finalizeReplaceSuccess` at line 555 reads `slugRef.current` (the new slug), so the post-replace search refresh hits the new URL while the replace POST hits the old one.
- **Impact:** Asymmetric slug state. Common failure mode: replace POST 404s with `searchProjectNotFound` for an op the user meant to target the now-renamed project. Worst case: replace succeeds against old slug while refresh queries new slug, leaving search results inconsistent with what was just replaced.
- **Suggested fix:** Add `isActionBusy()` gate to `handleUpdateProjectTitle` (or to the saveProjectTitle entry in `useProjectTitleEditing`). Optionally also read slug from `slugRef.current` inside both mutate callbacks so replace and refresh always agree.
- **Confidence:** High
- **Found by:** Concurrency & State

## Suggestions

- **[S1]** `useProjectEditor.ts:577, 610` — `handleStatusChange` uses bare `catch {` without binding or logging; sibling handlers all `console.warn`. User-visible signal exists via `onError`, but developers lose the network-layer cause. Found by Error Handling.
- **[S2]** `useEditorMutation.ts:344-356` — finally's `setEditable(true)` swallows throws with `console.warn`; for `reloadActiveChapter: false` directives there is no follow-on remount, so editor stays read-only without a banner. Self-healing on next remount but transiently confusing. Found by Logic & Correctness.
- **[S3]** `useEditorMutation.ts:65-69` — latest-ref pattern assigns during render rather than in `useEffect` as the design at `design.md:158-163` specified. Likely a deliberate tightening (closes a same-tick gap), but undocumented deviation. Found by Plan Alignment.
- **[S4]** `useProjectEditor.ts:16` — `ReloadOutcome` enum return type for `reloadActiveChapter` deviates from design's `onError`-callback-reject pattern (`design.md:348-349`). Justified by I5 fix, but expands `useProjectEditor`'s public surface beyond what plan sanctioned. Found by Plan Alignment.
- **[S5]** `EditorPage.tsx:730-735` — 2xx BAD_JSON project-scope replace clears only the active chapter's cache; other affected chapters retain stale pre-replace drafts. Tradeoff is explicitly justified in the in-code comment but the lock banner is dismissible, so a user who switches chapter without refreshing can PATCH stale content over a server-committed replace. Found by Logic & Correctness.
- **[S6]** `EditorPage.tsx:427-442, 488-493` — `handleRestoreSnapshot` possibly_committed/unknown branches use `safeSetEditable(editorRef, false)` against `editorRef.current` (current active chapter), not the restore-target chapter. Mitigated by `actionBusyRef` preventing chapter switches during the run, but defense-in-depth is fragile. Found by Logic & Correctness.
- **[S7]** `EditorPage.tsx:561-577` — `finalizeReplaceSuccess` reloadFailed path splits lock signal between sync `safeSetEditable(false)` and async `setEditorLockedMessage`. Future caller relying on `editorLockedMessageRef.current` alone (without checking editor's own editable state) would race the render-commit window. Found by Logic & Correctness.
- **[S8]** `useEditorMutation.ts:336-357` — `reloadSuperseded` bypass calls `setEditable(true)` on new editor synchronously while banner-clearing useEffect runs on next React commit. Brief window where banner shows yet new editor is editable; data-loss prevented only by `handleSaveLockGated` ref check. Found by Logic & Correctness.
- **[S9]** `useSnapshotState.ts:144-147` — chapter-switch snapshot list fetch silently swallows all errors; `snapshotCount` stuck at `null` indefinitely until the user opens-and-closes the panel (which calls `refreshCount`). Found by Error Handling.
- **[S10]** `useEditorMutation.ts:143` and `EditorPage.tsx:1156, 1711, 1762` — `(await editorRef.current?.flushSave()) ?? true` treats `undefined` (null editor) as flushed; hook uses tighter `flushed === false`. Inconsistency latent if `EditorHandle.flushSave` return contract ever broadens to `Promise<void>`. Found by Error Handling.
