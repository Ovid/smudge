## Important Issues

### [I1] `handleStatusChangeWithError` bypasses the editor-lock banner

- **File:** `packages/client/src/pages/EditorPage.tsx:1196-1209`
- **Bug:** The sidebar status-change wrapper gates on `isActionBusy()` only. Every sibling sidebar guard (`handleReorderChaptersGuarded` at 1230, `handleCreateChapterGuarded`, `requestDeleteChapter`, `openTrashGuarded`) also checks `editorLockedMessageRef.current !== null` and raises `STRINGS.editor.lockedRefusal`. This one does not.
- **Impact:** When the lock banner is up (server state ambiguous after 2xx BAD_JSON restore/replace, or reload failure), the status-dropdown still PATCHes the chapter. That contradicts the load-bearing invariant the lock banner exists to enforce: no further server mutations until refresh.
- **Suggested fix:** Add the same `editorLockedMessageRef.current !== null` → `setActionInfo(STRINGS.editor.lockedRefusal)` refusal branch at the top of the handler.
- **Confidence:** High
- **Found by:** Logic & Correctness B, Error Handling B (merged)

### [I2] `handleRenameChapterWithError` bypasses the editor-lock banner

- **File:** `packages/client/src/pages/EditorPage.tsx:1211-1223`
- **Bug:** Same shape as I1 — sidebar rename wrapper only checks `isActionBusy()`. The inline chapter-title editor via `useChapterTitleEditing` does gate on `isEditorLocked`, so the inline-title path is protected; only the sidebar entry point is not.
- **Impact:** A sidebar inline-rename while the lock banner is up fires a title PATCH against a chapter the user was told to refresh.
- **Suggested fix:** Add the lock-banner refusal branch matching the other guards.
- **Confidence:** High
- **Found by:** Logic & Correctness B, Error Handling B (merged)

### [I3] `handleProjectSettingsUpdate` merges stale project metadata across navigation

- **File:** `packages/client/src/pages/EditorPage.tsx:1429-1445`
- **Bug:** The `.then` callback merges `{...data, chapters: prev.chapters}` without verifying `data.id === prev.id` or that the request's captured `slug` still matches `slugRef.current`. The S7 busy recheck guards against in-flight mutations, but does not guard cross-project navigation.
- **Impact:** If the user navigates A→B while `api.projects.get(A)` is in flight, A's `{title, slug, mode, target_word_count, target_deadline}` are spliced onto B's chapter list — B's sidebar renders A's metadata. The error is silent and persists until the next settings refresh or refresh-page.
- **Suggested fix:** `if (!prev || prev.id !== data.id) return prev;` inside the `setProject` updater, or compare the captured `slug` closure to `slugRef.current` before calling `setProject`.
- **Confidence:** High
- **Found by:** Concurrency & State, Logic & Correctness B (partial)

### [I4] Snapshot stale-chapter-switch branches don't exit snapshot view

- **File:** `packages/client/src/pages/EditorPage.tsx:486-491` (possibly_committed stale-chapter-switch), `555-562` (unknown-reason stale-chapter-switch)
- **Bug:** Both branches set `setActionError(...)` and refresh the snapshot list, but do NOT call `exitSnapshotView()`. Compare with `corrupt_snapshot` (518), `cross_project_image` (525), `not_found` (533) which all exit snapshot view.
- **Impact:** `SnapshotBanner` stays pinned to a snapshot for a chapter the user is no longer viewing. Both "Restore" and "Back to editing" buttons remain live on a banner that references a chapter the user has navigated away from — confusing and invites repeat restore attempts against the wrong chapter.
- **Suggested fix:** Call `exitSnapshotView()` in both stale-chapter-switch branches before returning.
- **Confidence:** Medium-High
- **Found by:** Logic & Correctness B
