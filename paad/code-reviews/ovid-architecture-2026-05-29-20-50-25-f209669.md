## Suggestions

- **[S1] `requestContext.ts:39` — Silent rejection of out-of-bound inbound `X-Request-Id`.** When a 129+ char trace id (some APM agents emit longer) or charset-violating value fails the regex, the middleware mints a fresh UUID with no diagnostic trail. Pure observability gap: a misconfigured upstream loses correlation invisibly. Emit `req.log.debug({ raw: incoming }, "discarded inbound x-request-id")` when rejecting. (Error Handling, confidence 65.)
- **[S2] `images.reaper.ts:41` — Reaper snapshots known-ids before walking the tree; a second concurrent process could race-delete a freshly-uploaded file.** Cannot fire under the documented single-process model (the reaper runs before `app.listen`, and CLAUDE.md says single-process), but a developer running `npm start` twice against the same `DATA_DIR` would silently lose images. Either document "concurrent server processes against the same data dir are unsupported" in the reaper's doc comment, add an advisory `flock` on `imagesRoot/.reaper.lock`, or re-query `db("images").where({id}).first()` per-candidate just before unlink. (Error Handling, downgraded from "important" — confidence 70.)
- **[S3] `app.ts:80-89` — Global error handler explicitly logs `req_id`/`method`/`path` alongside `logger.error` when the same fields are already bound to `req.log`.** Today benign (Express restores `req.url` before the global handler runs); the redundant explicit binding adds noise and risks inconsistency. Use `req.log.error({ err, status }, "Unhandled request error")` with a `req.log ? ... : logger.error(...)` fallback for the pre-middleware case. (Error Handling, confidence 70.)
- **[S4] `images.reaper.ts:33-39, 50` — `try { readdir } catch { return 0 }` swallows ALL errors, conflating ENOENT (legitimate fresh install) with EACCES/EIO (operator error or hardware fault).** A real filesystem failure becomes a "confusing upload error" later instead of a startup signal. Inspect the error code and only swallow ENOENT; log warn on other codes. Apply to both the outer and inner per-project catches. (Error Handling, confidence 65.)
- **[S5] `clientLog.ts:17` — `import.meta.env?.DEV` silently degrades to "always silent" in any context that doesn't populate `import.meta.env`.** Today the only consumers are Vite-built client code and Vitest, both of which populate the env. The optional-chain hides the silent-everywhere failure mode for a future Electron preload / SSR / Storybook MDX consumer. Either drop the `?.` or normalize: `const isDev = typeof import.meta.env !== "undefined" && import.meta.env.DEV === true;`. (Error Handling, confidence 60.)
- **[S6] `images.reaper.ts:11` — `IMAGE_FILE_RE = /^(<uuid>)\.[a-z0-9]+$/i` duplicates `MIME_TO_EXT`'s contract by approximation.** The producer writes exactly jpg/png/gif/webp; the reaper accepts any `<uuid>.<alphanumeric>` and would unlink an externally-placed `<uuid>.bak`/`<uuid>.tmp`/`<uuid>.orig`. The two definitions can drift. Derive the alternation from `MIME_TO_EXT` directly: ``new RegExp(`^(${UUID_PATTERN})\\.(${Object.values(MIME_TO_EXT).join("|")})$`, "i")``, or export an `IMAGE_EXT_PATTERN` constant from `images.paths.ts` that both producer and reaper consume. Add a regression test asserting `<uuid>.bak` is NOT reaped. (Contract & Integration, confidence 65.)
- **[S7] `Editor.tsx:69` — `ImagePasteOptions.getUploadHandler` return type is `((file: File) => void) | null`, but the handler installed at line 323 is `async (file: File) => {...}`.** TypeScript accepts the widening; the typed contract no longer documents the async semantics. No observable bug today (the handler has a top-level try/catch), but a future refactor that strips the inner try/catch would silently regress to uncaught promise rejections. Widen the type: `((file: File) => Promise<void> | void) | null`. (Contract & Integration, confidence 65.)
- **[S8] `Editor.tsx:322` — F-17 status reason in the architecture report claims "behavior byte-preserved," but the rewritten registration effect adds an `if (!editor) return;` early-return the pre-fix code did not have.** Functionally harmless (the paste/drop plugin can only fire once the editor is mounted, so the previous "handler registered before editor existed" window was unreachable too), but the report's "byte-preserved" phrasing overstates the rewrite. Either drop the guard to match the prior behavior literally, or amend the F-17 status reason to acknowledge the defensive guard explicitly. (Spec Compliance, deviation, confidence 65.)
