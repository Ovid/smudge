## Verification Notes

Direct checks performed before and alongside the specialist dispatch:

- **F-9 `apiFetch` 204 handling** — `packages/client/src/api/client.ts:200` (`if (res.status === 204) return undefined as T;`) sits inside the `res.ok` block and *before* the `res.json().catch(...)` body-read, so a 204 short-circuits with no body read and the `BAD_JSON`/`possiblyCommitted` path is provably unreachable on success. Pre-existing logic, already used by the F-16 DELETE endpoints.
- **F-9 caller contract** — every production caller of `api.settings.update` (`ProjectSettingsDialog.tsx:251`, `useTimezoneDetection.ts:24`) and `api.projects.reorderChapters` (`useChapterCrud.ts:713`) `await`s the promise and discards the resolved value. The only `.message` read is destructured from `mapApiError(err, …)` on the error path. The `{message}`→`undefined` change is safe.
- **F-9 route success/error split** — both routes reach `res.status(204).end()` only after every validation branch has `throw`n its `AppError` (REORDER_MISMATCH 400, invalid-settings 400, 404, 413, 500), so error responses still render the JSON envelope; only the success response changed.
- **F-9 test coverage** — server integration tests updated to assert `204` + empty body (`projects.test.ts:367`, `settings.test.ts:28,41`); client tests assert `result` is `undefined` (`api-client.test.ts:118,362`) and mocks resolve `undefined`. No e2e references these endpoints' bodies.
