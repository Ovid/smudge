## Verification Notes (what was checked and cleared)

The following were the genuine risk points in this diff; each was traced to ground truth:

- **F-7 (`deleteProject` ‚Üí `applyImageRefDiff`), the highest-risk change:** The transaction
  store (`txStore: ProjectStore`) exposes both `incrementImageReferenceCount` and the
  newly-required `findImagesByIds` (`stores/project-store.types.ts:102`,
  `sqlite-project-store.ts:217`), so the swap from the raw decrement loop does not hit a
  missing method. `applyImageRefDiff(txStore, ch.content, null, project.id)` is behaviorally
  equivalent to the old loop for refcount arithmetic (once-per-chapter-per-image decrement,
  including images referenced across multiple chapters), treats corrupt/null old content as an
  empty old set (no decrement ‚Äî matching the old skip-on-parse-failure), and now applies the
  cross-project `image.project_id !== projectId` guard that is the point of the fix. A red
  test was added (`projects.service.test.ts:119`).
- **F-14 (`/api/health` liveness probe):** `getDb()` is called *inside* the `try`, so the
  "not initialized" throw is caught ‚Üí 503 rather than an unhandled rejection; `db.raw("SELECT 1")`
  is awaitable under knex+better-sqlite3; the raw error is logged server-side only and never
  reaches the response body (`{status:"error"}` only). `request-context.test.ts` and
  `health.test.ts` both initialize a DB so the endpoint stays quiet.
- **uuid ‚Üí `crypto.randomUUID()`:** CSPRNG-backed, same lowercased hyphenated v4 format,
  available on the pinned Node 22. No remaining `from "uuid"` imports in `packages/server/src`
  or `package.json`; no non-CSPRNG source (`Math.random`) introduced in any ID path.
- **F-6 (`createKnexConfig()`):** Preserves the old `DB_PATH` behavior ‚Äî `getDbPath()` honors
  `process.env.DB_PATH`; migrations dir, `loadExtensions`, `useNullAsDefault`, and `client`
  all match the removed inline block.
- **F-13 (`ScopeEntry` move):** The `apiErrorMapper ‚Üî scopes` cycle is removed, no new cycle
  introduced, and the re-export chain (`apiErrorMapper.ts` re-exports from `scopes.ts`) keeps
  every existing importer resolving.
- **F-10 (DEV-gating):** `devWarn`‚Üí`clientWarn` and `apiErrorMapper.safeExtrasFrom`‚Üí`clientError`
  both self-gate on `isDev()` (a stricter guard than the removed `import.meta.env?.DEV`
  optional-chain); browser-console only, never the rendered UI, no-op in production.
