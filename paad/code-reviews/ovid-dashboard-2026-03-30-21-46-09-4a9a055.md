## Executive Summary

The Writer's Dashboard feature is well-implemented with solid architecture, good accessibility, and comprehensive test coverage. One critical issue found: the client-side cache is cleared on 4xx save rejections, violating the spec's data-safety guarantee and risking content loss. Five important bugs relate to race conditions in the save/status pipeline and an API contract inconsistency where `status_label` is missing from several chapter endpoints. Overall code quality is high; most issues are edge cases in concurrent state management.

## Critical Issues

### [C1] Cache cleared on 4xx save rejection violates spec
- **File:** `packages/client/src/hooks/useProjectEditor.ts:86-88`
- **Bug:** When the server rejects a save with a 4xx error, `clearCachedContent(savingChapterId)` is called, destroying the user's locally cached draft. The CLAUDE.md spec states "client-side cache holds unsaved content until server confirms." A 4xx rejection is not a confirmation — the content was not saved. The editor DOM still holds the content, but if the user switches chapters or reloads before re-saving successfully, the draft is permanently lost.
- **Impact:** Data loss. User edits rejected by the server (e.g., malformed JSON) lose their cache safety net. Switching chapters after a 4xx error means the unsaved work is gone.
- **Suggested fix:** Remove `clearCachedContent(savingChapterId)` from the 4xx error branch (line 87). Only clear cache on successful save (already done at line 82). Additionally, surface the specific error reason to the user rather than showing a generic "Unable to save."
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling & Edge Cases

## Important Issues

### [I1] Save status is global, not chapter-scoped — wrong status shown after chapter switch
- **File:** `packages/client/src/hooks/useProjectEditor.ts:63-83`
- **Bug:** `handleSave` calls `setSaveStatus("saving")` on line 67 and `setSaveStatus("saved")` on line 83, but save status is a single global state atom. During the retry window (up to ~14 seconds), if the user switches to chapter B and starts editing (status becomes "unsaved"), chapter A's late-arriving save result overwrites the status to "saved" — making the user believe chapter B is saved when it is not.
- **Impact:** Silent data confidence issue. User may navigate away from an unsaved chapter believing it was saved, leading to content loss.
- **Suggested fix:** Guard `setSaveStatus` calls with `if (activeChapterRef.current?.id === savingChapterId)` so only the currently-displayed chapter's status is updated.
- **Confidence:** High
- **Found by:** Logic & Correctness, Concurrency & State

### [I2] After restore changes project slug, browser URL is not updated
- **File:** `packages/client/src/pages/EditorPage.tsx:128-130`
- **Bug:** When restoring a chapter that also restores a soft-deleted project, the server may assign a new slug. The `setProject` updater correctly updates `project.slug` in state, but `navigate()` is never called to update the browser URL. The URL still shows the old (now invalid) slug. A page refresh or bookmark results in a 404.
- **Impact:** After a restore-with-project-revival, any page refresh fails with a 404. The user must manually navigate back to the home page and re-enter the project.
- **Suggested fix:** After `setProject` updates the slug in `handleRestore`, check if the new slug differs from the current URL param and call `navigate(\`/projects/${newSlug}\`, { replace: true })`.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [I3] GET/PATCH /chapters/:id and POST /chapters missing status_label in response
- **File:** `packages/server/src/routes/chapters.ts:26,94` and `packages/server/src/routes/projects.ts:264`
- **Bug:** `GET /chapters/:id`, `PATCH /chapters/:id`, and `POST /:slug/chapters` all return chapters via `parseChapterContent()` without adding `status_label`. The `GET /projects/:slug` endpoint does add `status_label` via `getStatusLabelMap()`. The `Chapter` type declares `status_label?: string`, and `DashboardView` renders `chapter.status_label` directly (line 238). After operations that use the chapter endpoints (save, select, create), `status_label` is undefined.
- **Impact:** Inconsistent API contract. After creating a chapter or saving, the chapter in state lacks `status_label`. Currently the sidebar derives labels from the `statuses` prop (unaffected), but the pattern creates a latent bug for any future code that reads `status_label` from chapter state.
- **Suggested fix:** Add `status_label` to all chapter endpoint responses using the existing `getStatusLabelMap()` helper. Alternatively, always derive labels client-side from the `statuses` array and remove `status_label` from the `Chapter` type.
- **Confidence:** High
- **Found by:** Contract & Integration

### [I4] Double-save race in debounce timer
- **File:** `packages/client/src/components/Editor.tsx:36-39`
- **Bug:** The debounced save callback is async: it `await`s `onSaveRef.current(...)` and only sets `debounceTimerRef.current = null` after the save completes. While the save is in-flight, `debounceTimerRef.current` still holds the (already-fired) timer ID. If `flushSave` is called during this window, it sees a truthy ref, clears it (no-op since the timer already fired), and starts a second concurrent save of the same content.
- **Impact:** Two concurrent saves of the same content fire simultaneously. With SQLite's single-writer model this won't corrupt data, but it wastes resources and can cause confusing save status transitions (e.g., "saving" -> "saved" -> "saving" -> "saved").
- **Suggested fix:** Set `debounceTimerRef.current = null` at the *start* of the setTimeout callback (before the `await`), not after. This way, `flushSave` knows the timer has already fired and won't double-save.
- **Confidence:** High
- **Found by:** Concurrency & State

### [I5] Rapid chapter switching has no cancellation — race condition
- **File:** `packages/client/src/hooks/useProjectEditor.ts:125-138`
- **Bug:** `handleSelectChapter` calls `await api.chapters.get(chapterId)` then `setActiveChapter(...)`. If the user clicks chapter A then immediately chapter B, both API calls are in flight. Whichever resolves last wins and calls `setActiveChapter`, which may not be the chapter the user last clicked. Unlike `loadProject` (which has a `cancelled` flag), there is no cancellation mechanism for chapter selection.
- **Impact:** Rapid chapter switching can display the wrong chapter's content. The user sees chapter A's content when they clicked chapter B last.
- **Suggested fix:** Use a request counter ref or AbortController. Increment a counter before each API call; only call `setActiveChapter` if the counter still matches when the response arrives.
- **Confidence:** Medium
- **Found by:** Concurrency & State

### [I6] handleStatusChange double failure silently leaves wrong UI state
- **File:** `packages/client/src/hooks/useProjectEditor.ts:234,250`
- **Bug:** If the status update API call fails (line 234 catch) and the subsequent project reload also fails (line 250 catch), the UI is left showing the optimistic (incorrect) status with no error notification. The user sees a status that doesn't match the server, with no indication anything went wrong.
- **Impact:** Silent data inconsistency. The UI shows a status that was never persisted, and the user has no way to know.
- **Suggested fix:** On double failure, either revert to the pre-optimistic status (save it before the update) or call `setError()` / show a notification.
- **Confidence:** Medium
- **Found by:** Logic & Correctness, Error Handling & Edge Cases

### [I7] Optimistic status update sets `status` but not `status_label`
- **File:** `packages/client/src/hooks/useProjectEditor.ts:226`
- **Bug:** The optimistic update in `handleStatusChange` does `{ ...c, status }` — only `status` is updated, not `status_label`. After the optimistic update, `status_label` holds the stale old value. Any component reading `status_label` directly from the chapter data will display the wrong label until data is re-fetched.
- **Impact:** Currently the sidebar derives labels from the `statuses` prop (unaffected), and the dashboard re-fetches on mount. But the inconsistency creates a latent bug for any code path that reads `status_label` from chapter state.
- **Suggested fix:** Also compute and set `status_label` in the optimistic update using the `statuses` array, or always derive labels client-side and remove reliance on `status_label`.
- **Confidence:** High
- **Found by:** Contract & Integration

### [I8] handleDeleteChapter uses stale closure for post-delete navigation
- **File:** `packages/client/src/hooks/useProjectEditor.ts:158`
- **Bug:** After `await api.chapters.delete()`, the code checks `activeChapter?.id === chapter.id` using the closure-captured `activeChapter`. If the user switched chapters during the async delete, `activeChapter` is stale. The code would enter the if-branch and force-navigate to `remaining[0]`, overwriting the chapter the user just navigated to.
- **Impact:** After a rapid delete-then-switch interaction, the user is unexpectedly navigated away from their intended chapter.
- **Suggested fix:** Use `activeChapterRef.current?.id` instead of `activeChapter?.id` on line 158, consistent with the ref pattern already used for `projectRef`.
- **Confidence:** Medium
- **Found by:** Logic & Correctness, Concurrency & State

## Suggestions

- **Dashboard sort order irrecoverable** (`DashboardView.tsx:196`): No column header for default manuscript order (sort_order). Once a user sorts by any column, they cannot return to natural chapter order without reloading. Add an "Order" column or "Reset sort" button. *(Logic & Correctness, conf 75%)*
- **Dashboard endpoint duplicates getStatusLabelMap()** (`projects.ts:349-354`): Identical status-label-map construction at lines 349-354, duplicating the `getStatusLabelMap()` helper at line 14. Replace with `const statusLabelMap = await getStatusLabelMap(db);` *(Contract & Integration, conf 90%)*
- **Chapter statuses fetch failure silently swallowed** (`EditorPage.tsx:90`): `.catch(console.error)` means failed status loading degrades the UI (empty dropdowns, no labels) without any user-facing indication. *(Error Handling, conf 70%)*
- **Keyboard nav doesn't guard against rapid re-entry** (`EditorPage.tsx:191`): Rapid Ctrl+Shift+Arrow presses fire overlapping async flush+switch sequences. Add a `navigatingRef` guard. *(Concurrency, conf 65%)*
- **Status change revert reloads entire project** (`useProjectEditor.ts:236-249`): On status change failure, `api.projects.get()` replaces the entire project state, potentially overwriting in-flight word count updates. Revert only the specific chapter's status instead. *(Concurrency, conf 65%)*
- **ChapterStatus Zod enum hardcodes DB-owned values** (`schemas.ts:5`): Static Zod enum can reject statuses that exist in the DB. Adding a status requires updates in three places (DB, Zod, STATUS_COLORS). *(Contract & Integration, conf 65%)*
- **STATUS_COLORS is a third copy of the status list** (`statusColors.ts:1`): Same five status keys in Zod enum, STATUS_COLORS, and DB migration. Consider serving colors from the API. *(Contract & Integration, conf 65%)*
- **Post-update SELECT outside transaction** (`chapters.ts:92`): Safe with SQLite but not strictly ACID. *(Error Handling, conf 65%)*
- **No security headers** (`app.ts`): No helmet, CSP, or X-Frame-Options. Low priority for single-user local deployment but worth adding before any network exposure. *(Security, conf 70%)*
- **POST /:slug/chapters returns chapter without status_label** (`projects.ts:264`): Related to I3. *(Contract & Integration, conf 75%)*
