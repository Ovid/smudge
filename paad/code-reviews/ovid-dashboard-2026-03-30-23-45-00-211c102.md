## Important Issues

### [I1] `handleContentChange` caches content under wrong chapter ID during rapid switch
- **File:** `packages/client/src/hooks/useProjectEditor.ts:107-116`
- **Bug:** `handleContentChange` is memoized with `[activeChapter]` dependency. During a chapter switch, TipTap can fire a content-change event asynchronously before React re-renders with the new callback. The stale closure writes the new content to the cache under the *old* chapter's ID via `setCachedContent(activeChapter.id, content)`.
- **Impact:** Wrong chapter gets a content blob in the cache. On next load of that chapter, the user sees incorrect content until a successful save clears the cache.
- **Suggested fix:** Use `activeChapterRef.current` inside the callback instead of the closure-captured `activeChapter`, and remove `activeChapter` from the dependency array.
- **Confidence:** High
- **Found by:** Concurrency & State

### [I2] Save retry loop continues after chapter switch with no cancellation
- **File:** `packages/client/src/hooks/useProjectEditor.ts:61-105`
- **Bug:** During the retry backoff sleep (`await new Promise(r => setTimeout(r, BACKOFF_MS[attempt]))`), the user can switch chapters. The retry loop has no cancellation mechanism — it continues firing `api.chapters.update(savingChapterId, { content })` for the old chapter. The `activeChapterRef` check at lines 84/99 only guards `setSaveStatus`, not the API call itself.
- **Impact:** Up to 3 redundant network requests for stale chapter saves after a switch, potentially confusing save-status indicators.
- **Suggested fix:** Introduce a save-sequence counter (like `selectChapterSeqRef`) and check it before each retry attempt. Alternatively, pass an `AbortSignal` to the fetch call.
- **Confidence:** High
- **Found by:** Concurrency & State

### [I3] No cleanup of retry timers in statuses fetch effect
- **File:** `packages/client/src/pages/EditorPage.tsx:89-104`
- **Bug:** The `useEffect` for fetching chapter statuses uses `setTimeout` for retries but has no cleanup function. If the component unmounts while a retry timer is pending, the callback fires and calls `setStatuses` on an unmounted component. No `cancelled` flag pattern (which is used correctly in `DashboardView.tsx` and `useProjectEditor.ts`).
- **Impact:** Dangling timers after unmount; up to 3 fetches fire post-unmount. React 18 makes the setState no-op safe, but React Strict Mode double-invoke exacerbates the issue.
- **Suggested fix:** Add a `cancelled` flag and `clearTimeout` in the effect cleanup, matching the pattern already used elsewhere in the codebase.
- **Confidence:** High
- **Found by:** Logic & Correctness, Concurrency & State

### [I4] Status change revert skipped when slug is null
- **File:** `packages/client/src/hooks/useProjectEditor.ts:242-273`
- **Bug:** The revert block on status-change failure is wrapped in `if (slug)` where `slug = projectSlugRef.current`. If the slug ref is null (race during project load or after a slug change), the entire revert path is skipped — both the server reload and the local `previousStatus` fallback. The UI stays in the optimistically-applied wrong state while the error banner displays.
- **Impact:** User sees a wrong status badge with no automatic recovery.
- **Suggested fix:** Move the local `previousStatus` fallback outside the `if (slug)` block so it always runs as a last resort.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [I5] Empty `statuses` prop causes silent blank status summary bar
- **File:** `packages/client/src/pages/EditorPage.tsx:85` + `packages/client/src/components/DashboardView.tsx:109,157`
- **Bug:** If the `fetchStatuses` call fails all retries, `statuses` remains `[]`. The dashboard's status summary bar iterates over `statuses.map(...)` — rendering nothing. Meanwhile `totalStatusCount` (computed from server-provided `status_summary`) is > 0, so the section wrapper renders but the bar and legend are empty. No error message or fallback is shown.
- **Impact:** Silent degradation — a blank status section with no user-facing explanation.
- **Suggested fix:** Show a fallback message when `statuses` is empty but `totalStatusCount > 0`, or derive status labels client-side from `status_summary` keys.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [I6] Stale dashboard data shown during slug change
- **File:** `packages/client/src/components/DashboardView.tsx:23-42`
- **Bug:** When `slug` changes (e.g., project title rename), the effect re-fetches but `data` is not reset to `null` first. During the fetch, the component continues displaying the *previous project's* dashboard data. The `cancelled` flag correctly prevents stale responses from being applied, but the intermediate render shows wrong data.
- **Impact:** Brief display of another project's dashboard during slug transitions.
- **Suggested fix:** Add `setData(null); setError(null);` at the start of the effect before the async call.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [I7] `status_label` absent from chapter create and restore API responses
- **File:** `packages/server/src/routes/projects.ts:264`, `packages/server/src/routes/chapters.ts:186`
- **Bug:** `GET /api/projects/:slug` adds `status_label` to every chapter. But `POST /:slug/chapters` (create) and `POST /:id/restore` return chapters without `status_label`. The client `Chapter` type has `status_label?: string`, so TypeScript doesn't catch this. Any code consuming create/restore responses and reading `status_label` gets `undefined`.
- **Impact:** Inconsistent API contract. Currently the Sidebar resolves labels from the `statuses` prop so no visible bug, but the inconsistency is a maintenance trap.
- **Suggested fix:** Either add `status_label` to all chapter-returning endpoints, or remove it from `GET /:slug` and resolve labels entirely client-side (which the sidebar already does correctly).
- **Confidence:** High
- **Found by:** Contract & Integration

### [I8] PATCH chapter response missing `status_label`
- **File:** `packages/server/src/routes/chapters.ts:97`
- **Bug:** The PATCH handler returns `parseChapterContent(updated)` without adding `status_label`. Combined with I7, this means only the project-level GET provides `status_label` on chapters. All other chapter-returning endpoints omit it.
- **Impact:** Same as I7 — contract inconsistency. The client currently discards the PATCH response body for status changes (uses optimistic state), so no visible bug today.
- **Suggested fix:** Same as I7 — standardize whether `status_label` is a server or client concern.
- **Confidence:** High
- **Found by:** Contract & Integration

## Plan Alignment

**Implemented:**
- Data model: `chapter_statuses` table, `chapters.status` column, Zod enum, shared types
- API: PATCH chapters with status, GET projects/:slug with status+status_label, GET chapter-statuses, GET projects/:slug/dashboard
- Sidebar status badge with dropdown picker, optimistic update, ARIA listbox pattern
- Resizable sidebar (180-480px, localStorage, keyboard accessible, role="separator")
- Dashboard as peer tab (Editor/Preview/Dashboard), empty state, health bar, status summary bar, sortable chapter table
- Chapter navigation shortcuts (Ctrl+Shift+Arrow, editor-only, no wrap-around, screen reader announcement)
- Keyboard help dialog updated with new shortcuts

**Not yet implemented:**
- Responsive badge text hiding at narrow sidebar widths (dot-only + tooltip at narrow widths)

**Deviations:**
- Save failure does not block chapter navigation (spec requires blocking)
- Shortcuts fire when shortcut-help dialog is open (spec requires disabling in modals)
- Dashboard `totals` from server includes timestamps but not chapter names; client derives both from chapter list
