## Important Issues

### [I1] Keyboard shortcuts fire through open modal dialogs (WCAG focus-trap violation)
- **File:** `packages/client/src/pages/EditorPage.tsx:191-251`
- **Bug:** The `handleKeyDown` listener on `document` has no guard for whether a modal dialog (`ConfirmDialog` for delete confirmation, or shortcut help dialog) is open. Pressing Ctrl+Shift+N during a delete confirmation creates a new chapter behind the dialog. Ctrl+Shift+Arrow navigates chapters while the dialog is visible.
- **Impact:** WCAG 2.1 AA requires modal dialogs to trap focus and disable background interactions. This allows state-changing actions while a dialog expects the user's attention, potentially leading to confusing state.
- **Suggested fix:** Add an early return at the top of `handleKeyDown` when `deleteTarget !== null` or `shortcutHelpOpen` is true (except for Escape, which already closes the shortcut dialog).
- **Confidence:** High
- **Found by:** Plan Alignment

## Suggestions

### [S1] Dashboard endpoint queries `chapter_statuses` table twice
- **File:** `packages/server/src/routes/projects.ts:354-357`
- **Bug:** The `/:slug/dashboard` handler calls `db("chapter_statuses")` directly (into `allStatuses`) and then calls `getStatusLabelMap(db)` (into `statusLabelMap`), which issues an identical query. Two round-trips where one suffices.
- **Suggested fix:** Build both `statusSummary` and `statusLabelMap` from a single query result.
- **Confidence:** High
- **Found by:** Logic & Correctness, Contract & Integration

### [S2] `getStatusLabel` and `getStatusLabelMap` are duplicated DB logic in separate files
- **File:** `packages/server/src/routes/chapters.ts:8-11`, `packages/server/src/routes/projects.ts:14-19`
- **Bug:** Two utility functions query `chapter_statuses` with different patterns (single-row vs batch). If fallback logic or table schema changes, both must be updated independently.
- **Suggested fix:** Extract a shared helper into a common module (e.g., `routes/chapter-status-helpers.ts`).
- **Confidence:** High
- **Found by:** Contract & Integration

### [S3] `Chapter.status` typed as `string` instead of `ChapterStatus` union type
- **File:** `packages/shared/src/types.ts:23`
- **Bug:** The `Chapter` interface uses `status: string` despite `ChapterStatus` being defined as a Zod enum in `schemas.ts`. This loses compile-time type safety — any string can be assigned to `chapter.status`.
- **Suggested fix:** Use `status: z.infer<typeof ChapterStatus>` or the explicit union `"outline" | "rough_draft" | "revised" | "edited" | "final"`.
- **Confidence:** Medium
- **Found by:** Plan Alignment

### [S4] Status sort degrades inconsistently for mixed known/unknown statuses
- **File:** `packages/client/src/components/DashboardView.tsx:96-103`
- **Bug:** When sorting by status, if one chapter's status is in the `statusSortOrder` lookup but the other's is not, both fall through to `localeCompare`. The known chapter's numeric sort order is discarded. This produces neither numeric-first nor consistent alphabetical ordering.
- **Suggested fix:** Sort known statuses before unknown ones: `if (orderA !== null && orderB === null) return -1;`
- **Confidence:** Medium
- **Found by:** Logic & Correctness, Error Handling

### [S5] `PreviewMode` stale `chapterRefs` entries never removed
- **File:** `packages/client/src/components/PreviewMode.tsx:15, 30, 53-55`
- **Bug:** The `chapterRefs` Map accumulates entries but never clears them. When chapters are removed, stale entries persist and get re-observed by the IntersectionObserver. Detached DOM nodes are observed (harmless but leaks).
- **Suggested fix:** Clear the map at the start of the IntersectionObserver effect, or handle the `null` case in the ref callback: `else { chapterRefs.current.delete(chapter.id); }`
- **Confidence:** Medium
- **Found by:** Logic & Correctness, Error Handling, Concurrency & State

### [S6] Post-write re-read in PATCH/restore handlers missing null guard
- **File:** `packages/server/src/routes/chapters.ts:102, 193`
- **Bug:** After updating/restoring a chapter, the re-fetch to build the response does `db("chapters").where({id}).first()` without a null check. If the row were somehow absent, `parseChapterContent(updated)` throws with an unhelpful error.
- **Suggested fix:** Add `if (!updated) { res.status(404)...; return; }` after the re-fetch.
- **Confidence:** Medium
- **Found by:** Security

### [S7] Unknown-status chapters silently excluded from `status_summary`
- **File:** `packages/server/src/routes/projects.ts:369-373`
- **Bug:** The dashboard's `status_summary` only counts chapters whose status is in the `chapter_statuses` table. If a chapter has an unrecognized status (via direct DB edit or migration drift), it's excluded from the summary but included in `totals.chapter_count`, creating a discrepancy.
- **Suggested fix:** Add an "unknown" bucket or log a warning when an unrecognized status is encountered.
- **Confidence:** Low-Medium
- **Found by:** Logic & Correctness

### [S8] `clearCachedContent` called after stale save without post-await seq check
- **File:** `packages/client/src/hooks/useProjectEditor.ts:86`
- **Bug:** After `await api.chapters.update(...)` succeeds, `clearCachedContent(savingChapterId)` runs without re-checking `saveSeqRef`. If the user switched chapters during the await, the cache for the old chapter is cleared even though newer edits may have been cached. The window is brief (next `handleContentChange` re-caches).
- **Suggested fix:** Add `if (seq !== saveSeqRef.current) return false;` after the await.
- **Confidence:** Low-Medium
- **Found by:** Concurrency & State

### [S9] Rapid concurrent status changes use stale `previousStatus` for local revert
- **File:** `packages/client/src/hooks/useProjectEditor.ts:223, 259-270`
- **Bug:** If two rapid status changes fire on the same chapter before the first API call returns, both capture `previousStatus` from `projectRef.current` at call time. The optimistic update from the first change is reflected in `project` state but `projectRef.current` may not have synced yet (ref updated via useEffect). If both fail, local revert can use stale data. The server-reload revert path (line 244) handles this correctly.
- **Suggested fix:** Read `previousStatus` from the server-reload response rather than the stale capture, or remove the local-revert fallback in favor of always reloading from server.
- **Confidence:** Low
- **Found by:** Concurrency & State

### [S10] Save status shows "Saving..." during multi-second retry backoff waits
- **File:** `packages/client/src/hooks/useProjectEditor.ts:96`
- **Bug:** During retry backoff (up to 8 seconds), `saveStatus` remains "saving" even though no network request is in-flight. The footer indicator is misleading during the sleep period.
- **Suggested fix:** Set `saveStatus` to "retrying" or "unsaved" during the sleep, reverting to "saving" when the next attempt starts.
- **Confidence:** Low
- **Found by:** Error Handling
