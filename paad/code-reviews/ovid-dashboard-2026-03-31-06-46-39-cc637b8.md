## Important Issues

### [I1] Dashboard data not refreshed after sidebar status changes
- **File:** `packages/client/src/components/DashboardView.tsx:25-44`
- **Bug:** The dashboard `useEffect` depends only on `[slug]`. When the user changes a chapter's status via the sidebar status dropdown while the Dashboard tab is active (or switches back to Dashboard after a change), the dashboard retains stale data from the initial fetch. Status summary counts, chapter table status labels, and the status bar segments are all outdated.
- **Impact:** User sees incorrect status counts and stale chapter metadata in the dashboard until they navigate away and return to the project. In a writing app where the dashboard is the "manuscript health" view, stale data undermines trust.
- **Suggested fix:** Accept a `refreshKey` prop (incremented by `EditorPage` on status changes) and include it in the `useEffect` dependency array. Alternatively, refetch when the Dashboard tab becomes active.
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling & Edge Cases, Concurrency & State

### [I2] Dashboard endpoint duplicates statusLabelMap construction instead of using extracted helper
- **File:** `packages/server/src/routes/projects.ts:348-353`
- **Bug:** The `GET /:slug/dashboard` endpoint queries `chapter_statuses` and builds `statusLabelMap` inline via `Object.fromEntries`. The `GET /:slug` endpoint (line 209) correctly uses the extracted `getStatusLabelMap(db)` helper from `status-labels.ts`. The dashboard endpoint should use the same helper.
- **Impact:** Logic duplication — if `getStatusLabelMap` is updated (e.g., to cache, filter, or change the mapping), the dashboard endpoint won't inherit the change. The extracted helper was introduced in commit f6d152b specifically to address this.
- **Suggested fix:** Replace lines 348-353 with `const statusLabelMap = await getStatusLabelMap(db);` and use the existing `allStatuses` query only for the status summary bar ordering.
- **Confidence:** High
- **Found by:** Contract & Integration

### [I3] `handleReorderChapters` does not update `sort_order` field on chapters
- **File:** `packages/client/src/hooks/useProjectEditor.ts:192-195`
- **Bug:** When chapters are reordered via drag-and-drop, the optimistic update reorders the array by position but does not update each chapter's `sort_order` property. The chapters retain their original `sort_order` values. The dashboard's "#" column renders `chapter.sort_order + 1` (`DashboardView.tsx:239`), so after a reorder it shows stale position numbers.
- **Impact:** After drag-and-drop reorder, switching to the dashboard shows incorrect chapter order numbers in the "#" column until page refresh.
- **Suggested fix:** Update `sort_order` on each chapter during the optimistic update:
  ```ts
  const reordered = orderedIds
    .map((id, index) => {
      const ch = prev.chapters.find((c) => c.id === id);
      return ch ? { ...ch, sort_order: index } : undefined;
    })
    .filter(Boolean) as Chapter[];
  ```
- **Confidence:** High
- **Found by:** Logic & Correctness (B)

### [I4] Missing null guard after re-read in create-chapter endpoint
- **File:** `packages/server/src/routes/projects.ts:257-262`
- **Bug:** After inserting a new chapter, the code re-reads it from the DB (`const chapter = await db("chapters").where({ id: chapterId }).first()`) and immediately calls `parseChapterContent(chapter)` and accesses `chapter.status` without checking for null. Other endpoints (PATCH chapter, restore chapter) have added null guards after similar post-write re-reads (commit 848d488), but this one was missed.
- **Impact:** If the re-read fails (extremely unlikely but possible under DB pressure), `parseChapterContent(undefined)` would throw an unhandled error, returning a 500 instead of a clear error message.
- **Suggested fix:** Add a null guard consistent with other endpoints:
  ```ts
  if (!chapter) {
    res.status(404).json({
      error: { code: "NOT_FOUND", message: "Chapter not found." },
    });
    return;
  }
  ```
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [I5] Word count updates on every keystroke in `aria-live` region
- **File:** `packages/client/src/hooks/useProjectEditor.ts:109-111`
- **Bug:** `handleContentChange` calls `setChapterWordCount(countWords(content))` on every editor update. The word count is rendered in a status footer with `aria-live="polite"`. The design doc (section 8.4 of the MVP spec) says: "updated on save, not on every keystroke" to avoid excessive screen reader announcements.
- **Impact:** Screen readers may announce word count changes on every keystroke, creating noise that interferes with writing. `aria-live="polite"` batches announcements but doesn't eliminate them.
- **Suggested fix:** Throttle word count updates to save events only, or debounce the `aria-live` announcement separately from the visual display (visual can update per-keystroke; live region updates on save).
- **Confidence:** Medium
- **Found by:** Plan Alignment

## Suggestions

- **Migration defense-in-depth:** `003_add_chapter_status.js:18` adds `status` column with `.defaultTo("outline")` but without `.notNullable()`. SQLite allows NULL despite the default. Add `.notNullable()` for consistency with other columns.
- **Status summary bar ordering fragile:** `DashboardView.tsx:162-178` uses `Object.entries(status_summary)` which relies on JavaScript object insertion order matching `sort_order`. Works today because the server builds it in order, but could break if server logic changes. Consider ordering explicitly using the `statuses` prop.
- **Dashboard `status_summary` silently drops unknown statuses:** `projects.ts:365-370` only counts chapters whose status exists in `chapter_statuses` table. A chapter with an unrecognized status would be excluded from the summary but included in `totals.chapter_count`, creating a discrepancy. Add an "unknown" bucket or remove the guard.
- **Server-provided `most_recent_edit`/`least_recent_edit` unused:** Dashboard response includes these fields, but `DashboardView.tsx:80-83` recomputes them client-side to get the chapter name alongside the date. Either enhance the server response to include the chapter title, or remove the unused fields.
- **Dashboard health bar uses absolute dates instead of relative time:** Design specifies "relative time + chapter name" (e.g., "3 days ago") but implementation shows absolute dates (e.g., "Mar 28, 2026"). Relative time provides faster temporal orientation.
- **`Chapter` type `status_label` gap:** `types.ts:24` has `status_label?: string` as optional. Both `GET /:slug` and dashboard endpoints always include it. Individual chapter endpoints (`GET /chapters/:id`, `PATCH /chapters/:id`) do not include it. The type correctly reflects this, but callers accessing `status_label` on chapters from project-level data must cast or assert. Consider a separate `ChapterWithLabel` type to make the contract explicit.
