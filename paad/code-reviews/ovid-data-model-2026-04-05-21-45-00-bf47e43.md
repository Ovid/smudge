## Important Issues

### [I1] afterAll destroys DB but never clears module singleton
- **File:** `packages/server/src/__tests__/test-helpers.ts:22-24`
- **Bug:** `afterAll` calls `testDb.destroy()` but never calls `closeDb()` or resets the singleton to `undefined`. The module-level `db` in `connection.ts` still references a destroyed Knex instance after teardown. Any code calling `getDb()` after teardown gets a destroyed connection instead of the expected "not initialized" error.
- **Impact:** Test isolation issue. If test files share the module singleton, a late call to `getDb()` silently uses a dead connection rather than throwing, leading to confusing failures.
- **Suggested fix:** Replace `await testDb.destroy()` with `await closeDb()` from `connection.ts`, which both destroys and clears the singleton.
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling, Concurrency & State

### [I2] Dual DB injection mechanisms in createApp
- **File:** `packages/server/src/app.ts:38-41`
- **Bug:** Old routers (`projectsRouter`, `chaptersRouter`) receive an explicit `db` parameter. New routers (`chapterStatusesRouter`, `settingsRouter`) take no parameter and call `getDb()` internally via their service layer. If the explicit `db` and the singleton ever diverge, the app would use two different database connections simultaneously.
- **Impact:** Split-brain risk. In tests, `setDb(testDb)` and `createApp(testDb)` happen to use the same instance, but the inconsistency is a maintenance trap.
- **Suggested fix:** Standardize on one approach. The plan's Task 8 removes the `db` parameter from `createApp` -- ensure this happens when remaining domains are extracted.
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling, Contract & Integration, Concurrency & State

### [I3] Duplicated getStatusLabel/getStatusLabelMap -- old copies still actively used
- **File:** `packages/server/src/routes/status-labels.ts` (old, actively imported)
- **File:** `packages/server/src/chapter-statuses/chapter-statuses.repository.ts:19-36` (new duplicate)
- **Bug:** `routes/chapters.ts` and `routes/projects.ts` import `getStatusLabel` and `getStatusLabelMap` from the old `routes/status-labels.ts`. The new `chapter-statuses.repository.ts` has identical functions that are unused by those routes. Bug fixes must be applied in two places.
- **Impact:** Maintenance hazard. Duplicated logic diverges over time and causes bugs.
- **Suggested fix:** When extracting chapters/projects domains (Tasks 5-6), switch imports to the new repository. Until then, document the duplication.
- **Confidence:** High
- **Found by:** Logic & Correctness, Contract & Integration

## Suggestions

- **[S1]** Dead code: `routes/chapter-statuses.ts` and `routes/settings.ts` still exist but are no longer imported by `app.ts`. Delete them. (Contract & Integration)
- **[S2]** `setDb()` bypasses WAL and foreign-key PRAGMA setup that `initDb()` performs. Test-helpers manually runs FK pragma but future callers may not. Consider having `setDb()` run PRAGMAs or documenting the requirement. (Logic & Correctness, Error Handling)
- **[S3]** Settings `upsert()` uses SELECT-then-INSERT/UPDATE instead of atomic `INSERT ... ON CONFLICT ... DO UPDATE`. Low risk for single-user SQLite but an anti-pattern. (Logic & Correctness, Error Handling)
- **[S4]** `beforeEach` in test-helpers does not reset `chapter_statuses` table. If any test mutates it, subsequent tests see stale state. (Error Handling)
- **[S5]** `toChapterStatus` mapping function in `chapter-statuses.service.ts` is a no-op (internal and shared types are identical). Intentional defensive design per the plan, but worth a comment. (Contract & Integration)
- **[S6]** Zod parse error message at `settings.routes.ts:22` is returned raw to the client. Consider a generic "Invalid request body" message. Low risk in single-user app. (Security)
