## Executive Summary

This refactoring extracts five domains (projects, chapters, velocity, settings, chapter-statuses) from flat route handlers into a layered routes/services/repositories architecture. The restructuring is substantially complete and well-executed. Three important issues were found: velocity side-effects can crash save/delete responses despite being documented as best-effort, two services bypass the repository layer with direct Knex calls, and `ProjectRepo.findById` doesn't filter soft-deleted records contrary to the documented convention. Eight additional suggestions address type-safety gaps, code duplication, and minor inconsistencies.

## Critical Issues

None found.

## Important Issues

### [I1] Velocity side-effects can crash save/delete/restore responses
- **File:** `packages/server/src/velocity/velocity.service.ts:222,241`
- **Call sites:** `packages/server/src/chapters/chapters.service.ts:104-111,141,180`, `packages/server/src/projects/projects.service.ts:189`
- **Bug:** `getTodayDate()` (which reads from the settings DB) is called before the try/catch blocks inside `recordSave()` and `updateDailySnapshot()`. If the settings read fails, the error propagates up and crashes the auto-save/delete/restore response even though the primary operation (content save, soft delete, restore) already succeeded. Additionally, the call sites in `chapters.service.ts` (lines 104-111, 141, 180) and `projects.service.ts` (line 189) have no try/catch wrapping the velocity calls at all.
- **Impact:** Undermines the "core trust promise" of reliable auto-save. A transient DB issue in the velocity subsystem would cause save failures visible to the user.
- **Suggested fix:** Wrap the velocity calls at each call site with try/catch, and/or move `getTodayDate()` inside the existing try/catch blocks in `recordSave` and `updateDailySnapshot`.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [I2] Direct `trx("projects")` calls in services bypass repository layer
- **File:** `packages/server/src/chapters/chapters.service.ts:170`, `packages/server/src/projects/projects.service.ts:161`
- **Bug:** Both services issue raw `trx("projects").where(...).update(...)` calls instead of using `ProjectRepo.update()`, which already exists and does the same thing. This breaks the encapsulation that the refactoring is specifically trying to achieve.
- **Impact:** Table/column name knowledge leaks into the service layer. Any future schema changes must be found in two places. Undermines the DB-swap feasibility goal stated in the design doc.
- **Suggested fix:** Replace both direct calls with `ProjectRepo.update(trx, id, updates)`.
- **Confidence:** High
- **Found by:** Contract & Integration, Plan Alignment

### [I3] `ProjectRepo.findById` does not filter `deleted_at IS NULL`
- **File:** `packages/server/src/projects/projects.repository.ts:12-16`
- **Bug:** `findById` returns projects regardless of soft-delete status, while `findBySlug` (line 19-24) correctly filters. CLAUDE.md states: "All queries must filter `deleted_at IS NULL`." The function is called after insert/update operations (projects.service.ts:79, 170) where returning soft-deleted rows would be unexpected, and in `restoreChapter` (chapters.service.ts:152, 185) where seeing deleted projects is intentional.
- **Impact:** Violates the documented soft-delete convention. Most callers expect active-only results.
- **Suggested fix:** Add `.whereNull("deleted_at")` to `findById` and create a separate `findByIdIncludingDeleted` for the restore path.
- **Confidence:** High
- **Found by:** Contract & Integration, Plan Alignment, Security

## Suggestions

- **[S1]** `packages/server/src/velocity/velocity.repository.ts:57` — `getRecentSaveEvents` SELECT omits `save_date` but return type is `SaveEventRow[]` which declares `save_date: string`. No caller currently accesses `save_date`, but this is a latent type-safety hole. Fix: add `"save_date"` to the select, or change the return type. *(Found by: Contract & Integration, Plan Alignment)*

- **[S2]** `packages/server/src/projects/projects.service.ts:277-279` — `getDashboard` manually builds a status label map via `Object.fromEntries(allStatuses.map(...))` instead of calling `ChapterStatusRepo.getStatusLabelMap(db)`, which is used at lines 99 and 222 in the same file. *(Found by: Contract & Integration)*

- **[S3]** `packages/server/src/settings/settings.repository.ts:17` — `upsert` accepts only `Knex.Transaction` while every other repository function across all domains accepts `Knex.Transaction | Knex`. Currently works because it's only called inside transactions, but inconsistent. *(Found by: Contract & Integration)*

- **[S4]** `packages/server/src/chapters/chapters.repository.ts:230-236` — `validateStatus` directly queries the `chapter_statuses` table, crossing domain boundaries. Should delegate to `ChapterStatusRepo.findByStatus()` which already exists. *(Found by: Plan Alignment)*

- **[S5]** `packages/server/src/velocity/velocity.service.ts:188-195` vs `packages/server/src/settings/settings.service.ts:5-11` — `safeTimezone` uses `Intl.DateTimeFormat` constructor while `SETTING_VALIDATORS.timezone` uses `Intl.supportedValuesOf`. These can disagree on which timezone strings are valid. Should use the same validation method. *(Found by: Error Handling & Edge Cases)*

- **[S6]** `packages/server/src/velocity/velocity.service.ts:331` — `dailyAvg30d` can go negative when chapters are soft-deleted, showing confusing negative words/day in the UI. Fix: `Math.max(0, Math.round(...))`. *(Found by: Error Handling & Edge Cases)*

- **[S7]** `packages/server/src/app.ts:55` — Global error handler maps all status < 500 to `VALIDATION_ERROR` code. A 413 or 429 from middleware would be mislabeled. Most routes craft their own error responses so this is a rare edge case. *(Found by: Error Handling & Edge Cases)*

- **[S8]** `packages/server/src/velocity/velocity.routes.ts` — Exports a single `velocityHandler` function rather than a `Router` factory like every other domain. Mounted inside projects router instead of independently in `app.ts`. Functionally correct but inconsistent pattern. *(Found by: Contract & Integration, Plan Alignment)*

## Plan Alignment

- **Implemented:** All five domain folders with types/repository/service/routes files; `setDb()` for test injection; `app.ts` mounts routers without passing `db`; services import `db` from singleton; repositories accept `Knex.Transaction | Knex`; velocity service injection for testing; `resolveUniqueSlug` in projects repository; dashboard in projects domain; helper files absorbed into domains.
- **Not yet implemented:** Complete type flow (services return `Record<string, unknown>` in several places instead of mapped shared types). This is incomplete but not a regression from the original code.
- **Deviations:** Two direct Knex calls in services bypass repositories (I2). Velocity routes don't follow the Router pattern (S8). Design doc specifies `Knex.Transaction` only but implementation pragmatically uses `Knex.Transaction | Knex` (correct choice, doc should be updated).
