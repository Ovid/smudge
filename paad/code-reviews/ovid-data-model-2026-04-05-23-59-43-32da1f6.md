## Executive Summary

This is a large-scale refactoring extracting the server package from monolithic route files into a layered architecture (routes → services → repositories) with domain-based grouping. The structural separation is solid and faithful to the design doc. The most significant issues are: (1) several repository and service functions return `Record<string, unknown>` instead of properly typed interfaces, undermining the type safety that was a core goal of the refactoring; (2) `deleteProject` has an unprotected velocity call that could cause a 500 after the delete already succeeded; and (3) a TOCTOU race in chapter reordering where validation runs outside the transaction.

## Critical Issues

None found.

## Important Issues

### [I1] `deleteProject` velocity call missing try/catch and injectable pattern
- **File:** `packages/server/src/projects/projects.service.ts:187`
- **Bug:** `deleteProject` calls `VelocityService.updateDailySnapshot(project.id)` without a try/catch. Every other velocity call site in `chapters.service.ts` (lines 100-109, 143-147, 189-192) wraps velocity calls in try/catch. Additionally, `deleteProject` imports `VelocityService` directly rather than using the `getVelocityService()` injectable pattern that `chapters.service.ts` established for testability.
- **Impact:** If the velocity call throws, the delete endpoint returns a 500 even though the soft-delete transaction already succeeded. The direct import also means this code path cannot use a test double for velocity failure testing.
- **Suggested fix:** Wrap in try/catch. Consider extracting the injectable velocity pattern into a shared utility used by both services.
- **Confidence:** High
- **Found by:** Error Handling, Contract & Integration (3 agents agreed)

### [I2] `ProjectRepo.findById` returns `undefined` while `ChapterRepo.findById` returns `null`
- **File:** `packages/server/src/projects/projects.repository.ts:12-16` vs `packages/server/src/chapters/chapters.repository.ts:43-49`
- **Bug:** `ProjectRepo.findById` returns `ProjectRow | undefined` (bare Knex `.first()`), while `ChapterRepo.findById` returns `ChapterRow | null` (explicit `row ? ... : null`). Same inconsistency on `findBySlug`, `findByIdIncludingDeleted`, and `findByTitle`.
- **Impact:** A caller checking `=== null` would miss the `undefined` case from ProjectRepo. Current callers use `if (!result)` which works, but the inconsistent contract is a correctness hazard for future callers.
- **Suggested fix:** Normalize all ProjectRepo find functions to return `| null` by adding `?? null`.
- **Confidence:** High
- **Found by:** Error Handling, Contract & Integration (2 agents agreed)

### [I3] `completion_threshold` typed as `string | null` with unsafe `as` cast
- **File:** `packages/server/src/projects/projects.types.ts:8` and `packages/server/src/velocity/velocity.service.ts:378`
- **Bug:** `ProjectRow.completion_threshold` is typed as `string | null`, while the shared `VelocityResponse.completion.threshold_status` expects `"outline" | "rough_draft" | "revised" | "edited" | "final" | null`. The velocity service casts with `as VelocityResponse["completion"]["threshold_status"]`, silencing the type checker.
- **Impact:** If the DB ever contains a stale or invalid value, the cast propagates an invalid type to the API response without any runtime validation.
- **Suggested fix:** Either import and use the shared `CompletionThresholdValue` type in `ProjectRow`, or validate at runtime before the cast.
- **Confidence:** High
- **Found by:** Contract & Integration

### [I4] `createChapter` leaks `content_corrupt` flag while `getProject` strips it
- **File:** `packages/server/src/projects/projects.service.ts:221-224` vs `projects.service.ts:101-107`
- **Bug:** In `getProject` (line 102), `content_corrupt` is explicitly stripped: `const { content_corrupt: _, ...rest } = ch`. But in `createChapter` (lines 221-224), the full chapter record is spread into the response without stripping.
- **Impact:** Inconsistent API contract. The internal `content_corrupt` flag could leak to the client if `parseContent` ever sets it on the created chapter.
- **Suggested fix:** Apply the same `content_corrupt` stripping pattern in `createChapter`.
- **Confidence:** Medium
- **Found by:** Contract & Integration

### [I5] Multiple repository/service functions return `Record<string, unknown>` instead of typed interfaces
- **File:** `packages/server/src/chapters/chapters.repository.ts` (lines 51-57, 59-64, 79-88, 90-110) and service return types throughout
- **Bug:** `findDeletedById`, `findByIdRaw`, `listMetadataByProject`, and `listDeletedByProject` return `Record<string, unknown>` instead of typed row interfaces. This propagates through the service layer, where `getProject`, `createChapter`, `getDashboard`, and `getTrash` all return untyped records. The plan specifies typed returns throughout.
- **Impact:** Undermines TypeScript's type safety — the core goal of the refactoring. Requires unsafe casts at boundaries and prevents compile-time detection of field-access bugs.
- **Suggested fix:** Define typed interfaces (`ChapterMetadataRow`, `DeletedChapterRow`, etc.) and use them as return types. Add mapping functions in services to produce shared types.
- **Confidence:** High
- **Found by:** Plan Alignment (6 sub-findings), Contract & Integration

### [I6] TOCTOU race in `reorderChapters` — chapter list validated outside transaction
- **File:** `packages/server/src/projects/projects.service.ts:243-258`
- **Bug:** `reorderChapters` reads existing chapter IDs at line 243 outside the transaction, validates them against the provided list, then opens a transaction at line 253 to apply the reorder. A concurrent request could add/delete a chapter between the check and the write.
- **Impact:** The reorder could succeed despite operating on a stale chapter set, breaking the API contract of "400 on mismatch."
- **Suggested fix:** Move the `listIdsByProject` call inside the transaction.
- **Confidence:** Medium
- **Found by:** Concurrency & State

### [I7] No graceful shutdown handler — WAL not checkpointed on SIGTERM
- **File:** `packages/server/src/index.ts`
- **Bug:** The server registers no SIGTERM/SIGINT handler. In Docker, a `docker stop` sends SIGTERM which will kill the process without closing the DB connection or checkpointing SQLite's WAL.
- **Impact:** Unclean WAL state on container restart. While SQLite can recover, this may leave large WAL files and, in rare edge cases, cause issues on restart.
- **Suggested fix:** Add a SIGTERM handler that calls `server.close()` then `closeDb()`.
- **Confidence:** Medium
- **Found by:** Concurrency & State

## Suggestions

- **[S3]** `packages/server/src/chapters/chapters.service.ts:153,195`: `restoreChapter` returns `null` for two distinct failure modes (chapter not found vs post-restore re-read failure), both mapped to the same 404. (Error Handling)
- **[S4]** `packages/server/src/chapters/chapters.service.ts:47`: `isCorruptChapter` uses double cast `as unknown as Record<string, unknown>`. Add `content_corrupt?: boolean` to `ChapterRow` to eliminate the cast. (Contract & Integration)
- **[S5]** `packages/shared/src/schemas.ts:39-44`: TipTap schema uses `.passthrough()` allowing arbitrary data storage. Document that this is intentional, or consider allowlisting known TipTap properties. (Security)
