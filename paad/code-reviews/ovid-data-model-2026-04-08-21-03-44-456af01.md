## Executive Summary

This large-scale refactoring extracts the server package from monolithic route files into a layered architecture (routes -> services -> repositories) with domain-based grouping, plus follow-up fixes from the previous code review. The structural separation is solid and the previous review's findings were addressed. The most significant remaining issues are: (1) a type mismatch between shared and server types for `completion_threshold` that could cause client-side null crashes; (2) the trash view is inaccessible for soft-deleted projects, leaving orphaned chapters unreachable; (3) the save pipeline can return a false 404 after a successful update; and (4) the graceful shutdown handler lacks a timeout.

## Important Issues

### [I1] Settings PATCH returns raw Zod error instead of consistent first-issue message
- **File:** `packages/server/src/settings/settings.routes.ts:23`
- **Bug:** Uses `parsed.error.message` which returns Zod's full verbose error string. Every other route in the codebase uses `parsed.error.issues[0]?.message ?? "Invalid input"` for a clean, human-readable first-issue message.
- **Impact:** API consumers receive an inconsistent, verbose error message from this one endpoint, violating the documented API error envelope contract (`"message": "Human-readable"`).
- **Suggested fix:** Change `parsed.error.message` to `parsed.error.issues[0]?.message ?? "Invalid input"`.
- **Confidence:** High
- **Found by:** Error Handling

### [I2] `getTrash` returns 404 for soft-deleted projects -- orphaned chapters unreachable
- **File:** `packages/server/src/projects/projects.service.ts:352-355`
- **Bug:** `getTrash` calls `findBySlug` which filters `deleted_at IS NULL`. When a project is soft-deleted, its trash view becomes inaccessible. Combined with the fact that restoring one chapter restores the project but leaves sibling chapters deleted, those siblings have no UI path for discovery or restoration.
- **Impact:** Users who delete a project and then restore one chapter cannot browse the remaining trashed chapters. They are effectively lost until the 30-day purge, accessible only if the user knows the chapter ID.
- **Suggested fix:** Use `findBySlugIncludingDeleted` (or create one) for the trash endpoint, since viewing a deleted project's trash is a valid use case.
- **Confidence:** High
- **Found by:** Error Handling (2 sub-findings merged)

### [I3] Restored chapter keeps stale `sort_order`, may collide with existing chapters
- **File:** `packages/server/src/chapters/chapters.repository.ts:224-228`
- **Bug:** `restore()` clears `deleted_at` and updates `updated_at` but preserves the original `sort_order`. If chapters were reordered after the deletion, the restored chapter's old `sort_order` value can collide with an existing chapter's position, producing unpredictable ordering in the sidebar.
- **Impact:** Chapter list may display in wrong order after restore. Two chapters could share the same `sort_order` with no deterministic tiebreaker.
- **Suggested fix:** In the restore flow, compute `getMaxSortOrder + 1` for the project and assign it to the restored chapter, placing it at the end (consistent with `createChapter`'s approach).
- **Confidence:** Medium
- **Found by:** Logic & Correctness

### [I4] `updateChapter` returns misleading 404 after successful database update
- **File:** `packages/server/src/chapters/chapters.service.ts:108-109`
- **Bug:** After the transaction at lines 93-96 successfully updates the chapter, the re-read at line 108 could return null (e.g., concurrent soft-delete between the transaction and re-read). The service returns `null`, which the route maps to 404 "Chapter not found" -- but the update already committed successfully. The client receives a false save failure.
- **Impact:** The save pipeline is the core trust promise (per CLAUDE.md). A false 404 after a successful save triggers unnecessary retries and could show "Unable to save" warnings when data was actually persisted.
- **Suggested fix:** Return a distinct error sentinel (e.g., `"read_after_update_failure"`) and handle it as a 500 in the route, not a 404. This follows the pattern already established for `restoreChapter`'s `"read_failure"` return.
- **Confidence:** Medium
- **Found by:** Logic & Correctness, Error Handling, Contract & Integration (3 agents agreed)

### [I5] `setDb()` bypasses WAL mode and foreign key PRAGMAs
- **File:** `packages/server/src/db/connection.ts:13-15`
- **Bug:** `setDb()` directly assigns the Knex instance without running `PRAGMA journal_mode = WAL` or `PRAGMA foreign_keys = ON`, which `initDb()` always does (lines 19-20). The test helper uses `setDb()`, so the entire test suite runs without foreign key enforcement.
- **Impact:** Tests will not catch FK constraint violations (e.g., inserting a chapter with a nonexistent `project_id`). FK-related bugs could ship to production undetected.
- **Suggested fix:** Run `PRAGMA foreign_keys = ON` in the test helper after `setDb()`, or have `setDb()` accept a flag to run PRAGMAs.
- **Confidence:** High
- **Found by:** Logic & Correctness

### [I6] Graceful shutdown has no timeout -- can hang indefinitely
- **File:** `packages/server/src/index.ts:58-68`
- **Bug:** `server.close()` waits for all keep-alive connections to finish before invoking the callback. There is no `setTimeout` to force exit. If any connection stays open, the process hangs and never calls `closeDb()`.
- **Impact:** In Docker, SIGTERM could leave the container hanging until Docker's SIGKILL fires, potentially leaving the SQLite WAL in an unclean state -- the exact scenario the shutdown handler was added to prevent.
- **Suggested fix:** Add a `setTimeout(() => process.exit(1), 10000)` after `server.close()` as a safety fallback.
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling (2 agents agreed)

### [I7] Shared `Project.completion_threshold` is non-nullable but server type allows null
- **File:** `packages/shared/src/types.ts:14` vs `packages/server/src/projects/projects.types.ts:10`
- **Bug:** The shared `Project` interface types `completion_threshold` as `CompletionThresholdValue` (non-nullable: `"outline" | "rough_draft" | "revised" | "edited" | "final"`). The server `ProjectRow` types it as `CompletionThresholdValue | null`. Since services return `ProjectRow` directly without mapping to the shared type, the API can send `null` for this field while client code assumes it's always present.
- **Impact:** Client-side TypeScript won't guard against null, potentially causing runtime crashes when accessing `completion_threshold` on a freshly created project before it's been configured.
- **Suggested fix:** Align the types -- either make the shared type nullable (`CompletionThresholdValue | null`) or ensure the DB always provides a default (migration 004 sets `defaultTo("final")`, so the DB side may already be correct -- verify and align the TS type accordingly).
- **Confidence:** High
- **Found by:** Contract & Integration

### [I8] `stripCorruptFlag` is dead code -- inline destructuring duplicated across 5 call sites
- **File:** `packages/server/src/chapters/chapters.service.ts:36-39`
- **Bug:** `stripCorruptFlag` is defined, exported, and tested, but never called. Instead, `const { content_corrupt: _, ...rest } = chapter` is repeated inline at 5 separate locations: `chapters.service.ts:50,116,189` and `projects.service.ts:143,265`.
- **Impact:** If the corrupt flag field name changes or stripping logic needs updating, 5 sites need manual changes. The utility exists precisely for this purpose but is unused.
- **Suggested fix:** Replace the 5 inline destructuring sites with calls to `stripCorruptFlag`, or move the stripping into the repository layer so chapters are returned clean.
- **Confidence:** High
- **Found by:** Contract & Integration (2 sub-findings merged)

## Suggestions

- **[S1]** `packages/server/src/velocity/velocity.repository.ts:17-18`: `saved_at` uses UTC but `save_date` uses the user's timezone. Near midnight they can disagree, causing sessions and streaks to attribute saves to different days. (Logic & Correctness)
- **[S2]** `packages/server/src/velocity/velocity.service.ts:210-235`: `recordSave` inserts the save event and upserts the snapshot in separate operations, not in a single transaction. If the upsert fails, the data is temporarily inconsistent. (Concurrency & State)
- **[S3]** `packages/server/src/chapters/chapters.service.ts:16` and `packages/server/src/projects/projects.service.ts:44`: Module-level mutable `velocityServiceOverride` singletons persist across tests. A test failure before `resetVelocityService()` poisons subsequent tests. (Concurrency & State)
- **[S4]** Multiple service functions (`projects.service.ts:70`, `chapters.service.ts:57`, etc.): Services accept `body: unknown` and do Zod parsing internally. The plan specifies routes should parse and pass typed data to services. (Plan Alignment)
