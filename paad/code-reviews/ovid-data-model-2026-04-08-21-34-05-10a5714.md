## Executive Summary

The layered architecture extraction is structurally sound and the previous review's important issues (I1-I8) have been addressed. The remaining issues are concentrated in three areas: (1) the velocity domain has a timezone mismatch between `saved_at` (UTC) and `save_date` (user timezone) that causes inconsistencies near midnight; (2) `getProject` silently serves corrupt chapters as `content: null` with no indicator, unlike the single-chapter endpoint which properly returns a corruption error; and (3) the velocity service injection pattern is duplicated and divergent across two files.

## Important Issues

### [I1] `restoreChapter` skips project `updated_at` when project is not deleted
- **File:** `packages/server/src/chapters/chapters.service.ts:161-179`
- **Bug:** When restoring a chapter whose parent project is NOT deleted (the common case), the transaction only calls `ChapterRepo.restore()` but never calls `ProjectRepo.updateTimestamp()`. The project timestamp is only updated inside the `if (parentProject.deleted_at)` block. Compare with `deleteChapter` (line 138-141) and `createChapter` (projects.service.ts:260) which both update the project timestamp after chapter mutations.
- **Impact:** Project list ordering and "last edited" displays will show stale timestamps after chapter restore.
- **Suggested fix:** Add `await ProjectRepo.updateTimestamp(trx, chapter.project_id)` inside the transaction, outside the `if (parentProject.deleted_at)` block.
- **Confidence:** High
- **Found by:** Logic & Correctness A

### [I2] `getProject` silently serves corrupt chapters as `content: null`
- **File:** `packages/server/src/projects/projects.service.ts:143-149`
- **Bug:** When `parseContent` detects corrupt JSON in a chapter (repository line 20), the chapter gets `content: null, content_corrupt: true`. After `stripCorruptFlag` removes the flag, the client receives `content: null` with no corruption indicator. Compare with `getChapter` (chapters.service.ts:43-53) which explicitly returns `"corrupt"` when `isCorruptChapter` is true. The project endpoint loses this signal entirely.
- **Impact:** A user opening a project sees a chapter that looks empty rather than corrupted. They might start typing, overwriting potentially recoverable data.
- **Suggested fix:** Include a `content_corrupt` boolean in the chapter response within the project endpoint, or filter corrupt chapters into a separate response field.
- **Confidence:** High
- **Found by:** Error Handling

### [I3] Duplicated and divergent `VelocityServiceInterface` + injection pattern
- **File:** `packages/server/src/chapters/chapters.service.ts:11-28` and `packages/server/src/projects/projects.service.ts:41-57`
- **Bug:** The `VelocityServiceInterface` type, the override variable, `setVelocityService()`, `resetVelocityService()`, and `getVelocityService()` are fully duplicated in two files. The two interface declarations differ: chapters version declares `{ recordSave, updateDailySnapshot }` while projects version declares only `{ updateDailySnapshot }`. These are independent copies that will diverge silently if the velocity service API changes.
- **Impact:** Changes to velocity service signatures must be synchronized across two files. The divergent interfaces mean test stubs must be set up separately in each module.
- **Suggested fix:** Extract a single `VelocityServiceInterface` and the injection mechanism into a shared module (e.g., `velocity/velocity.injectable.ts`), imported by both services.
- **Confidence:** High
- **Found by:** Contract & Integration, Concurrency & State

### [I4] `saved_at` uses UTC while `save_date` uses timezone-adjusted date
- **File:** `packages/server/src/velocity/velocity.repository.ts:17` and `packages/server/src/velocity/velocity.service.ts:191-205`
- **Bug:** `saved_at` is `new Date().toISOString()` (UTC), while `save_date` comes from `getTodayDate()` using the user's timezone. Session window filtering uses `saved_at >= sinceTimestamp` (UTC-based) while streak calculations use `save_date` (timezone-based). Near timezone boundaries, a save event could count toward one day in sessions but a different day in streaks.
- **Impact:** For users far from UTC (e.g., UTC+12), writing sessions and streaks can be inconsistent near midnight. A session could appear on one day while the streak counts it toward another.
- **Suggested fix:** Use the same time basis consistently -- either filter sessions by `save_date` (timezone-aware) or compute streaks from `saved_at` (UTC).
- **Confidence:** Medium
- **Found by:** Logic & Correctness B
