## Important Issues

### [I1] restoreChapter missing project timestamp update
- **File:** `packages/server/src/chapters/chapters.service.ts:162-179`
- **Bug:** When restoring a chapter whose parent project is NOT soft-deleted, the transaction restores the chapter but never calls `ProjectRepo.updateTimestamp()`. The timestamp update only happens inside the `if (parentProject.deleted_at)` branch. This is inconsistent with `deleteChapter` (line 140), `updateChapter` (line 101), and `createChapter` which all update the project timestamp.
- **Impact:** Project's `updated_at` becomes stale after chapter restore, causing incorrect sort order on the project list and misleading "last edited" display.
- **Suggested fix:** Add `await ProjectRepo.updateTimestamp(trx, chapter.project_id);` inside the transaction at line 164, outside the `if (parentProject.deleted_at)` block.
- **Confidence:** High
- **Found by:** Logic & Correctness A

### [I2] Dead code — empty if-block in catch unconditionally falls through to throw
- **File:** `packages/server/src/projects/projects.service.ts:114-121` (also lines 205-212)
- **Bug:** The catch block checks `if (err.message.includes("UNIQUE constraint failed"))` with an empty body (just a comment), then unconditionally executes `throw err`. The if-block does nothing — the error is always rethrown regardless.
- **Impact:** Confusing dead code that suggests special handling was intended but never implemented. A future developer might add logic after the `if` without realizing the `throw` below executes unconditionally.
- **Suggested fix:** Remove the empty if-block and keep the comment with the `throw err`, or restructure to make intent unambiguous.
- **Confidence:** High
- **Found by:** Logic & Correctness A, Error Handling, Concurrency & State

### [I3] Fragile string matching for SQLite error detection
- **File:** `packages/server/src/chapters/chapters.service.ts:181`, `packages/server/src/projects/projects.service.ts:115,206`
- **Bug:** SQLite errors are detected via `err.message.includes("UNIQUE constraint failed")` which depends on the exact English error message text from SQLite. Different SQLite versions or Knex wrappers could change this text.
- **Impact:** Error handling could silently break on SQLite/Knex upgrades, leading to unhandled errors or incorrect error responses.
- **Suggested fix:** Check for SQLite error codes instead (e.g., `err.code === 'SQLITE_CONSTRAINT_UNIQUE'`).
- **Confidence:** Medium
- **Found by:** Error Handling

### [I4] Duplicated VelocityServiceInterface with divergent definitions
- **File:** `packages/server/src/chapters/chapters.service.ts:11-28` and `packages/server/src/projects/projects.service.ts:41-57`
- **Bug:** Both files independently define `VelocityServiceInterface`, `velocityServiceOverride`, and the `set/reset/getVelocityService` functions. The interfaces differ: chapters has `recordSave` + `updateDailySnapshot`; projects has only `updateDailySnapshot`. These are fully independent copies.
- **Impact:** Maintenance burden — changes to the velocity service API require updating two independent copies. Tests must mock two separate overrides. The divergent interfaces could mask missing method calls.
- **Suggested fix:** Extract a single shared velocity injection module with a unified interface imported by both services.
- **Confidence:** High
- **Found by:** Logic & Correctness A, Contract & Integration

### [I5] Chapter restore hardcodes `new Date().toISOString()` for updated_at
- **File:** `packages/server/src/chapters/chapters.repository.ts:224-232`
- **Bug:** The `restore` function computes `updated_at: new Date().toISOString()` internally rather than accepting a `now` parameter. Other mutation functions (`softDelete`, `softDeleteByProject`, `insert`) accept a `now` parameter for consistency and testability.
- **Impact:** Inconsistent timestamp handling; harder to test deterministically; the restore timestamp could differ from other timestamps set in the same transaction.
- **Suggested fix:** Add a `now: string` parameter to `restore()` and pass it from the caller.
- **Confidence:** Medium
- **Found by:** Logic & Correctness A

### [I6] createChapter returns null after successful insert (misleading 500)
- **File:** `packages/server/src/projects/projects.service.ts:267-268`
- **Bug:** After successfully inserting a chapter in a transaction, the re-read at line 267 could theoretically return null. The route interprets null as a 500 `INTERNAL_ERROR`, but the chapter IS created. The client may retry, potentially creating duplicates. Compare with `updateChapter`'s `"read_after_update_failure"` sentinel.
- **Impact:** Client sees 500 for a chapter that was actually created. Retry could create duplicates.
- **Suggested fix:** Return a distinct sentinel like `"read_after_create_failure"` so the route can communicate the situation accurately and prevent retry.
- **Confidence:** Medium
- **Found by:** Logic & Correctness A

### [I7] Title uniqueness check outside transaction — TOCTOU race
- **File:** `packages/server/src/projects/projects.service.ts:81-84` (also `168-175` for `updateProject`)
- **Bug:** `createProject` checks title uniqueness at line 81 outside the transaction at line 91. Two concurrent requests with the same title could both pass the check. Since there's no DB-level UNIQUE constraint on `title` (only on `slug`), both inserts would succeed.
- **Impact:** Duplicate project titles in the database, violating the application's intended uniqueness invariant.
- **Suggested fix:** Move the title uniqueness check inside the transaction, or add a partial unique index on `(title) WHERE deleted_at IS NULL`.
- **Confidence:** High
- **Found by:** Concurrency & State

### [I8] restoreChapter reads parent project outside transaction — TOCTOU
- **File:** `packages/server/src/chapters/chapters.service.ts:154-162`
- **Bug:** `restoreChapter` reads the parent project at line 158 outside the transaction starting at line 162. Between the read and the transaction, the project could be purged, causing the transaction to operate on a non-existent project.
- **Impact:** Could result in a restored chapter pointing to a purged project (orphaned data).
- **Suggested fix:** Move the parent project lookup inside the transaction.
- **Confidence:** Medium
- **Found by:** Error Handling

### [I9] ProjectRepo.insert/update cast potentially-undefined result as ProjectRow
- **File:** `packages/server/src/projects/projects.repository.ts:8-9` (also line 82-83)
- **Bug:** `insert` does `return trx("projects").where({ id: data.id }).first() as Promise<ProjectRow>`. Knex's `.first()` returns `T | undefined`, but the cast hides the `undefined` possibility.
- **Impact:** Type-safety hole — callers assume non-null `ProjectRow` but could receive `undefined` at runtime.
- **Suggested fix:** Handle the `undefined` case explicitly or add a runtime check.
- **Confidence:** Medium
- **Found by:** Error Handling

### [I10] CLAUDE.md data model section is stale
- **File:** `CLAUDE.md` (Data Model section)
- **Bug:** CLAUDE.md says "Two tables: Project and Chapter" but the codebase has 6 tables: projects, chapters, chapter_statuses, settings, save_events, daily_snapshots. Column lists are also incomplete (missing slug, target_word_count, status, etc.).
- **Impact:** AI agents and developers relying on CLAUDE.md for architectural guidance will have an incorrect mental model of the database.
- **Suggested fix:** Update the Data Model section to list all 6 tables with their current columns. Update the project structure section to reflect the new domain-module layout.
- **Confidence:** High
- **Found by:** Error Handling, Contract & Integration
