## Executive Summary

This branch refactors the server from flat route handlers into a layered domain architecture (Routes -> Services -> Repositories) across five domains. The refactoring is well-executed: API contracts are preserved, transaction boundaries are mostly correct, and all planned tasks are implemented. Two important issues were found: a TOCTOU race on the auto-save hot path where `updateChapter` can silently modify a soft-deleted chapter, and overly permissive `Record<string, unknown>` type signatures on repository update functions that bypass compile-time safety. No critical issues found.

## Critical Issues

None found.

## Important Issues

### [I1] `updateChapter` TOCTOU race on auto-save hot path -- FIXED (8e9c054)
- **File:** `packages/server/src/chapters/chapters.repository.ts`
- **Fix:** Added `whereNull("deleted_at")` to `ChapterRepo.update`. A concurrent delete during auto-save now results in a no-op (0 rows affected) instead of silently modifying a trashed chapter. Return type changed to `Promise<number>` so callers can detect the no-op.

### [I2] Repository update methods accept `Record<string, unknown>` -- FIXED (6b24a25)
- **Files:** `packages/server/src/projects/projects.types.ts`, `packages/server/src/chapters/chapters.types.ts`
- **Fix:** Introduced `UpdateProjectData` and `UpdateChapterData` typed interfaces. Repository `update` methods now accept these instead of `Record<string, unknown>`, providing compile-time safety against accidental writes to immutable columns.

## Suggestions

### [S1] `updateProject` slug lookup outside transaction
- **File:** `packages/server/src/projects/projects.service.ts:145`
- `findBySlug` called outside the transaction at line 162. Stale project reference used inside. Minimal real-world risk in single-user app but architecturally inconsistent with the title-uniqueness check (which correctly runs inside the transaction).

### [S2] `createProject` discards transaction return value
- **File:** `packages/server/src/projects/projects.service.ts:71-104`
- `ProjectRepo.insert` returns the row, but the service ignores it and re-reads outside the transaction at line 100. Unnecessary extra query.

### [S3] `recordSave` non-atomic operations
- **File:** `packages/server/src/velocity/velocity.service.ts:210-236`
- Three separate DB operations (insertSaveEvent, sumWordCountByProject, upsertDailySnapshot) without a transaction. Eventually consistent via `ON CONFLICT ... DO UPDATE`. Acceptable given velocity is explicitly best-effort by design.

### [S4] `recordSave` catch masks infrastructure failures
- **File:** `packages/server/src/velocity/velocity.service.ts:215,233`
- `getDb()` failure caught and logged identically to transient velocity failures. Low practical impact since infrastructure failures would manifest elsewhere first.

### [S5] Duplicated status-label enrichment across services
- Status-label enrichment logic appears in 5+ places across `projects.service.ts` and `chapters.service.ts` using different strategies (single lookup vs batch map). A shared helper would consolidate the strip-corrupt-flag + status-label-lookup pattern.

### [S6] No type mapping layer between DB rows and API types
- **File:** `packages/server/src/projects/projects.service.ts`
- Services return `ProjectRow` directly to routes. Design doc calls for `toProject()` mapping functions to decouple DB schema from API contract. No current functional impact.

### [S7] `getDashboard` queries `chapter_statuses` twice
- **File:** `packages/server/src/projects/projects.service.ts:285-286`
- Both `ChapterStatusRepo.list(db)` and `getStatusLabelMap(db)` query the same table. The label map could be derived from the list result without a second query.

### [S8] `getVelocityBySlug` queries `chapter_statuses` twice
- **File:** `packages/server/src/velocity/velocity.service.ts:357-359`
- `findByStatus()` then `list()` called separately. The threshold row is already in the list result.

### [S9] `ChapterRepo.insert` returns `void` vs `ProjectRepo.insert` returns `ProjectRow`
- **File:** `packages/server/src/chapters/chapters.repository.ts:172`
- Inconsistent repository API. Callers of `ChapterRepo.insert` must do separate re-reads.

### [S10] Velocity injectable re-exported from two service modules
- **Files:** `packages/server/src/projects/projects.service.ts:26`, `packages/server/src/chapters/chapters.service.ts:13`
- Tests can import `setVelocityService`/`resetVelocityService` from either service, creating a confusing import surface for the same global singleton. Tests should import directly from `velocity/velocity.injectable.ts`.

### [S11] `getChapterNamesMap` includes deleted chapters without naming indication
- **File:** `packages/server/src/chapters/chapters.repository.ts:147-157`
- No `deleted_at IS NULL` filter, intentionally correct for velocity tracking (shows names of deleted chapters in session history). Function name should indicate this (e.g., `getAllChapterNamesMap`).
