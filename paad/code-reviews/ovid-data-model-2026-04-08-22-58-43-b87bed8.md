## Important Issues

### [I1] `ProjectRepo.update` lacks `deleted_at` filter — can mutate soft-deleted projects
- **File:** `packages/server/src/projects/projects.repository.ts:84-93`
- **Bug:** `update()` uses `trx("projects").where({ id }).update(data)` without `.whereNull("deleted_at")`. Contrast with `ChapterRepo.update()` at `chapters.repository.ts:194` which includes the guard. Commit 8e9c054 explicitly fixed this for chapters but the same fix was not applied to projects.
- **Impact:** If `updateProject` service has a TOCTOU race (project deleted between `findBySlug` and `update`), the update succeeds on a soft-deleted project — potentially resurrecting its slug into the active namespace. The `restoreChapter` flow intentionally updates deleted projects, so a separate `updateIncludingDeleted` method is needed for that path.
- **Suggested fix:** Add `.whereNull("deleted_at")` to the main `update()` method. Create a separate `updateIncludingDeleted()` for the restore path in `chapters.service.ts:166-170`.
- **Confidence:** High
- **Found by:** Logic & Correctness, Contract & Integration

### [I2] `updateChapter` ignores update return value — velocity fires for no-op updates
- **File:** `packages/server/src/chapters/chapters.service.ts:89-101`
- **Bug:** `ChapterRepo.update()` returns the number of rows affected, but the service discards this value at line 90. If the chapter was concurrently deleted between the `findByIdRaw` check (line 61) and the `update` call (line 90), the update affects 0 rows (because `ChapterRepo.update` filters `deleted_at IS NULL`), but velocity side-effects at lines 95-101 still fire, recording a phantom save event. The `findById` re-read at line 104 returns null, and the service returns `"read_after_update_failure"` (a 500 error) when it's semantically a 404.
- **Impact:** Phantom velocity data for deleted chapters. Misleading 500 error instead of 404 for the auto-save client.
- **Suggested fix:** Check the return value of `ChapterRepo.update()`. If 0 rows affected, return `null` (not found) before firing velocity.
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling, Concurrency & State (4 specialists agreed)

### [I3] `deleteProject` records 0-word daily snapshot after soft-deleting chapters
- **File:** `packages/server/src/projects/projects.service.ts:193-203`
- **Bug:** The transaction at lines 193-196 soft-deletes all chapters first, then calls `updateDailySnapshot` at line 199 (outside the transaction). `sumWordCountByProject` filters `whereNull("deleted_at")`, so after soft-deleting all chapters it returns 0. This upserts a daily snapshot with `total_word_count = 0`.
- **Impact:** If the project is later restored, the velocity history shows a false dip to 0 words on the deletion date. The 30-day daily average calculation is also affected by this phantom zero.
- **Suggested fix:** Either skip the daily snapshot on delete, or record it before performing the soft-delete transaction.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [I4] `SMUDGE_PORT` NaN causes silent random port binding
- **File:** `packages/server/src/index.ts:6`
- **Bug:** `parseInt(process.env.SMUDGE_PORT ?? "3456", 10)` returns `NaN` for non-numeric values like `"abc"`. Node's `http.Server.listen(NaN)` treats it as `listen(0)`, binding to a random port with no error.
- **Impact:** Server starts on an unreachable port with no diagnostic message. In a Docker deployment, the container appears running but the health check fails.
- **Suggested fix:** Add `Number.isNaN(PORT)` check after parsing and exit with a clear error message.
- **Confidence:** High
- **Found by:** Security

## Suggestions

- **[S1]** `chapters.repository.ts:148-158` — `getChapterNamesMap` intentionally includes deleted chapters (for velocity chapter name display) but doesn't document this. Rename to `getChapterNamesMapIncludingDeleted` or add a doc comment. *(Found by: Plan Alignment)*

- **[S2]** `chapters.repository.ts:197-204` — `updateSortOrders` doesn't filter `deleted_at IS NULL`. Callers only pass non-deleted IDs, so this is defense-in-depth only. *(Found by: Plan Alignment)*

- **[S3]** `chapters.service.ts:177` — `SQLITE_CONSTRAINT_UNIQUE` catch in `restoreChapter` is likely unreachable because `resolveUniqueSlug` already prevents slug collisions within the same transaction. If kept as a safety net, the comment should note it's defensive. If a future unique constraint is added, this catch could misinterpret the error. *(Found by: Logic & Correctness, Error Handling, Contract & Integration)*

- **[S4]** `db/connection.ts:13-17` — `setDb()` doesn't close the previous Knex instance before reassigning. Resource leak in test environments where connections and file handles accumulate. *(Found by: Concurrency & State)*

- **[S5]** `projects/projects.types.ts:31` — `UpdateProjectData.completion_threshold` is typed `string | null` but the DB column is `NOT NULL DEFAULT 'final'`. The Zod schema likely prevents null at validation, but the TypeScript type is misleading and could cause a runtime constraint violation if used directly. *(Found by: Contract & Integration)*

- **[S6]** `index.ts:83-88` — `SQLITE_IOERR` detection is overly broad. Transient I/O errors (disk full, permissions) trigger the message "Database file is corrupt or has stale WAL files" which could lead users to unnecessarily delete their database. *(Found by: Error Handling)*

- **[S7]** `db/connection.ts:13-17` — `setDb()` doesn't enable `PRAGMA journal_mode = WAL` unlike `initDb()`. Test behavior doesn't perfectly match production. Since `setDb` is test-only and uses in-memory SQLite, this is low impact. *(Found by: Logic & Correctness)*
