## Important Issues

### [I1] `updateProject` TOCTOU: concurrent soft-delete causes 500 instead of 404
- **File:** `packages/server/src/projects/projects.service.ts:139` and `packages/server/src/projects/projects.repository.ts:89-92`
- **Bug:** `findBySlug` is called outside the transaction (line 139). If the project is soft-deleted between that read and the `ProjectRepo.update` call inside the transaction, `update` silently affects 0 rows (it filters `whereNull("deleted_at")`), then re-reads and finds nothing, throwing `Error("Project ${id} not found after update")`. This propagates as an unhandled 500 instead of a 404.
- **Impact:** An operation that should 404 (resource gone) returns 500, violating the API contract. Low probability in a single-user app but the error response is still wrong.
- **Suggested fix:** Either move `findBySlug` inside the transaction, or check rows affected in `ProjectRepo.update` and return a typed signal (e.g., `null`) instead of throwing.
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling, Concurrency & State

### [I2] `initDb` leaks old Knex connection pool if called twice
- **File:** `packages/server/src/db/connection.ts:22-29`
- **Bug:** `initDb` unconditionally sets `db = knex(...)` without checking if `db` is already set or calling `destroy()` on the old connection. If called twice (e.g., double-initialization in tests), the old Knex connection pool is leaked — file handles to the SQLite file are left open and WAL files accumulate. `setDb` handles this correctly (it destroys the old instance), but `initDb` does not.
- **Impact:** Connection pool leak in test environments; potential WAL lock contention on shared SQLite file.
- **Suggested fix:** Add a guard: `if (db) await db.destroy();` before creating the new instance.
- **Confidence:** High
- **Found by:** Concurrency & State

## Suggestions

### [S1] Empty `catch {}` on velocity side-effects discards all errors silently
- **File:** `packages/server/src/chapters/chapters.service.ts:99-105` (also lines 139, 191)
- **Bug:** The `catch {}` blocks around velocity calls (`recordSave`, `updateDailySnapshot`) discard all errors including those from `getVelocityService()` itself. The velocity service has its own internal `console.error` logging, but the outer catch discards anything that escapes. Additionally, `updates.word_count as number` (line 102) is a fragile cast that bypasses type narrowing.
- **Suggested fix:** `catch (err) { console.error("Velocity side-effect failed:", err); }` at the call site. Replace `as number` with a proper guard.
- **Found by:** Logic & Correctness, Error Handling, Contract & Integration

### [S2] `VelocityResponse.completion.threshold_status` typed as nullable but never null
- **File:** `packages/shared/src/types.ts` (VelocityResponse type) and `packages/server/src/velocity/velocity.service.ts:369`
- **Bug:** The shared type allows `threshold_status: CompletionThresholdValue | null`, but the service always assigns `project.completion_threshold` which is non-nullable (DB column has `NOT NULL DEFAULT 'final'`). Clients guard against null unnecessarily.
- **Suggested fix:** Change to `threshold_status: CompletionThresholdValue` in the shared type.
- **Found by:** Contract & Integration

### [S3] `getDashboard` reimplements `getStatusLabelMap` by hand
- **File:** `packages/server/src/projects/projects.service.ts:279-283`
- **Bug:** `getDashboard` manually builds `statusLabelMap` with a for-loop over `ChapterStatusRepo.list(db)`, while `ChapterStatusRepo.getStatusLabelMap(db)` already does exactly this and is used correctly in `getProject` and `createChapter`.
- **Suggested fix:** Use `ChapterStatusRepo.getStatusLabelMap(db)` instead of the manual build.
- **Found by:** Contract & Integration

### [S4] `getVelocityBySlug` builds `statusSortMap` independently; no shared helper
- **File:** `packages/server/src/velocity/velocity.service.ts:356-361`
- **Bug:** Builds `statusSortMap` with the same for-loop pattern as the label map duplication in S3. No shared `getStatusSortMap` helper exists in the chapter-statuses repository.
- **Suggested fix:** Add `getStatusSortMap(): Promise<Record<string, number>>` to `chapter-statuses.repository.ts`.
- **Found by:** Contract & Integration

### [S5] `projects.service.ts` re-exports `setVelocityService`/`resetVelocityService` — unused
- **File:** `packages/server/src/projects/projects.service.ts:14,22`
- **Bug:** `projects.service` imports and re-exports velocity injection functions, but never uses them. All actual consumers import from `chapters.service` or `velocity.injectable` directly. This is a dead re-export that misleads readers.
- **Suggested fix:** Remove the import and re-export from `projects.service.ts`.
- **Found by:** Contract & Integration, Plan Alignment

### [S6] `updated_at` timestamp captured before transaction in `updateProject`
- **File:** `packages/server/src/projects/projects.service.ts:142-144`
- **Bug:** `updated_at: new Date().toISOString()` is computed before the transaction begins (line 156). If the transaction takes time to acquire a lock, the timestamp reflects pre-lock time. A concurrent `createChapter` could set a later `updated_at` via `updateTimestamp`, then `updateProject`'s commit overwrites it with the earlier timestamp, rolling it backward.
- **Suggested fix:** Move `new Date().toISOString()` inside the transaction body.
- **Found by:** Concurrency & State

### [S7] `initDb` enables FK enforcement before migrations
- **File:** `packages/server/src/db/connection.ts:25-27`
- **Bug:** `PRAGMA foreign_keys = ON` is set at line 26, before `db.migrate.latest()` at line 27. SQLite documentation recommends FK enforcement be disabled during schema migrations that drop/recreate tables. Migration 005 drops and recreates tables with `ON DELETE CASCADE`.
- **Suggested fix:** Move `PRAGMA foreign_keys = ON` after `db.migrate.latest()`.
- **Found by:** Error Handling

### [S8] `unsafe-inline` in `styleSrc` CSP
- **File:** `packages/server/src/app.ts:26`
- **Bug:** `'unsafe-inline'` for styles allows injected `<style>` tags, weakening XSS protection. Likely required for TipTap/Tailwind but should be documented.
- **Suggested fix:** Document the trade-off; consider nonce-based or hash-based approach if feasible.
- **Found by:** Security
