## Important Issues

### [I1] `initDb()` does not destroy existing connection before replacing it
- **File:** `packages/server/src/db/connection.ts:22-28`
- **Bug:** `initDb()` directly assigns `db = knex(...)` without checking if `db` already exists and calling `destroy()`. In contrast, `setDb()` properly destroys the old instance. If `initDb()` is called twice, the first Knex instance is leaked.
- **Impact:** Resource leak — file handles and WAL locks on SQLite could accumulate, causing `SQLITE_BUSY` or resource exhaustion in test scenarios or retry paths.
- **Suggested fix:** Add `if (db) { await db.destroy(); }` at the top of `initDb()`, or route through `setDb()`.
- **Confidence:** High
- **Found by:** Error Handling, Concurrency & State

### [I2] `project_slug: string | undefined` silently omitted from JSON response on restore
- **File:** `packages/server/src/chapters/chapters.types.ts:58` and `packages/server/src/chapters/chapters.service.ts:200-207`
- **Bug:** `RestoredChapterResponse` defines `project_slug: string | undefined`. If `ProjectRepo.findById` returns null after the restore transaction commits, `project_slug` is `undefined`. `JSON.stringify` silently omits `undefined` fields, so the client receives a response with no `project_slug` at all.
- **Impact:** The client needs the slug to navigate to the restored project. A missing field could cause client-side errors or broken navigation after restore.
- **Suggested fix:** Assert that `updatedProject` exists (it should, since the project was just restored in the committed transaction), or change the type to `string | null` with explicit assignment: `project_slug: updatedProject?.slug ?? null`.
- **Confidence:** Medium
- **Found by:** Contract & Integration

## Suggestions

- **[S1]** `packages/server/src/chapters/chapters.service.ts:177-186` — `SQLITE_CONSTRAINT_UNIQUE` catch in `restoreChapter` is effectively dead code (resolveUniqueSlug prevents collisions inside the same transaction) and overly broad (catches any unique constraint, not just slug). - **[S2]** `packages/server/src/velocity/velocity.service.ts:210-236` — `recordSave` does two independent non-transactional writes. If save_event succeeds but daily_snapshot fails, velocity data is temporarily inconsistent until next save. Documented as best-effort, so this is intentional, but worth noting. *(Logic, Concurrency, Error Handling)*
- **[S3]** `packages/server/src/projects/projects.service.ts:22` — Dead re-export of `setVelocityService`/`resetVelocityService`. Projects service never calls velocity methods. Remove these re-exports; test files should import from `velocity.injectable` or `chapters.service`. *(Contract)*
- **[S4]** `packages/server/src/projects/projects.service.ts:279-282` — `getDashboard` manually builds a `statusLabelMap` from `ChapterStatusRepo.list()`, duplicating the existing `getStatusLabelMap()` repo method that other functions already use. *(Contract)*
- **[S5]** `packages/server/src/db/connection.ts:13-20` — `setDb()` skips `PRAGMA journal_mode = WAL`, while `initDb()` sets it. Tests run without WAL mode. Low impact for in-memory test DBs. *(Logic)*
- **[S6]** `packages/server/src/db/connection.ts:14-16` — If `db.destroy()` throws in `setDb()`, the function exits without updating `db`, leaving the module pointing at the broken old instance. Wrap in try/catch and proceed regardless. *(Error Handling)*
- **[S7]** `packages/server/src/projects/projects.repository.ts:118-141` — `resolveUniqueSlug` iterates up to 1000 sequential queries. A single SQL query (`SELECT slug FROM projects WHERE slug LIKE ?`) could replace the loop. Low practical impact for single-user app. *(Security)*
- **[S8]** `packages/server/src/settings/settings.routes.ts:20` — Zod validation (`UpdateSettingsSchema.safeParse`) in settings route instead of service. All other domains do validation in the service layer. *(Plan Alignment)*
- **[S9]** `packages/server/src/chapters/chapters.repository.ts:173` — `ChapterRepo.insert` returns `void` while `ProjectRepo.insert` returns `ProjectRow`. Creates API asymmetry and requires an extra `findById` query on every chapter creation. *(Plan Alignment)*
