## Important Issues

### [I1] Tests use invalid chapter status "draft"
- **File:** `packages/server/src/__tests__/chapters.service.test.ts:46`, `packages/server/src/__tests__/chapters.repository.test.ts:48`
- **Bug:** Test helpers insert chapters with `status: "draft"`, but the only valid statuses (seeded by migration 003) are: `outline`, `rough_draft`, `revised`, `edited`, `final`. No FK constraint exists, so SQLite accepts the invalid value. The `getStatusLabel()` fallback (`?? status`) silently returns `"draft"` as the label, so tests pass but exercise a code path that cannot occur in production.
- **Impact:** Tests give false confidence that status label resolution works. Any future change to the fallback behavior (e.g., returning null or throwing on unknown status) would cause unexpected test failures. Tests are not validating the real label-lookup logic.
- **Suggested fix:** Change `"draft"` to `"outline"` in both test helpers, and assert the resolved label is `"Outline"` (not just truthy).
- **Confidence:** High
- **Found by:** Logic & Correctness, Contract & Integration

### [I2] Velocity daily average inflated for sparse data
- **File:** `packages/server/src/velocity/velocity.service.ts:306-337`
- **Bug:** When no baseline snapshot exists before the 30-day window, `baselineDate` falls back to `firstSnapshot.date` and `daysCovered` becomes the span of available data (e.g., 2 days) instead of 30. The `Math.min(30, ...)` cap prevents going above 30 but doesn't prevent the undercount. Example: writing 1000 words on day 1 and 1000 on day 2 yields a 500/day average instead of ~67/day over 30 days.
- **Impact:** Inflated daily average feeds into `calculateProjection`, producing overly optimistic projected completion dates for new projects with sparse writing history.
- **Suggested fix:** When there is no baseline snapshot before the 30-day window, divide by 30 (or the number of days since the first snapshot if less than 30) rather than just the span of available data.
- **Confidence:** High
- **Found by:** Logic & Correctness

### [I3] Repository softDelete functions lack `whereNull("deleted_at")` guard
- **File:** `packages/server/src/chapters/chapters.repository.ts:211`, `packages/server/src/projects/projects.repository.ts:115`
- **Bug:** `ChapterRepo.softDelete` and `ProjectRepo.softDelete` run `update({ deleted_at: now })` without filtering `whereNull("deleted_at")`. A second delete call overwrites the original `deleted_at` timestamp. The service layer guards against this (checks exist via `findByIdRaw`/`findBySlug` which filter `deleted_at IS NULL`), but the repository methods themselves are unguarded reusable building blocks.
- **Impact:** In a multi-tab scenario, double-deleting a project or chapter resets the `deleted_at` timestamp, extending the item's time in trash beyond the original 30-day purge window.
- **Suggested fix:** Add `.whereNull("deleted_at")` to both `softDelete` functions, consistent with how `ChapterRepo.update` and `ProjectRepo.update` already filter.
- **Confidence:** Medium
- **Found by:** Error Handling & Edge Cases, Concurrency & State
