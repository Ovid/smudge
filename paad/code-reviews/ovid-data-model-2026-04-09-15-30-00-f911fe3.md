## Suggestions

1. **`projects/projects.service.ts:12`** â€” Cross-service import of `stripCorruptFlag` from `chapters/chapters.service`. This is a pure utility with no IO; move it to `chapters/chapters.types.ts` or a utility file to avoid lateral service coupling. (Confidence: 80, Found by: Contract & Integration)

2. **`projects/projects.service.ts:51,128`** â€” Service returns internal `ProjectRow` instead of shared `Project` type. Shapes are identical today so no runtime divergence, but any future DB-only column added to `ProjectRow` would leak to the API without a compile error. Add a `toProject()` mapping function. (Confidence: 75, Found by: Contract & Integration)

3. **`chapters/chapters.service.ts`** â€” Same pattern: returns `ChapterWithLabel` (internal type) instead of shared `Chapter`. Same risk as above. (Confidence: 75, Found by: Contract & Integration)

4. **`velocity/velocity.injectable.ts:8`** â€” Module-level mutable `velocityServiceOverride` can leak between test files if a test errors before calling `resetVelocityService()`. Audit test files for missing `afterEach` cleanup. (Confidence: 70, Found by: Concurrency & State)

5. **`chapter-statuses/chapter-statuses.repository.ts`** â€” Two strategies for status label lookup: `getStatusLabel` (per-call DB hit used in chapters.service) vs `getStatusLabelMap` (batch used in projects.service). Consolidating on `getStatusLabelMap` would remove an N+1 pattern and prevent the two paths from diverging. (Confidence: 70, Found by: Contract & Integration)

6. **`settings/settings.service.ts:22-28`** â€” `UpdateSettingsSchema` accepts any string key; unknown keys are correctly rejected at the service layer via the `SETTING_VALIDATORS` map, but the Zod schema doesn't enforce allowed keys. Adding a `z.enum` for valid key names would provide defense-in-depth at the schema layer. (Confidence: 65, Found by: Security)

7. **`projects/projects.service.ts:136`** â€” `findBySlug` runs outside the transaction, `update` runs inside. If the project is soft-deleted between the two calls, the error surfaces as a 500 instead of a 404. Extremely low risk in a single-user app. (Confidence: 65, Found by: Contract & Integration, Concurrency & State)

8. **`velocity/velocity.service.ts:291-297`** â€” Error-path fallback when `getPreWindowBaselines` fails: uses in-window event word_count as baseline, which zeroes first-session net_words instead of computing correctly. Only triggered on DB failure. (Confidence: 65, Found by: Error Handling)

9. **`shared/src/schemas.ts:39-44`** â€” `TipTapDocSchema` uses `.passthrough()` with `z.record(z.unknown())`, accepting unbounded nesting depth. Mitigated by the 5MB `express.json` body limit, but no explicit depth guard. (Confidence: 65, Found by: Security)
