## Executive Summary

The document export feature (Word .docx and EPUB) is functionally complete — all planned files, integration wiring, tests, and e2e coverage are in place. The most significant finding is a systematic deviation from the design document's heading level shift specification: the design says all formats should shift H3→H1, but the implementation preserves heading hierarchy instead. This affects all five export formats and is reflected consistently in both code and tests, suggesting a deliberate (but undocumented) design change. There is also one important error-handling gap where failed chapter rendering silently produces empty content in exports.

None found.

## Important Issues

### [I1] Heading level shift not implemented in HTML/Markdown/EPUB (design deviation)
- **File:** `packages/server/src/export/export.renderers.ts:30-38`
- **Bug:** The design document (§Heading Level Shift) states: "All export formats shift heading levels: TipTap H3→H1, H4→H2, H5→H3." Plan Task 2 describes adding a `shiftHeadingLevels()` helper to `chapterContentToHtml()`. The function was never implemented — `chapterContentToHtml()` returns raw `generateHTML()` output with H3/H4/H5 intact. Tests at lines 114-148 (HTML), 204-234 (Markdown), and 843-877 (EPUB) explicitly assert headings are preserved, not shifted.
- **Impact:** Exported HTML, Markdown, and EPUB files use H3/H4/H5 for body headings instead of H1/H2/H3 as the design specifies. However, the current behavior arguably produces a more correct heading hierarchy for HTML/Markdown (H1=project title, H2=chapter title, H3+=body headings). For EPUB, where each chapter is a standalone XHTML file, shifting would be more appropriate.
- **Suggested fix:** Either (a) implement `shiftHeadingLevels()` selectively for EPUB only (where chapters are standalone documents) and update the design doc to reflect no shift in HTML/Markdown, or (b) implement the shift across all formats as designed and fix the HTML/Markdown renderers to use shifted headings within chapter sections. Either way, update the design document to match the chosen behavior.
- **Confidence:** High
- **Found by:** Logic & Correctness, Plan Alignment

### [I2] Docx heading map off by one level from design
- **File:** `packages/server/src/export/docx.renderer.ts:17-25`
- **Bug:** The code maps `H3→HeadingLevel.HEADING_2`, `H4→HEADING_3`, `H5→HEADING_4`. The design document and plan both specify `H3→Heading 1`, `H4→Heading 2`, `H5→Heading 3`. Chapter titles use `HEADING_1` (line 330), so the implementer shifted body headings down one level to create a parent-child hierarchy. The tests at line 409 confirm this mapping was intentional.
- **Impact:** Word documents have body headings starting at Heading 2 (under chapter title Heading 1) instead of Heading 1 as designed. This is arguably better document structure but contradicts the spec.
- **Suggested fix:** If the current hierarchy is preferred (Heading 1 for chapter titles, Heading 2+ for body), update the design document. If the design's flat structure is preferred (both chapter titles and top body headings at Heading 1), update the code to match.
- **Confidence:** High
- **Found by:** Logic & Correctness, Plan Alignment

### [I3] EPUB CSS targets wrong heading levels
- **File:** `packages/server/src/export/epub.renderer.ts:9-31`
- **Bug:** The embedded CSS defines styles for `h1`, `h2`, `h3`. Since heading shift is not implemented (I1), actual chapter HTML contains `<h3>`, `<h4>`, `<h5>`. Only `h3` gets custom styling — `h4` and `h5` headings get unstyled browser defaults in e-readers.
- **Impact:** Sub-headings (H4, H5) in EPUB exports lack the intended font sizing and margin styling, producing inconsistent visual appearance.
- **Suggested fix:** Either implement heading shift for EPUB (resolves this automatically), or add CSS rules for `h4` and `h5` to match current unshifted output.
- **Confidence:** High
- **Found by:** Logic & Correctness

### [I4] chapterContentToHtml silently returns empty string on error — data loss in exports
- **File:** `packages/server/src/export/export.renderers.ts:30-37`
- **Bug:** When `generateHTML()` throws (e.g., corrupted TipTap JSON), the function catches the error, logs a warning, and returns `""`. The export proceeds with a chapter title but no body content. The user receives a complete-looking export with silently missing chapter text and no indication anything is wrong.
- **Impact:** Users could download exports with missing chapters and not notice, especially in large books. This undermines the "core trust promise" described in CLAUDE.md.
- **Suggested fix:** Either propagate the error (so the route returns 500 with context about which chapter failed), or insert a visible placeholder like `[Error: chapter content could not be exported]` in the output. At minimum, include the chapter title in the log message for debugging.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases
