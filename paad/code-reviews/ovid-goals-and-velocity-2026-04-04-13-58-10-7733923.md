## Important Issues

### [I1] `hasFetchedInitial` ref never resets on project switch
- **File:** `packages/client/src/pages/EditorPage.tsx:129-140`
- **Bug:** `hasFetchedInitial` is a `useRef(false)` set to `true` on first velocity fetch but never reset when `slug` changes. When navigating to a different project, the ref remains `true`, so the initial velocity fetch is skipped. The new project's velocity data won't load until the first save triggers a throttled fetch (up to 60 seconds later).
- **Impact:** Users switching between projects see stale or missing velocity data in the status bar for up to a minute.
- **Suggested fix:** Add an effect that resets `hasFetchedInitial.current = false` and `lastVelocityFetch.current = 0` when `slug` changes.
- **Confidence:** High
- **Found by:** Concurrency & State

### [I2] `lastSession` not cleared on project switch
- **File:** `packages/client/src/pages/EditorPage.tsx:100,132-157`
- **Bug:** `lastSession` state is never cleared when `slug` changes. Combined with I1, when the user navigates to a different project, the footer continues displaying the previous project's session data ("Last session: 85 min, +1,200 words") until new data loads.
- **Impact:** Misleading session statistics shown for the wrong project.
- **Suggested fix:** Clear `lastSession` to `null` when `slug` changes (in the same effect as I1's fix).
- **Confidence:** High
- **Found by:** Concurrency & State

### [I3] Deleting/restoring a chapter does not trigger `upsertDailySnapshot`
- **File:** `packages/server/src/routes/chapters.ts:138-161` (delete), `:163-243` (restore)
- **Bug:** When a chapter is soft-deleted or restored, neither handler calls `upsertDailySnapshot`. The daily snapshot's `total_word_count` sums only non-deleted chapters, so after deletion the snapshot remains inflated (includes deleted chapter's words) until the next save on another chapter triggers a new snapshot. Restoring similarly leaves the snapshot understated.
- **Impact:** Daily snapshot data can be stale for hours or days after a delete/restore, affecting the 30-day daily average, "words today" metric, and burndown chart accuracy.
- **Suggested fix:** Call `upsertDailySnapshot(db, chapter.project_id)` after the transaction in both `DELETE /chapters/:id` and `POST /chapters/:id/restore`.
- **Confidence:** High
- **Found by:** Logic & Correctness (Server)

### [I4] `calculateWordsToday` returns entire project total on first day
- **File:** `packages/shared/src/schemas.ts:73-88`
- **Bug:** When no prior-day snapshot exists (first day of velocity tracking), `lastPrior` is undefined and the function returns `currentTotal` (the entire project word count). If a project already had 50,000 words before velocity tracking began, "words today" shows +50,000.
- **Impact:** Misleading "words today" metric on the first day of tracking for projects with existing content. The migration seeds a baseline snapshot, which mitigates this for projects that existed before the migration, but new projects created and immediately written to would show accurate numbers (starting from 0).
- **Suggested fix:** When there's no prior-day snapshot but there IS a same-day snapshot from earlier, use that as baseline. Alternatively, return 0 when there's no prior-day snapshot at all (conservative approach).
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [I5] `ChapterTargetPopover` silently swallows save failures
- **File:** `packages/client/src/components/ChapterTargetPopover.tsx:34-41`
- **Bug:** `handleSave` catches all errors with an empty catch block. The user sees the updated draft value in the UI but receives no indication that the save failed. The popover may close while the save is still pending or has failed. On next page load, the old value reappears.
- **Impact:** Silent data loss for per-chapter word count targets. The user believes the target was set but it was never persisted.
- **Suggested fix:** Add error state and show an error indicator. Revert `draft` to previous `targetWordCount` on failure so the displayed value matches server state.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [I6] Silent catch on baseline query inflates all session `net_words`
- **File:** `packages/server/src/routes/velocity.ts:243-261`
- **Bug:** The try/catch around the pre-window baselines query silently defaults to empty baselines (`preWindowBaselines = {}`) on failure. If this query fails, every chapter's first event in the 30-day window is treated as baseline 0, attributing the chapter's entire word count to the first session.
- **Impact:** All session `net_words` would be massively inflated (showing thousands of words written when only dozens were), and this would persist silently with no indication to the user or server logs.
- **Suggested fix:** Add `console.error` logging in the catch block so failures are visible. Consider a simpler query that's less likely to fail (e.g., `MAX(saved_at)` per chapter).
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases, Security

## Suggestions

- **[S1]** `VelocityView.tsx:16-37` — `computeDailyNetWords` drops the first day from the bar chart when 2+ snapshots exist. The loop starts at `i=1`, using day 0 as baseline only. Consider including day 0 as `{ date, net_words: total_word_count }` for consistency. *(Logic & Correctness — Client)*
- **[S2]** `velocity.ts:346-349` — `chapter_names` map excludes soft-deleted chapters, but sessions reference all chapters including deleted ones. Client handles this gracefully with "Unknown chapter" fallback, but including deleted chapter names would be more informative. *(Logic & Correctness — Server)*
- **[S3]** `SummaryStrip.tsx:58` vs `VelocityView.tsx:95` — Inconsistent date parsing: SummaryStrip uses `"T00:00:00"` (local), VelocityView uses `"T00:00:00Z"` (UTC). Could cause projected date and days-remaining to disagree by one day for users in negative UTC offsets. Use a consistent approach. *(Logic & Correctness — Client)*
- **[S4]** `RecentSessions.tsx:12-15` — "Today"/"Yesterday" labels use browser-local time rather than the server's configured timezone. Pass `data.today` through and use it for the comparison. *(Contract & Integration)*
- **[S5]** `ProjectSettingsDialog.tsx:61-76` — No concurrency guard on `saveField`. Overlapping saves can race, and the error-revert path uses potentially stale prop values. Consider disabling inputs while a save is in-flight or tracking "last known good" values independently. *(Concurrency & State, Logic — Client, Error Handling)*
- **[S6]** `velocity.ts:228-230` — 30-day window boundary is constructed at UTC midnight of the timezone-adjusted date, not at midnight in the user's timezone. This can shift the boundary by up to ~24 hours. *(Logic & Correctness — Server, Contract & Integration)*
- **[S7]** `RecentSessions.tsx:52` — Array index used as React key for session list items. Sessions lack unique IDs, but `key={session.start}` would be more stable. *(Logic & Correctness — Client)*
- **[S8]** `velocity.ts:296` + `velocityHelpers.ts:54` + `projects.ts:421` — Project total word count computed independently in three places. Consider extracting a shared helper. *(Contract & Integration)*
- **[S9]** `schemas.ts:65` — Settings key not constrained in Zod schema to known values. Server validates at runtime, but constraining to `z.enum(["timezone"])` would make the contract explicit. *(Security)*
- **[S10]** `SettingsDialog.tsx:62-68` — Timezone list recomputed via `Intl.supportedValuesOf("timeZone")` on every render when dialog is open. Move outside component or `useMemo`. *(Logic & Correctness — Client)*
- **[S11]** `VelocityView.tsx:16-37` vs `schemas.ts:73-88` — Duplicated snapshot-diffing logic across client `computeDailyNetWords` and shared `calculateWordsToday`. Consider consolidating. *(Contract & Integration)*
- **[S12]** `schemas.ts:39-44` — No size/depth limit on TipTap content payload in Zod schema. Express body-parser limits provide some defense, but an explicit schema constraint would be more robust. *(Security)*
- **[S13]** `ProjectSettingsDialog.tsx:87-89` — Deadline `onChange` fires `saveField` on each change. On browsers where date inputs fire onChange for partial typed input, this triggers repeated 400 errors from the server's format validation. *(Error Handling, Concurrency)*
- **[S14]** `ChapterTargetPopover.tsx:19` — Draft state not synced when `targetWordCount` prop changes while popover is open. Narrow window in a single-user app but could confuse. *(Logic & Correctness — Client)*
