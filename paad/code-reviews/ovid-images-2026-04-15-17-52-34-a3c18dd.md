This branch implements Phase 4a (Reference Panel & Images) — a substantial feature adding image upload, storage, reference tracking, gallery UI, and export embedding. ## Critical Issues

### [C1] Orphaned file on DB insert failure
- **File:** `packages/server/src/images/images.service.ts:83-94`
- **Bug:** `writeFile` writes the image to disk (line 84), then `imagesRepo.insert` inserts the DB row (line 87). If the DB insert throws (constraint violation, disk full on WAL, etc.), the file remains on disk permanently with no DB record. No try/catch or compensating `unlink` exists. The purge job only cleans up by `project_id`, so these orphaned files have no recovery path.
- **Impact:** Silent disk leak. Over time, failed uploads accumulate unreferenced binary files that can never be cleaned up without manual intervention.
- **Suggested fix:** Wrap the DB insert in a try/catch that unlinks the file on failure, or write to a temp path first and atomically rename after successful insert.
- **Confidence:** High
- **Found by:** Logic & Correctness (Server), Error Handling, Concurrency & State

### [C2] `deleteChapter` does not decrement image `reference_count`
- **File:** `packages/server/src/chapters/chapters.service.ts:144-164`
- **Bug:** `deleteChapter` soft-deletes the chapter without extracting image IDs from chapter content or calling `incrementReferenceCount` with a negative delta. Compare with `updateChapter` (lines 105-121) which correctly diffs old/new image IDs and adjusts counts. After soft-delete, the referenced images retain an inflated `reference_count`, causing the gallery to display them as "in use" and blocking deletion.
- **Impact:** Users cannot delete images that are only referenced by deleted chapters. The gallery permanently shows them as in-use with no way to unblock short of a live-check triggered by another action.
- **Suggested fix:** In `deleteChapter`, after the soft-delete, walk the chapter's content via `extractImageIds`, and call `incrementReferenceCount(db, id, -1)` for each. This is the symmetric operation to `updateChapter`.
- **Confidence:** High
- **Found by:** Logic & Correctness (Server)

### [C3] WebP images silently dropped in DOCX export
- **File:** `packages/server/src/export/docx.renderer.ts:205-210`
- **Bug:** `MIME_TO_DOCX_TYPE` maps jpeg/png/gif/bmp but not `image/webp`. The upload route accepts webp (`ALLOWED_MIMES` in `images.service.ts:17`). When a webp image is encountered during DOCX export, `docxType` is `undefined`, a warning is logged, and `return []` silently omits the image from the document.
- **Impact:** WebP is the most common modern upload format. Users who upload webp images will silently lose them in DOCX exports with no user-visible error — only a server-side log warning.
- **Suggested fix:** Add `"image/webp": "png"` to `MIME_TO_DOCX_TYPE` with in-memory conversion via `sharp`, or add webp support directly if the `docx` library version supports it. Also remove the unreachable `"image/bmp": "bmp"` entry (bmp is not in `ALLOWED_MIMES`).
- **Confidence:** High
- **Found by:** Contract & Integration

## Important Issues

### [I1] `liveCheckImageReferences` writes DB on every GET (side effect on read path)
- **File:** `packages/server/src/images/images.references.ts:77`
- **Bug:** Every call to `getImageReferences` (the GET references endpoint) and `deleteImage` triggers `UPDATE images SET reference_count = ...`. This means a GET to `/images/:id/references` mutates the database. It also creates competing writers: `updateChapter` increments/decrements `reference_count` via delta operations (`chapters.service.ts:113-118`), while `liveCheckImageReferences` does an absolute overwrite. Under concurrent saves, the two paths race to set the same column.
- **Impact:** `reference_count` can drift — images could get stuck at 0 (allowing deletion of in-use images) or at an inflated count (blocking deletion of unused images).
- **Suggested fix:** Split into a pure `scanImageReferences` (no DB write, used by GET) and a `reconcileReferenceCount` (write, used only by delete). Alternatively, remove the `reference_count` column entirely and always compute live on demand.
- **Confidence:** High
- **Found by:** Error Handling, Contract & Integration, Concurrency & State

### [I2] Caption regex matches all images of same MIME type in `resolveImagesInHtml`
- **File:** `packages/server/src/export/image-resolver.ts:90-93`
- **Bug:** The figcaption replacement regex is `(<img[^>]*src="data:{mimeType}[^"]*"[^>]*>)` where `mimeType` is e.g. `image/png`. If a chapter contains two PNG images, both match. The first pass wraps one in `<figure>`, the second pass's regex also matches the first image's already-resolved `<img>` tag (since both start with `data:image/png`), producing incorrect caption attribution and potential double-wrapping.
- **Impact:** Incorrect captions in exported HTML/Markdown documents when a chapter contains multiple images of the same format.
- **Suggested fix:** Embed a `data-smudge-id` attribute during the `src` replacement step and match on that for figcaption insertion, or build the figcaption in a single templated pass per unique image ID.
- **Confidence:** High
- **Found by:** Logic & Correctness (Server), Error Handling

### [I3] `handleDelete` in ImageGallery has no error handling
- **File:** `packages/client/src/components/ImageGallery.tsx:159-170`
- **Bug:** `handleDelete` is `async` and calls `api.images.delete`, but there is no try/catch. If the server returns a non-409 error (500, network failure), `api.images.delete` throws an `ApiRequestError` that becomes an unhandled promise rejection. The component stays in `confirmingDelete=true` state with no feedback.
- **Impact:** Network errors during delete show no user feedback; the UI freezes on the confirmation dialog.
- **Suggested fix:** Wrap in try/catch, announce the error, and reset `confirmingDelete` state.
- **Confidence:** High
- **Found by:** Error Handling

### [I4] `api.images.delete` calls `res.json()` before checking `res.ok`
- **File:** `packages/client/src/api/client.ts:220-229`
- **Bug:** `const body = await res.json()` is called unconditionally before any `res.ok` check. If the server returns a non-JSON error body (nginx 502 HTML page, plain-text 500), `res.json()` throws a `SyntaxError` that bypasses all error handling. Every other endpoint uses the `apiFetch` wrapper which handles this safely.
- **Impact:** Combined with [I3], a non-JSON error response causes a raw `SyntaxError` unhandled rejection instead of a meaningful `ApiRequestError`.
- **Suggested fix:** Use `await res.json().catch(() => null)` before status checks, matching the pattern in `apiFetch` and `api.images.upload`.
- **Confidence:** High
- **Found by:** Error Handling

### [I5] Duplicate `ImageRow` definition — server-local type vs shared type
- **File:** `packages/server/src/images/images.types.ts` and `packages/shared/src/types.ts:61-73`
- **Bug:** Both files define an `ImageRow` interface with identical fields. The server imports its local copy; the client imports from `@smudge/shared`. If the two definitions ever diverge (e.g., a new column added to one but not the other), the type system will not catch the mismatch because the server never imports the shared type.
- **Impact:** Structural divergence risk. The shared type is the API contract, but the server uses a shadow copy that can silently drift.
- **Suggested fix:** Delete `ImageRow` from `images.types.ts` and import from `@smudge/shared`. Keep server-only types (`CreateImageRow`, `UpdateImageData`, `ImageWithUsage`) local.
- **Confidence:** High
- **Found by:** Contract & Integration

### [I6] Purge uses pre-filter project ID set for disk cleanup
- **File:** `packages/server/src/db/purge.ts:27-52`
- **Bug:** `purgedProjectIds` is built from `projectsToPurge` (all projects with expired `deleted_at`). But the DB `DELETE` at lines 38-41 excludes projects with remaining live chapters. So `purgedProjectIds` can contain projects that were not actually deleted from the DB. Their image directories on disk are then removed at lines 52-58 anyway.
- **Impact:** Edge case: a project soft-deleted while its chapters were independently restored would have its image files purged even though DB rows and live chapters survive.
- **Suggested fix:** Compute `purgedProjectIds` from the actual set of deleted IDs (post-filter), not the pre-filter list.
- **Confidence:** Medium
- **Found by:** Error Handling, Concurrency & State

### [I7] `handleInsert` uses unsaved `formState.alt_text` instead of persisted value
- **File:** `packages/client/src/components/ImageGallery.tsx:155`
- **Bug:** `handleInsert` calls `onInsertImage(url, formState.alt_text)` — the local draft value, not the persisted server value. If the user edits alt text and clicks "Insert at cursor" without saving, the editor gets the draft alt text but the DB holds the old value.
- **Impact:** The alt text in the TipTap document diverges from the alt text in the `images` table. Export and accessibility features that read from the DB will use the stale value.
- **Suggested fix:** Auto-save before inserting when the form has unsaved changes, or require save before allowing insert.
- **Confidence:** Medium
- **Found by:** Logic & Correctness (Client)

### [I8] Module-level `imageUploadHandler` is not instance-safe
- **File:** `packages/client/src/components/Editor.tsx:30, 193-212`
- **Bug:** `imageUploadHandler` is a module-level `let` variable shared across all Editor instances. The `useEffect` sets it to the current instance's handler and resets it on cleanup. If two Editor instances exist simultaneously (tests, future multi-pane), the last mounted instance wins; the first instance's paste handler uploads to the wrong project.
- **Impact:** Safe for the current single-editor layout, but a latent bug for tests or any future refactor that mounts two editors.
- **Suggested fix:** Move to a `useRef` inside the component and pass through a closure or React context.
- **Confidence:** High
- **Found by:** Logic & Correctness (Client), Concurrency & State

## Suggestions

- **[S1]** `getDataDir()` and `mimeToExt`/`MIME_TO_EXT` are duplicated across `images.service.ts`, `image-resolver.ts`, and `epub.renderer.ts`. Extract to a shared server-side module to prevent the kind of divergence that caused [C3]. (`images.service.ts:19-24,52-54`, `image-resolver.ts:20-31`, `epub.renderer.ts:20-28`)
- **[S2]** `api.projects.export` config type is missing `epub_cover_image_id?: string`. Works at runtime (JSON.stringify sends the field), but the type annotation gap is a maintenance hazard. (`api/client.ts:110-114`)
- **[S3]** `isUsed` (from stale `reference_count`) and `references` (from async live-check) can disagree in the ImageGallery detail view. When `isUsed=true` but `references` hasn't loaded, the delete-blocked message shows an empty chapter list. (`ImageGallery.tsx:232,362-366`)
- **[S4]** MIME type is trusted from client `Content-Type` header without magic-byte verification. Defense-in-depth gap — the `file-type` package (MIT) can verify in ~3 lines. Low risk in single-user app. (`images.service.ts:61-65`)
