## Important Issues

### [I1] `restoreChapter` does not re-increment image reference counts
- **File:** `packages/server/src/chapters/chapters.service.ts:183-275`
- **Bug:** `deleteChapter` decrements `reference_count` for all images in the deleted chapter's content (lines 155-169). `restoreChapter` performs no corresponding increment. After a soft-delete + restore cycle, image reference counts are permanently understated.
- **Impact:** Gallery shows "unused" badge on images that are actively referenced by the restored chapter. The `liveCheckImageReferences` self-heals on delete attempts, but the gallery read path uses the stale DB count — users see incorrect "unused" indicators indefinitely.
- **Suggested fix:** After the restore transaction commits, add a best-effort block (mirroring `deleteChapter`) that parses the restored chapter's content, extracts image IDs via `extractImageIds`, and calls `incrementReferenceCount(db, id, 1)` for each.
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling & Edge Cases, Contract & Integration

### [I2] `handleInsert` proceeds even when pre-save fails
- **File:** `packages/client/src/components/ImageGallery.tsx:162-177`
- **Bug:** The `try/catch` block that auto-saves metadata before inserting catches the error and sets `saveStatus("idle")`, but execution falls through to `onInsertImage()` unconditionally. The image is inserted into the editor with alt text the server doesn't know about.
- **Impact:** DB and editor alt text diverge. Exports that read `alt_text` from the DB will get stale data. The comment says "Auto-save pending metadata changes before inserting so the DB stays in sync" — the code violates this intent on failure.
- **Suggested fix:** Add `return;` inside the `catch` block to prevent the insert from proceeding on save failure.
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling & Edge Cases

### [I3] Silent delete failure on 409 IMAGE_IN_USE — no user feedback
- **File:** `packages/client/src/components/ImageGallery.tsx:183-196`
- **Bug:** When the server returns a 409 (image in use), `api.images.delete()` returns `{ error: { code: "IMAGE_IN_USE", chapters: [...] } }` instead of throwing. The guard `if ("error" in result) { return; }` exits silently — no `announce()` call, no `setConfirmingDelete(false)`, no UI state change.
- **Impact:** The user clicks the red "Delete" confirmation button, nothing visible happens, and they receive no explanation of why the delete was blocked. The error response contains the blocking chapter names but they are never surfaced.
- **Suggested fix:** Call `announce(S.deleteBlocked(...))` with the chapter titles from the error response and reset `setConfirmingDelete(false)` before returning.
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling & Edge Cases

### [I4] Reference count update runs outside the chapter-save transaction
- **File:** `packages/server/src/chapters/chapters.service.ts:104-122`
- **Bug:** The chapter content update runs inside `store.transaction()` (lines 84-89). The `incrementReferenceCount` calls for added/removed images run after the transaction commits, in a separate best-effort block (lines 104-122). If the process crashes or the loop throws partway through, reference counts drift from reality.
- **Impact:** The "unused" badge in the gallery can be wrong for an unbounded period. The `liveCheckImageReferences` corrects this only when a delete is attempted — between save and delete, any code trusting `reference_count` displays incorrect state.
- **Suggested fix:** Move `incrementReferenceCount` calls inside the same transaction as the content update.
- **Confidence:** High
- **Found by:** Logic & Correctness, Concurrency & State

### [I5] `scanImageReferences` uses substring match instead of AST walk
- **File:** `packages/server/src/images/images.references.ts:66-74`
- **Bug:** `scanImageReferences` uses `ch.content.includes("/api/images/${imageId}")` — a raw string search on the JSON-serialized content. The same file provides `extractImageIds()` which does a proper JSON parse + AST walk checking only `type:"image"` nodes. The substring approach can false-positive on any JSON value containing the URL string (e.g., in a caption, link href, or other metadata).
- **Impact:** `liveCheckImageReferences` (delete guard) and `getImageReferences` (gallery where-used list) both delegate to `scanImageReferences`. A false positive prevents image deletion even when the image is not actually embedded in any chapter's content.
- **Suggested fix:** Replace the `includes()` scan with `extractImageIds(JSON.parse(ch.content))` to match the approach used in the chapter-save reference diffing.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases, Contract & Integration

### [I6] Source and license fields not used in export output
- **File:** `packages/server/src/export/image-resolver.ts:12-13`, `docx.renderer.ts:370-377`, `epub.renderer.ts`
- **Bug:** `ResolvedImage` carries `source` and `license` fields populated from the DB. The design document specifies these should be appended to captions in exports (e.g., "A quiet street at dusk (Photo: Jane Doe, CC BY 4.0)"). No renderer uses them — captions are rendered without source/license in HTML, DOCX, and EPUB.
- **Impact:** Writers who populate source and license metadata expecting it to appear in exports will get incomplete captions.
- **Suggested fix:** In each renderer, append source and license to the caption string when present, following the format in the design document.
- **Confidence:** High
- **Found by:** Plan Alignment

## Suggestions

- **[S1]** `listImages` returns 200 with empty array for non-existent project IDs; `uploadImage` returns 404. Add project existence check for consistency. (`images.service.ts:87`, `images.routes.ts:66`) — Logic, ErrorHandling, Contract
- **[S2]** `mimeToExt()` returns `"bin"` fallback, making the `if (!ext) return null` guard in `serveImage()` dead code. Either return `null` for unknown types or remove the guard. (`images.paths.ts:16`, `images.service.ts:103`) — ErrorHandling, Contract, Security
- **[S3]** UUID regex in `image-resolver.ts:45` and `docx.renderer.ts:314` uses loose `[0-9a-f-]{36}` instead of the strict pattern in `images.references.ts:17`. Standardize on a shared constant. — Logic, ErrorHandling
- **[S4]** DOCX export replaces webp images with text placeholders. Server accepts webp uploads, creating an expectation gap. Consider converting webp to PNG for DOCX or surfacing a UI warning. (`docx.renderer.ts:205-209`) — Contract
- **[S5]** `resolveImagesForEpub` duplicates the regex-scan-and-replace logic from `resolveImagesInHtml`. Extract a shared `resolveImageUrls(html, strategy)` function. (`epub.renderer.ts:17-48` vs `image-resolver.ts:40-83`) — Contract
- **[S6]** Module-level `activeEditorId` is set on both mount effect and `onFocus`, creating a potential race during chapter switch. Consider relying solely on `onFocus`. (`Editor.tsx:33-37, 186, 207`) — Logic, Concurrency
- **[S7]** TOCTOU: `deleteImage` reference check and delete are not atomic. Concurrent chapter save could add a reference between scan and delete. Mitigated by SQLite write serialization but not guaranteed. (`images.service.ts:135-159`) — Concurrency
- **[S8]** TOCTOU: `uploadImage` writes file to disk before DB insert. Process crash between them orphans the file with no reconciliation. (`images.service.ts:64-82`) — Concurrency
- **[S9]** Purge filesystem cleanup can race with concurrent upload to the same project directory. (`purge.ts:62-70`) — Concurrency
- **[S10]** MIME type validated by client-claimed `Content-Type` only; no magic-byte inspection of file buffer. Mitigated by `X-Content-Type-Options: nosniff` header. (`images.service.ts:42`) — Security
- **[S11]** `file.originalname` stored in DB with no length limit or character sanitization. React escapes on render but the raw value is a latent hazard. (`images.service.ts:73`) — Security, ErrorHandling
- **[S12]** Zero-byte file upload passes all validation, producing a broken image in the gallery and exports. Add `file.size < 1` check. (`images.service.ts:48`) — ErrorHandling
- **[S13]** `getImage()` exported from service but never called by any route. Dead code. (`images.service.ts:92`) — Contract
- **[S14]** `liveCheckImageReferences` writes corrected `reference_count` outside the delete transaction, making the correction non-atomic with the subsequent delete. (`images.references.ts:83-94`) — Concurrency
