## Important Issues

### [I1] EPUB exports missing image captions, source, and license
- **File:** `packages/server/src/export/epub.renderer.ts:17-49`
- **Bug:** `resolveImagesForEpub()` replaces `/api/images/{uuid}` URLs with `file://` paths but does NOT wrap images in `<figure>`/`<figcaption>` with caption text. The HTML renderer (`image-resolver.ts:83-98`) and DOCX renderer (`docx.renderer.ts:369-379`) both render captions. EPUB is the only format missing this. Root cause: duplicated image resolution logic between `epub.renderer.ts` and `image-resolver.ts` that has already diverged.
- **Impact:** Caption and attribution data (source, license) is silently lost in EPUB exports, violating the design document requirement.
- **Suggested fix:** After URL resolution in `resolveImagesForEpub`, add a caption/figure pass using `buildCaptionText()` from `image-resolver.ts`, or refactor to share the resolution logic.
- **Confidence:** High
- **Found by:** Contract & Integration

### [I2] WebP images silently dropped in DOCX export
- **File:** `packages/server/src/export/docx.renderer.ts:205-209`
- **Bug:** `MIME_TO_DOCX_TYPE` only maps jpeg, png, and gif. WebP is an allowed upload type (`images.paths.ts:6` includes `image/webp`), but WebP images in DOCX exports are silently replaced with italic text placeholders `[Image: ...]`. Only a server-side log warning is produced; the user gets no indication.
- **Impact:** Users who upload WebP images and export to DOCX will lose their images with no warning.
- **Suggested fix:** Either convert WebP to PNG at export time, or surface a warning to the user in the export dialog when WebP images are present and DOCX is selected.
- **Confidence:** High
- **Found by:** Logic & Correctness (Server), Error Handling & Edge Cases, Contract & Integration

### [I3] Reference count updates on chapter delete/restore are outside the transaction
- **File:** `packages/server/src/chapters/chapters.service.ts:156-170` (delete), `packages/server/src/chapters/chapters.service.ts:264-278` (restore)
- **Bug:** `deleteChapter` soft-deletes inside a transaction (lines 151-154) but decrements image reference counts outside it (lines 157-169, best-effort). `restoreChapter` similarly increments outside the transaction (lines 265-278). In contrast, `updateChapter` (lines 92-108) correctly does ref count changes inside the transaction. If the server crashes between the transaction commit and the ref count update, counts drift permanently. Partial failures in the loop leave some images with stale counts.
- **Impact:** Images may show incorrect "used in N chapters" data in the gallery. The `liveCheckImageReferences` on the delete path mitigates data loss, but the UX can be misleading and the inconsistency with `updateChapter` suggests this was an oversight.
- **Suggested fix:** Move the reference count decrements/increments inside the same transaction, matching the pattern in `updateChapter`.
- **Confidence:** High
- **Found by:** Logic & Correctness (Server), Error Handling & Edge Cases, Concurrency & State, Contract & Integration

### [I4] Image metadata save failure is completely silent
- **File:** `packages/client/src/components/ImageGallery.tsx:150-160`
- **Bug:** When the `handleSave` API call fails, the catch block at line 157 sets `setSaveStatus("idle")` with no user feedback -- no announcement, no error message, no visual indication. Compare with `handleDelete` (lines 194-198) and `handleFileSelect` which both announce failures via the `aria-live` region.
- **Impact:** The user clicks Save, it fails silently, and the button resets to "Save". The user may leave the detail view thinking alt text/caption/source/license were saved when they weren't -- silent data loss.
- **Suggested fix:** Add an `announce()` call in the catch block (e.g., `announce("Save failed")`) and consider a visible error state on the Save button.
- **Confidence:** High
- **Found by:** Logic & Correctness (Client)

### [I5] Delete-blocked message shows empty chapter list before references load
- **File:** `packages/client/src/components/ImageGallery.tsx:263, 393-397`
- **Bug:** When `confirmingDelete` is true and `isUsed` is true but `referencesLoaded` is false (references API call hasn't returned yet), the UI renders `S.deleteBlocked(references.map(r => r.title))` with `references` still as `[]`. This produces a grammatically broken message like "This image is used in: . Remove it from those chapters first."
- **Impact:** Confusing error message on slow connections -- the user is told the image is in use but no chapters are listed.
- **Suggested fix:** Show a loading indicator or disable the delete confirmation until `referencesLoaded` is true, or use a different message when references haven't loaded yet.
- **Confidence:** Medium
- **Found by:** Logic & Correctness (Client)

## Suggestions

- **[S1]** `packages/server/src/export/epub.renderer.ts:4-5`, `packages/server/src/export/image-resolver.ts:2-3` — Export renderers import `imagesRepo` and `getDb()` directly, bypassing the ProjectStore abstraction used elsewhere. Route image lookups through `getProjectStore()` for architectural consistency. *(Found by: Logic & Correctness (Server))*

- **[S2]** `packages/server/src/images/images.service.ts:40-44` — MIME type validation trusts the client-supplied `Content-Type` header (via multer). Adding magic byte verification (JPEG: `FF D8 FF`, PNG: `89 50 4E 47`, GIF: `47 49 46 38`, WebP: `52 49 46 46...57 45 42 50`) would close this defense-in-depth gap. *(Found by: Security)*

- **[S3]** `packages/client/src/components/ImageGallery.tsx:162-178` — `handleInsert` skips the auto-save when `saveStatus === "saving"`. If the in-flight save fails or the user edited metadata after clicking Save, the database and inserted editor node can get out of sync. Consider awaiting the in-flight save or firing a new one. *(Found by: Logic & Correctness (Client), Error Handling, Concurrency)*

- **[S4]** `packages/server/src/images/images.service.ts:138-162` — TOCTOU race in `deleteImage`: reference check and file/DB deletion are not in a transaction. A concurrent chapter save could add a reference between the check and delete. Low practical risk in single-user mode, but wrapping in a transaction would be more robust. *(Found by: Error Handling, Concurrency, Security)*

- **[S5]** `packages/server/src/db/migrations/012_create_images.js` — No index on `images.project_id`. SQLite does not auto-index FK columns. Add `table.index("project_id")` for project-scoped queries. Low impact at current scale. *(Found by: Logic & Correctness (Server))*

- **[S6]** `images.references.ts:17` vs `image-resolver.ts:62`, `epub.renderer.ts:18`, `docx.renderer.ts:314` — Inconsistent UUID regex patterns. The reference counter uses a strict UUID format; the export renderers use a loose `[0-9a-f-]{36}` that matches invalid strings. Extract a shared constant. *(Found by: Error Handling, Contract & Integration)*

- **[S7]** `packages/server/src/images/images.service.ts:132-135` — Non-null assertion `updated!` after `findImageById` in `updateImageMetadata`. If a concurrent delete removes the image between update and re-read, `null` flows through as the response body. Replace with a null guard. *(Found by: Error Handling)*
