## Executive Summary

Phase 4a adds a comprehensive image management system: upload, gallery UI, reference counting, export embedding, and purge cascade. The implementation is well-structured and closely aligns with the design document. Three important bugs were found: the delete operation orders disk unlink before DB removal (creating ghost records on failure), the client crashes on non-JSON 409 responses during image deletion, and unresolved images leak internal API URLs in HTML/Markdown exports. The remaining findings are suggestions for defense-in-depth, code deduplication, and minor accessibility corrections.

## Important Issues

### [I1] deleteImage unlinks file before removing DB row — ghost records on partial failure

- **File:** `packages/server/src/images/images.service.ts:150-162`
- **Bug:** `unlink(filePath)` runs on line 154, then `store.removeImage(id)` on line 161. If `removeImage` throws (e.g., DB locked, WAL full), the file is gone but the DB record persists. The image appears in the gallery but serves a 404. The upload path (lines 70-83) correctly handles the reverse case — it cleans up the file if the DB insert fails — showing awareness of the pattern but applying it inconsistently.
- **Impact:** Users see a broken image in the gallery with no recovery path other than deleting it again.
- **Suggested fix:** Reverse the order: delete the DB record first, then unlink the file. Orphaned files (DB delete succeeds, unlink fails) are less harmful than ghost records — they waste disk space but cause no user-visible inconsistency.
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling & Edge Cases

### [I2] Client api.images.delete crashes on non-JSON 409 response

- **File:** `packages/client/src/api/client.ts:222-223`
- **Bug:** The delete method calls `res.json().catch(() => null)`. If a 409 response has a non-JSON body (e.g., reverse proxy HTML error page), `body` is `null`. Line 223 returns `null` directly. The caller in `ImageGallery.tsx:186` does `"error" in result` — the `in` operator on `null` throws `TypeError: Cannot use 'in' operator to search for 'error' in null`.
- **Impact:** Any infrastructure component (reverse proxy, CDN) returning a non-JSON 409 crashes the gallery UI with an unhandled exception.
- **Suggested fix:** Guard the 409 path:
  ```typescript
  if (res.status === 409) {
    if (body && typeof body === "object") return body;
    throw new ApiRequestError("Delete blocked (conflict)", 409);
  }
  ```
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [I3] Unresolved images leave broken /api/images/ URLs in HTML/Markdown exports

- **File:** `packages/server/src/export/image-resolver.ts:56-79`
- **Bug:** `resolveImagesInHtml` only replaces `src` attributes for images it can resolve (where `resolveImage` returns non-null). If an image file is missing from disk, the original `/api/images/{uuid}` URL remains in the exported HTML. This leaks internal API paths and produces a broken image. By contrast, the EPUB renderer removes the `<img>` tag entirely, and the DOCX renderer omits it silently — three formats handle missing images three different ways.
- **Impact:** Exported HTML/Markdown files contain non-functional internal URLs. If shared, these reveal the application's API structure.
- **Suggested fix:** After resolving all found images, do a cleanup pass to replace any remaining `/api/images/` URLs:
  ```typescript
  resolvedHtml = resolvedHtml.replace(
    /<img[^>]*src="\/api\/images\/[^"]*"[^>]*>/gi,
    '[Image not found]'
  );
  ```
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

## Suggestions

- **Dead-code guard in plain text renderer** (`export.renderers.ts:250`): `match.startsWith("[Image:")` can never be true — `match` is always an `<img>` tag. Harmless but misleading. Remove the guard. (Logic & Correctness)
- **Duplicated image resolution logic** (`epub.renderer.ts:18-83` vs `image-resolver.ts:56-100`): Both do regex-based URL replacement with caption/figcaption wrapping, with subtle behavioral differences. EPUB renderer bypasses image-resolver entirely. Consider refactoring to a shared function with strategy parameter. (Contract & Integration)
- **HTML escaping duplicated inline** (`image-resolver.ts:86-91`, `epub.renderer.ts:68-73`): `escapeHtml()` is exported from `export.renderers.ts` and epub.renderer.ts even imports it, but both files inline the same 5-replacement chain for caption escaping. Use the existing function. (Contract & Integration)
- **Double-click on Insert can trigger duplicate insertion** (`ImageGallery.tsx:163-179`): `handleInsert` is async with no concurrency guard. The `saveStatus === "saving"` check partially mitigates but doesn't prevent two `onInsertImage` calls. Add an `isInserting` ref guard. (Concurrency & State)
- **Inline SQL in SqliteProjectStore** (`sqlite-project-store.ts:231-242`): `setImageReferenceCount` and `listChapterContentByProject` have inline Knex queries instead of delegating to repository modules, breaking the store facade pattern. (Contract & Integration)
- **Thumbnail grid missing `<ul>` list semantics** (`ImageGallery.tsx:230`): Design specifies `<ul>` with `role="list"` for the thumbnail grid; implementation uses a `<div>` grid. Affects screen reader list navigation. (Plan Alignment)
- **No file-magic validation for uploads** (`images.service.ts:40-44`): MIME type validation relies solely on the Content-Type header. Adding a magic-bytes check for JPEG/PNG/GIF/WebP would be straightforward defense-in-depth. (Security)
- **Image API uses project ID while other endpoints use slug** (`client.ts:170`): Image endpoints take the project UUID while all other project endpoints use the slug. May be intentional (images tied to UUIDs on disk) but is a surprising API asymmetry. (Contract & Integration)
- **No orphan file cleanup** (`images.service.ts:154-158` + `purge.ts`): If file unlink fails during normal delete, the DB record is removed but the file persists. Purge only cleans up entire project directories, not individual orphans. Slow disk leak over time. (Error Handling)
- **TOCTOU in deleteImage** (`images.service.ts:138-162`): liveCheckImageReferences runs outside a transaction before the delete. A concurrent auto-save could add a reference between the check and delete. Low risk in a single-user app but worth noting. (Concurrency & State)
- **localStorage key naming deviation** (`useReferencePanelState.ts:6-7`): Design says `smudge-ref-panel-*` (hyphens), code uses `smudge:ref-panel-*` (colons). The implementation plan matches the code, so this was an intentional plan-time decision. (Plan Alignment)
