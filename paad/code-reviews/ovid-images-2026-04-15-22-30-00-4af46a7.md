## Executive Summary

The Phase 4a (Reference Panel & Images) implementation is comprehensive and closely follows the design/plan docs, with several positive architectural deviations. However, one critical bug will produce corrupt DOCX exports (image dimensions in EMUs instead of pixels), and several important issues exist around reference count drift during project delete/restore, missing EPUB cleanup, and silent WebP degradation in DOCX. Overall code quality is high — security architecture is well-designed (UUID-based serving eliminates path traversal), and the `liveCheckImageReferences` safety net catches most reference count drift at delete time.

## Critical Issues

### [C1] DOCX export image dimensions use EMU values where the library expects pixels
- **File:** `packages/server/src/export/docx.renderer.ts:200-203`
- **Bug:** `DEFAULT_IMAGE_WIDTH = 3657600` and `DEFAULT_IMAGE_HEIGHT = 2743200` are English Metric Unit values, but the `docx` library's `ImageRun` `transformation.width/height` fields expect pixels. The library internally multiplies by 9525 to convert to EMUs, so the resulting images would be `3657600 * 9525 / 914400 ~= 38,100 inches` wide — completely breaking every DOCX export containing an image.
- **Impact:** Every DOCX export with images produces a corrupt/unusable document.
- **Suggested fix:** Use pixel values: `const DEFAULT_IMAGE_WIDTH = 400; const DEFAULT_IMAGE_HEIGHT = 300;` (~4.17in x 3.125in at 96 DPI).
- **Confidence:** High
- **Found by:** Contract & Integration — Server

## Important Issues

### [I1] Project soft-delete does not decrement image reference counts
- **File:** `packages/server/src/projects/projects.service.ts:164-166`
- **Bug:** `deleteProject` calls `softDeleteChaptersByProject` which bulk-marks chapters as deleted but does NOT decrement image `reference_count`. Compare with individual `deleteChapter` (in `chapters.service.ts:170-177`) which properly decrements. When chapters are later restored via `restoreChapter` (line 247-249), it always increments — creating double-counted references after a delete+restore cycle.
- **Impact:** Reference counts inflate after project delete/restore. Gallery shows incorrect "used/unused" badges. Images may be incorrectly blocked from deletion. The `liveCheckImageReferences` safety net corrects drift at image-delete time, limiting the worst outcome, but gallery UX is degraded.
- **Suggested fix:** In `deleteProject`, collect all image IDs from non-deleted chapters and decrement their reference counts within the same transaction, mirroring what `deleteChapter` does per-chapter.
- **Confidence:** High
- **Found by:** Logic & Correctness — Server, Concurrency & State

### [I2] EPUB resolver doesn't strip unresolvable image URLs
- **File:** `packages/server/src/export/epub.renderer.ts:18-76`
- **Bug:** When a DB lookup returns null (`if (!row) continue` at line 31) or `mimeToExt` returns null, the original `<img src="/api/images/...">` tag is left in the EPUB HTML. Compare with `image-resolver.ts:97` which has a catch-all `replace(/<img[^>]*src="\/api\/images\/[^"]*"[^>]*>/gi, "")`. The EPUB version lacks this cleanup.
- **Impact:** Exported EPUB files contain broken `<img>` tags with internal API URLs that won't render and leak server structure.
- **Suggested fix:** Add the same catch-all cleanup at the end of `resolveImagesForEpub`.
- **Confidence:** High
- **Found by:** Contract & Integration — Server

### [I3] WebP images silently replaced with text placeholder in DOCX export
- **File:** `packages/server/src/export/docx.renderer.ts:205-209`
- **Bug:** `MIME_TO_DOCX_TYPE` maps only `image/jpeg`, `image/png`, and `image/gif`. The app accepts `image/webp` uploads, but DOCX export renders WebP images as `[Image: {alt} — format not supported in DOCX]` with only a `logger.warn`. No user-facing warning before or during export.
- **Impact:** Users who upload WebP images and export to DOCX silently lose their images. They may not notice until reviewing the exported document.
- **Suggested fix:** Either convert WebP to PNG server-side during DOCX export, or warn the user in the export dialog when WebP images exist and DOCX is selected.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [I4] Multi-image drop silently discards all but the first image
- **File:** `packages/client/src/components/Editor.tsx:61-72`
- **Bug:** The `handleDrop` callback iterates over dropped files but `return true` on the first image match (line 68), breaking out of the loop. All subsequent images in a multi-file drop are silently discarded. Same issue in `handlePaste` (line 56). No user feedback about discarded files.
- **Impact:** Users who drop multiple images lose all but the first with no indication. Note: the design doc states "Paste/drop handles one image at a time" — so single-image handling is by design, but the silent discard of extras is the issue.
- **Suggested fix:** Announce "Only one image can be inserted at a time" via `onImageAnnouncement` when additional images are present in the drop/paste.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

## Suggestions

### [S1] Duplicate aria-live announcements on image insert
- **File:** `packages/client/src/pages/EditorPage.tsx:460-464` and `packages/client/src/components/ImageGallery.tsx:178`
- **Bug:** When "Insert at cursor" is clicked, two separate `aria-live="polite"` regions fire: ImageGallery announces with the filename, EditorPage announces with the alt text. Screen readers may read both competing announcements.
- **Suggested fix:** Remove the announcement from one location — either EditorPage's `onInsertImage` callback or ImageGallery's `handleInsert`.
- **Found by:** Logic & Correctness — Client, Contract & Integration — Client

### [S2] Timer leak in EditorPage image announcements
- **File:** `packages/client/src/pages/EditorPage.tsx:438-441, 462-463`
- **Bug:** Both `onImageAnnouncement` and `onInsertImage` use bare `setTimeout` calls without tracking or cleanup. Rapid successive announcements cause earlier timeouts to prematurely clear later announcements. Compare with ImageGallery's `announce()` which properly uses `announcementTimerRef`.
- **Suggested fix:** Store timer IDs in refs and clear previous timers before setting new ones.
- **Found by:** Logic & Correctness — Client

### [S3] TOCTOU race in deleteImage live check
- **File:** `packages/server/src/images/images.service.ts:138-166`
- **Bug:** `deleteImage` runs `liveCheckImageReferences` (scan + correct count) then `store.removeImage` as separate non-transactional operations. A concurrent chapter save could insert a reference between the check and the delete. Low risk for a single-user app but architecturally improvable.
- **Suggested fix:** Wrap the live check + delete in a single transaction.
- **Found by:** Concurrency & State

### [S4] Thumbnail grid uses div instead of semantic list
- **File:** `packages/client/src/components/ImageGallery.tsx:230`
- **Bug:** The image thumbnail grid uses `<div className="grid grid-cols-2 ...">` instead of `<ul role="list">` with `<li>` wrappers as specified in the design document's accessibility section. Given CLAUDE.md's emphasis on semantic HTML, this should use list semantics for screen reader navigation.
- **Suggested fix:** Wrap the grid in `<ul role="list">` and each thumbnail `<button>` in `<li>`.
- **Found by:** Plan Alignment

### [S5] Duplicated image resolution logic between HTML and EPUB exporters
- **File:** `packages/server/src/export/epub.renderer.ts:18-77` vs `packages/server/src/export/image-resolver.ts:57-100`
- **Bug:** `resolveImagesForEpub` duplicates regex matching, caption building, figure/figcaption wrapping, and cleanup logic from `resolveImagesInHtml`. This duplication is the root cause of [I2] — the EPUB version is missing the catch-all cleanup that the HTML version has. Logic will continue to diverge over time.
- **Suggested fix:** Extract shared helpers for regex matching, caption building, and cleanup, parameterized by URL resolution strategy (base64 data URI vs file URL).
- **Found by:** Contract & Integration — Server
