## Important Issues

### [I1] Malformed 409 response silently treated as successful image deletion
- **File:** `packages/client/src/components/ImageGallery.tsx:186-191`
- **Bug:** `handleDelete` checks `if ("error" in result)` to detect a 409 "image in use" response. If the 409 body is a valid JSON object but lacks an `error` key (e.g., proxy interference, server format change), the check fails and execution falls through to lines 192-194 which clear `selectedImage` and refresh the grid — the user sees the image disappear even though deletion failed.
- **Impact:** Silent data inconsistency. The image remains on disk and in the DB, but the UI shows it as deleted until the next page refresh.
- **Suggested fix:** Check for the positive success case (`"deleted" in result && result.deleted`) instead of only checking for the error case.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [I2] Incomplete ARIA tab pattern in ReferencePanel (WCAG 2.1 AA violation)
- **File:** `packages/client/src/components/ReferencePanel.tsx:71-83`
- **Bug:** The tab bar has `role="tablist"` and the tab has `role="tab"` with `aria-selected`, but neither the tab nor the tabpanel has an `id` attribute, `aria-controls`, or `aria-labelledby`. These are required for the ARIA tabs pattern per WAI-ARIA Authoring Practices. The project's CLAUDE.md mandates WCAG 2.1 AA compliance.
- **Impact:** Screen readers cannot programmatically associate the tab with its panel. Keyboard-only users have no arrow-key navigation between tabs.
- **Suggested fix:** Add `id="images-tab"` to the tab button, `aria-controls="images-tabpanel"` to the tab, and `id="images-tabpanel" aria-labelledby="images-tab"` to the tabpanel div.
- **Confidence:** High
- **Found by:** Plan Alignment

### [I3] Delete-blocked message shows chapter titles as plain text, not clickable links
- **File:** `packages/client/src/components/ImageGallery.tsx:394-419`
- **Bug:** When an image is referenced and the user tries to delete it, the chapter titles are rendered as plain text in an aria-live announcement and inline text. The design document specifies: "Chapter titles are clickable links" that navigate to the chapter in the editor.
- **Impact:** Users must manually find and navigate to chapters referencing the image to remove it before deletion. Reduces usability for the delete workflow.
- **Suggested fix:** Render chapter titles in the delete-blocked section as `<button>` elements that call `onNavigateToChapter(chapterId)`, matching the clickable links in the "Used in" section above.
- **Confidence:** High
- **Found by:** Plan Alignment

- **File:** `packages/server/src/chapters/chapters.service.ts:84-98`
- **Description:** The image reference diff is computed before the transaction begins (line 100). A concurrent save to the same chapter could cause reference count drift. Mitigated by SQLite single-writer semantics and single-user design, and by the corrective live-check in `deleteImage`.
- **Confidence:** Medium
- **Found by:** Logic & Correctness, Concurrency & State

### [S5] 409 response body returned to client without schema validation
- **File:** `packages/client/src/api/client.ts:222-225`
- **Description:** On 409, `api.images.delete` returns whatever JSON the server sent (`if (body && typeof body === "object") return body`) without verifying the expected shape. The caller casts `result.error` without guards. Related to I1.
- **Confidence:** Medium
- **Found by:** Logic & Correctness, Error Handling, Contract & Integration

### [S6] handleInsert skips metadata save when another save is in-flight
- **File:** `packages/client/src/components/ImageGallery.tsx:163-176`
- **Description:** When `saveStatus === "saving"`, the auto-save guard is skipped and the image is inserted with potentially stale metadata in the DB. The editor gets the correct local alt text, but the DB may disagree until the in-flight save completes (or if it fails).
- **Confidence:** Medium
- **Found by:** Logic & Correctness, Error Handling

### [S7] `file://` URLs in EPUB don't encode special characters and leak server paths
- **File:** `packages/server/src/export/epub.renderer.ts:39`
- **Description:** `file://${filePath}` constructed by string concatenation. Paths with spaces or special characters produce invalid URLs. Also, absolute server paths are embedded in the EPUB content, leaking filesystem structure if the EPUB is shared.
- **Suggested fix:** Use `url.pathToFileURL(filePath).href` from Node.js `url` module.
- **Confidence:** Medium
- **Found by:** Error Handling, Security

### [S8] Duplicated image resolution logic between HTML and EPUB renderers

- **Description:** `resolveImagesForEpub` reimplements the three-pass image resolution from `resolveImagesInHtml` (regex match, src replacement with `data-image-id`, figcaption wrapping, unresolved cleanup). The only difference is `data:` URIs vs `file://` URLs. Bug fixes in one must be manually replicated.

### [S9] Three copies of UUID extraction regex with two different patterns

- **Description:** The image-resolver and DOCX renderer use `[0-9a-f-]{36}` (loose), while the reference counter uses a strict UUID pattern `[0-9a-f]{8}-[0-9a-f]{4}-...`. They could match different strings.
- **Suggested fix:** Extract a shared regex constant.
- **Confidence:** Medium
- **Found by:** Contract & Integration

### [S10] WebP images labeled as PNG in DOCX package
- **File:** `packages/server/src/export/docx.renderer.ts:205-213`
- **Description:** `MIME_TO_DOCX_TYPE` maps `image/webp` to `"png"`, embedding raw WebP bytes with a PNG label. Word 2019+ handles this via magic-byte detection, but older versions show broken images with no warning. The comment at line 333 about "e.g. webp uploads" is stale since WebP IS in the map.
- **Confidence:** Medium
- **Found by:** Error Handling

### [S11] MIME type trusted from client Content-Type header without magic-byte verification
- **File:** `packages/server/src/images/images.service.ts:40`
- **Description:** `uploadImage` validates `file.mimetype` against `ALLOWED_MIMES`, but the mimetype comes from the client-supplied Content-Type header via multer. No server-side magic-byte verification. A crafted request could upload non-image content. Low severity for a single-user no-auth app.
- **Confidence:** Medium
- **Found by:** Security

### [S12] Plain text export uses bare `[Image]` without filename fallback
- **File:** `packages/server/src/export/export.renderers.ts:251-252`
- **Description:** The design specifies `[Image: {alt text or filename}]` but the implementation produces `[Image: ]` for empty alt text (trailing space) or `[Image]` for missing alt attribute. The renderer operates on HTML strings and has no access to image metadata for a filename fallback.
- **Confidence:** High
- **Found by:** Plan Alignment

### [S13] `multer.memoryStorage()` buffers entire file into RAM
- **File:** `packages/server/src/images/images.routes.ts:7`
- **Description:** The design calls for streaming rejection to "prevent memory exhaustion from large or malicious uploads." The implementation uses `memoryStorage()` which buffers the full file (up to 10MB) into a Node.js Buffer. For a single-user app this is unlikely to cause issues in practice.
- **Confidence:** Medium
- **Found by:** Plan Alignment

### [S14] `GET /api/images/{id}/references` endpoint not in design API table
- **File:** `packages/server/src/images/images.routes.ts:104-117`
- **Description:** The design lists 5 API endpoints but the implementation adds a 6th (`/references`). This is an additive deviation needed by the gallery's "Where-used display" feature. Not a bug — just undocumented.

- **Found by:** Plan Alignment

### [S15] All images loaded into memory simultaneously during HTML export
- **File:** `packages/server/src/export/image-resolver.ts:64-79`
- **Description:** Every matched image is read from disk into a Buffer and base64-encoded into the HTML string. With 10MB per image, an image-heavy project could exhaust Node.js memory during export. No size guard exists.
- **Confidence:** Medium
- **Found by:** Error Handling
