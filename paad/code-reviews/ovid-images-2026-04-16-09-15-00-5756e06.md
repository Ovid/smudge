## Important Issues

### [I1] handleInsert fires unnecessary PATCH and has silent failure path
- **File:** `packages/client/src/components/ImageGallery.tsx:164-178`
- **Bug:** `handleInsert` auto-saves metadata when `saveStatus !== "saved"`, which includes the `"idle"` state (form untouched). An unnecessary PATCH fires on every first insert. If that PATCH fails, the catch block at line 174 silently returns with no `announce()` call -- the user sees nothing happen with no error feedback.
- **Impact:** Confusing UX on insert failure. Every other error path in this component announces the failure.
- **Suggested fix:** Add `announce(S.saveFailed)` in the catch block. Optionally, track a `dirty` flag to skip the save when the form is unchanged.
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling & Edge Cases

### [I2] api.images.delete may return null body, causing TypeError in caller
- **File:** `packages/client/src/api/client.ts:222,243`
- **Bug:** At line 222, `res.json().catch(() => null)` can produce `null`. At line 243, `return body` returns `null` on a successful response with non-JSON body. The caller (`ImageGallery.tsx:188`) does `"deleted" in result`, which throws `TypeError: Cannot use 'in' operator to search for 'deleted' in null`.
- **Impact:** Network proxy, CDN corruption, or unexpected server response format causes uncaught TypeError, leaving the delete confirmation UI stuck.
- **Suggested fix:** Guard against null: `return body ?? { deleted: true }` or `if (!body || typeof body !== "object") throw new ApiRequestError(...)`.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [I3] Blockquote context drops listDepth in DOCX export
- **File:** `packages/server/src/export/docx.renderer.ts:264-267`
- **Bug:** The `bqCtx` object constructed for blockquotes includes `indent` and `extraRunProps` but does not propagate `listDepth` from the parent context. A list > blockquote > list structure would have the inner list reset to depth 0 instead of continuing from the outer list's depth.
- **Impact:** Incorrect indentation in DOCX exports for nested list-blockquote-list structures.
- **Suggested fix:** Add `listDepth: ctx?.listDepth` to the `bqCtx` object.
- **Confidence:** High
- **Found by:** Logic & Correctness

### [I4] updateChapter reads content and computes image ref diff before transaction
- **File:** `packages/server/src/chapters/chapters.service.ts:58,84-98,100-116`
- **Bug:** The chapter content is read at line 58 and the image reference diff computed at lines 84-98, both outside the transaction starting at line 100. Two concurrent saves for the same chapter could both read the same "old" content, computing stale diffs. Example: chapter has image A; save 1 removes A (diff: -A); save 2 also reads old content and removes A (diff: -A); both decrements fire, driving the reference count below its correct value.
- **Impact:** Reference count drift. Mitigated by the live-check in `deleteImage`, but the gallery "unused" badge could be incorrect between operations.
- **Suggested fix:** Move `findChapterByIdRaw` and diff computation inside the transaction.
- **Confidence:** Medium
- **Found by:** Concurrency & State

### [I5] deleteProject reads chapter content outside transaction
- **File:** `packages/server/src/projects/projects.service.ts:168-179,181-189`
- **Bug:** Same stale-read pattern as I4. `listChapterContentByProject` reads chapter content outside the transaction at line 168. A concurrent chapter save between this read and the transaction at line 181 could cause the image reference count decrements to be based on stale content.
- **Impact:** Reference count drift, same mitigation as I4.
- **Suggested fix:** Move `listChapterContentByProject` inside the transaction.
- **Confidence:** Medium
- **Found by:** Concurrency & State

### [I6] EPUB cover image lookup not scoped to project
- **File:** `packages/server/src/export/epub.renderer.ts:153-168`
- **Bug:** `store.findImageById(options.coverImageId)` at line 155 looks up any image by ID without verifying it belongs to the exporting project. A client could specify a `coverImageId` from a different project or a soft-deleted project's image.
- **Impact:** In single-user mode the risk is low (user's own data). A soft-deleted project's image could be used as cover, and when that project is purged, the file disappears, causing future export failures.
- **Suggested fix:** After fetching the row, verify `row.project_id` matches the project being exported.
- **Confidence:** Medium
- **Found by:** Error Handling & Edge Cases, Security

### [I7] MIME type validated from Content-Type header only, not magic bytes
- **File:** `packages/server/src/images/images.service.ts:40`
- **Bug:** `ALLOWED_MIMES.has(file.mimetype)` validates only the MIME type from the HTTP Content-Type header. No validation of actual file content magic bytes. A crafted request could upload a non-image file (e.g., HTML, executable) with a spoofed `image/jpeg` Content-Type.
- **Impact:** For single-user app, risk is self-harm. Defense-in-depth concern: if `X-Content-Type-Options: nosniff` header is ever lost, a disguised HTML file could enable stored XSS. The file is stored to disk and served as-is.
- **Suggested fix:** Validate magic bytes of the uploaded buffer (first 4-8 bytes) against expected patterns for each MIME type, or use the `file-type` package.
- **Confidence:** Medium
- **Found by:** Security
