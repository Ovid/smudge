## Important Issues

### [I1] `deleteProject` double-decrements `reference_count` for images shared across multiple chapters
- **File:** `packages/server/src/projects/projects.service.ts:169-186`
- **Bug:** `allImageIds` is built by `push(...extractImageIds(content))` per chapter. `extractImageIds` deduplicates within a single chapter (uses a `Set`), but `allImageIds` concatenates results without cross-chapter deduplication. If image X appears in chapters A and B, `incrementImageReferenceCount(X, -1)` is called twice, driving `reference_count` to -1.
- **Impact:** Negative `reference_count` causes false "Unused" badges in the gallery. The `deleteImage` live-check corrects drift at delete-time, so no silent data loss, but the count is wrong in between.
- **Suggested fix:** Deduplicate before decrementing: `const uniqueImageIds = [...new Set(allImageIds)]; for (const imageId of uniqueImageIds) { ... }`
- **Confidence:** High
- **Found by:** Logic & Correctness (Server)

### [I2] `deleteChapter` reads content outside the transaction (TOCTOU race)
- **File:** `packages/server/src/chapters/chapters.service.ts:155-179`
- **Bug:** `findChapterByIdRaw(id)` at line 157 reads content outside the transaction. A concurrent `updateChapter` between this read and the transaction's `softDeleteChapter` could change the chapter's images, causing reference count decrements to use stale image IDs. Compare with `deleteProject` and `updateChapter`, which both read content inside the transaction.
- **Impact:** `reference_count` drift — either under-decremented (phantom references block future deletes) or over-decremented (false "Unused" badges).
- **Suggested fix:** Move the content read inside the transaction, mirroring `deleteProject`:
  ```typescript
  await store.transaction(async (txStore) => {
    const chapter = await txStore.findChapterByIdRaw(id);
    if (!chapter) return;
    // parse imageIds here
    await txStore.softDeleteChapter(id, now);
    // decrement here
  });
  ```
- **Confidence:** High
- **Found by:** Logic & Correctness (Server), Error Handling, Contract & Integration

### [I3] `deleteImage` live-check ignores soft-deleted chapters; restoring breaks image refs
- **File:** `packages/server/src/images/images.service.ts:157`
- **Bug:** `listChapterContentByProject` filters `deleted_at IS NULL`. An image referenced by a soft-deleted chapter is invisible to the live-check, so `reference_count` is corrected to 0 and the image is deleted. If the chapter is later restored from trash (within the 30-day window), its content JSON still points to `/api/images/{id}` but the image is gone — broken image.
- **Impact:** Silent, unrecoverable data loss that contradicts the 30-day trash recovery promise.
- **Suggested fix:** Either (a) also scan soft-deleted chapters and include them as references (treating trash content as still live), or (b) when deleting an image that was found in soft-deleted chapters, warn the user but allow it. Option (a) is safest.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [I4] Missing UUID validation on route params enables path traversal
- **File:** `packages/server/src/images/images.routes.ts:39,67,90,107,122,145`
- **Bug:** `req.params.projectId` and `req.params.id` are passed directly to `getImagePath()` without UUID format validation. `getImagePath` uses `path.join(getDataDir(), "images", projectId, ...)`. A `projectId` like `../../etc` traverses relative to the data directory. The upload path (`mkdir` + `writeFile`) is highest risk.
- **Impact:** In a single-user, no-auth app the attack surface is limited to local network access, but the traversal is real. An attacker could write arbitrary binary data to attacker-chosen filesystem paths.
- **Suggested fix:** Validate params as UUID format at the route layer before passing to services:
  ```typescript
  const UUID_RE = /^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i;
  if (!UUID_RE.test(req.params.projectId)) return res.status(400).json({ ... });
  ```
- **Confidence:** High
- **Found by:** Security
