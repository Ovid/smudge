## Important Issues

### [I1] Missing delete-success announcement for screen readers
- **File:** `packages/client/src/components/ImageGallery.tsx:191-195`
- **Bug:** When image deletion succeeds, the code resets state and refreshes the list but never calls `announce()`. Upload success, insert success, and errors all announce via `aria-live`, but delete success is missing.
- **Impact:** Screen reader users get no confirmation that their delete action worked, violating the project's WCAG 2.1 AA mandate. All other destructive/constructive actions in the gallery have announcements.
- **Suggested fix:** Add `announce(S.deleteSuccess)` (or similar) in the success path before resetting state. Add the corresponding string to `STRINGS.imageGallery`.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [I2] Image list load failure indistinguishable from empty gallery
- **File:** `packages/client/src/components/ImageGallery.tsx:59-72`
- **Bug:** The `.catch(() => {})` swallows the API error entirely. The `images` state remains `[]`, so the UI shows the "No images" empty state whether the server is unreachable or the gallery is genuinely empty.
- **Impact:** If the server is temporarily unavailable, the user sees "No images" which is misleading. They may upload duplicates thinking their images were lost, or not realize the server is down.
- **Suggested fix:** Track a `loadError` state and show a distinct message like "Failed to load images" with a retry option. At minimum, use `announce()` to inform screen reader users.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [I3] WebP images declared as PNG in DOCX export break on Word 2016 and earlier
- **File:** `packages/server/src/export/docx.renderer.ts:205-213`
- **Bug:** The `MIME_TO_DOCX_TYPE` map assigns `"image/webp": "png"`, embedding raw WebP bytes but declaring them as PNG in the DOCX package. The code comment acknowledges this only works in "modern Word (2019+) and LibreOffice." Word 2016 and earlier cannot render WebP at all, resulting in broken/missing images.
- **Impact:** Users targeting older Word versions will see broken images for any WebP uploads. There's no warning in the export dialog about this limitation.
- **Suggested fix:** Either transcode WebP to PNG during DOCX export (using `sharp` or similar), or add a user-facing warning in the export dialog when WebP images are present and DOCX format is selected.
- **Confidence:** Medium
- **Found by:** Contract & Integration

### [I4] Stale references in ImageGallery block valid image deletion
- **File:** `packages/client/src/components/ImageGallery.tsx:84-104, 278, 409-451`
- **Bug:** References are fetched once when the detail view opens (triggered by `selectedImageId`). If the user opens the detail view, switches to the editor, removes the image from content, and auto-save fires, the `references` array remains stale. The client-side `isUsed` check (via `references.length > 0`) blocks the delete UI, even though the server's live-check would correctly allow deletion.
- **Impact:** The user gets stuck seeing "image in use" when it isn't. They must close and reopen the detail panel to refresh references. This is a state synchronization bug, not just a UX polish issue.
- **Suggested fix:** Re-fetch references when the user clicks the delete button (before deciding whether to show "blocked" or "confirm" UI), or skip the client-side guard and rely on the server's 409 response as the authority.
- **Confidence:** Medium
- **Found by:** Concurrency & State
