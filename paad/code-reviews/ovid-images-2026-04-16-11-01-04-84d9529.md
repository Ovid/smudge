## Suggestions

### [S1] `setImageReferenceCount` called unconditionally before delete guard
- **File:** `packages/server/src/images/images.service.ts:180`
- **Bug:** `setImageReferenceCount(id, activeRefCount)` runs before the `if (referencingChapters.length > 0)` guard. When only soft-deleted chapters reference an image, `activeRefCount = 0` is written to the DB, but the delete is blocked. The image remains with `reference_count = 0`, causing the gallery to show an "unused" badge for an image that cannot be deleted.
- **Impact:** Misleading UI state — user sees "unused" but delete is blocked with no clear explanation.
- **Suggested fix:** Move the `setImageReferenceCount` call after the guard (only correct the count when the delete is blocked, or skip it when the image will be removed anyway).
- **Found by:** Logic & Correctness

### [S2] Duplicated image-resolution pipeline in EPUB vs HTML/MD/text exports
- **File:** `packages/server/src/export/epub.renderer.ts:19-79` vs `packages/server/src/export/image-resolver.ts:59-102`
- **Bug:** `resolveImagesForEpub` reimplements the same three-phase pipeline as `resolveImagesInHtml`: (1) regex scan for image IDs, (2) DB+disk lookup, (3) src replacement + figcaption wrapping + unresolved tag stripping. The only difference is the output format (file:// URLs vs base64 data URIs).
- **Impact:** If caption logic, escaping, or the stripping regex changes, it must be updated in two places with no compiler enforcement. Divergence risk.
- **Suggested fix:** Extract a shared `resolveImageSrcs(html, resolveFn)` helper parameterized on a callback that maps `(row, filePath) => string`.
- **Found by:** Contract & Integration

### [S3] Zero-byte upload produces misleading error message
- **File:** `packages/server/src/images/images.service.ts:46` (magic-byte check runs before size check at `:52`)
- **Bug:** A zero-byte file hits `validateMagicBytes` first (which returns `false` for `buffer.length < 12`), producing "File content does not match declared type image/jpeg" instead of an empty-file error.
- **Impact:** Confusing error message for an edge case.
- **Suggested fix:** Add a zero-size guard before the magic-byte check: `if (file.size === 0) return { validationError: "File is empty." };`
- **Found by:** Error Handling & Edge Cases

### [S4] Upload-then-unmount can leave an orphaned image
- **File:** `packages/client/src/components/Editor.tsx:205-218`
- **Bug:** If the editor unmounts (chapter switch) while a paste/drop upload is in flight, the upload resolves after unmount. The handler closure still references the possibly-destroyed TipTap editor, and calling `editor.chain().focus().setImage()` on it may silently no-op. The image is uploaded but never inserted into content, leaving it with `reference_count = 0` in the gallery.
- **Impact:** Orphaned image appears as "unused" in the gallery after a fast chapter switch during paste.
- **Suggested fix:** Check `editor.isDestroyed` before calling chain methods.
- **Found by:** Concurrency & State

### [S5] Soft-deleted chapters in 409 delete response are not navigable
- **File:** `packages/server/src/images/images.service.ts:160-176`
- **Bug:** `deleteImage` scans all chapters (including soft-deleted) and returns blocking chapter titles in the 409 body. The client renders these as clickable "navigate to chapter" buttons, but trashed chapters aren't accessible via normal navigation.
- **Impact:** User clicks a chapter name to understand why deletion is blocked, but the chapter is in the trash — confusing UX.
- **Suggested fix:** Distinguish active vs. trashed chapters in the 409 body, or indicate in the UI that the blocking chapter is in the trash.
- **Found by:** Error Handling & Edge Cases

### [S6] UUID regex defined locally in routes instead of using shared constant
- **File:** `packages/server/src/images/images.routes.ts:11`
- **Bug:** `UUID_RE` is defined inline as a local constant, duplicating the pattern from `images.paths.ts:9` (`UUID_PATTERN`). If the canonical pattern changes, the routes validator won't update.
- **Impact:** Drift risk across UUID validation points.
- **Suggested fix:** Import `UUID_PATTERN` from `images.paths.ts` and construct the anchored regex from it.
- **Found by:** Contract & Integration
