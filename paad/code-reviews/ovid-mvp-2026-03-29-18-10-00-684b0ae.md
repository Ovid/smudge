## Executive Summary

The MVP is well-structured with good test coverage and consistent patterns. However, the save pipeline -- the app's core trust promise -- has three critical data-loss vectors: save responses can silently revert user typing, editor unmount discards pending saves, and preview mode is fundamentally broken due to unparsed JSON content. The server also lacks Express async error handling, meaning any database error will crash requests silently.

## Critical Issues

### [C1] Save response overwrites active chapter, reverting user typing
- **File:** `packages/client/src/hooks/useProjectEditor.ts:57`
- **Bug:** `handleSave` calls `setActiveChapter(updated)` with the server response. Editor.tsx has a `useEffect` (line 104-113) that syncs `activeChapter.content` back into TipTap. If the user types after a save request is sent but before the response arrives, the response overwrites `activeChapter`, the useEffect detects a difference, and calls `setContent` -- reverting the user's new typing. Overlapping saves with no sequence protection compound this.
- **Impact:** Silent data loss -- user's typing is reverted without warning.
- **Suggested fix:** Don't call `setActiveChapter(updated)` from save responses. The editor already holds the current truth. Only update `word_count` in `project.chapters`. Or add a chapter-ID + sequence guard to ignore stale responses.
- **Confidence:** High
- **Found by:** Logic, Concurrency (4 separate findings merged)

### [C2] Editor unmount discards pending save instead of flushing
- **File:** `packages/client/src/components/Editor.tsx:44-51`
- **Bug:** The unmount cleanup clears the debounce timer but does NOT save the pending content. Comment says "Flush pending save on unmount" but code only calls `clearTimeout`. The `flushSave` via `editorRef` only covers explicit chapter switching in `handleSelectChapterWithFlush`, not other unmount paths (browser navigation, route changes, preview toggle).
- **Impact:** Up to 1.5 seconds of writing silently lost on any non-chapter-switch navigation.
- **Suggested fix:** In the unmount cleanup, call `onSaveRef.current(editor.getJSON())` if `dirtyRef.current` is true.
- **Confidence:** High
- **Found by:** Logic, Error Handling, Concurrency (3 agents)

### [C3] `GET /api/projects/:id` returns unparsed JSON string for chapter content
- **File:** `packages/server/src/routes/projects.ts:111-117`
- **Bug:** The projects route returns chapters via `select("*")` from SQLite. The `content` column is stored as a JSON string, but `parseChapterContent()` (which does `JSON.parse`) is never called. Individual chapter routes in `chapters.ts` do call it. As a result, `project.chapters[*].content` is a raw string, not an object. PreviewMode's `generateHTML()` expects an object and fails silently (shows "Unable to render content").
- **Impact:** Preview mode is completely broken -- no chapter content is displayed. Also affects any client code reading `project.chapters[*].content` directly.
- **Suggested fix:** Apply `parseChapterContent` to each chapter in the projects route: `chapters: chapters.map(parseChapterContent)`.
- **Confidence:** High
- **Found by:** Contract & Integration

## Important Issues

### [I1] No retry logic on save failure (spec violation)
- **File:** `packages/client/src/hooks/useProjectEditor.ts:50-64`
- **Bug:** Spec requires "3 retries with exponential backoff (2s/4s/8s)". Implementation makes a single API call and immediately sets `saveStatus` to `"error"` on failure.
- **Impact:** Transient network issues cause immediate error state instead of automatic recovery.
- **Suggested fix:** Add retry loop with backoff delays before setting error status.
- **Confidence:** High
- **Found by:** Logic, Error Handling, Plan Alignment

### [I2] No `beforeunload` guard (spec violation)
- **File:** All client source (missing entirely)
- **Bug:** Spec requires `beforeunload` warning when unsaved changes exist. No `beforeunload` listener exists anywhere in the client codebase.
- **Impact:** User can close/reload the tab and lose all unsaved work without warning.
- **Suggested fix:** Add `beforeunload` listener in Editor.tsx keyed to `dirtyRef.current`.
- **Confidence:** High
- **Found by:** Error Handling, Plan Alignment

### [I3] Express 4.x async route handlers don't catch errors
- **File:** `packages/server/src/routes/projects.ts` and `chapters.ts` (all handlers)
- **Bug:** All route handlers are `async` but Express 4.x does not auto-catch promise rejections. The global error handler in `app.ts:19-27` is never reached for async errors. Any `await` that throws (DB error, disk full) produces an unhandled rejection -- request hangs, process may crash.
- **Impact:** Database errors crash the server silently.
- **Suggested fix:** Wrap all handlers with `asyncHandler(fn) => (req, res, next) => Promise.resolve(fn(req, res, next)).catch(next)`, or upgrade to Express 5.
- **Confidence:** High
- **Found by:** Error Handling

### [I4] Preview shows stale content (not current edits); total word count stale
- **File:** `packages/client/src/hooks/useProjectEditor.ts:57`, `packages/client/src/pages/EditorPage.tsx:315-318, 401-410`
- **Bug:** `handleSave` updates `activeChapter` but not the corresponding entry in `project.chapters`. Preview reads from `project.chapters`, so it shows content from initial load. Footer total word count (`project.chapters.reduce(...)`) is similarly stale.
- **Impact:** Preview doesn't reflect edits; total word count never updates during editing.
- **Suggested fix:** Sync updated chapter data back into `project.chapters` after save. Or compute total as `otherChaptersWordCount + chapterWordCount`.
- **Confidence:** High
- **Found by:** Logic (2 findings)

### [I5] Async API call inside React state updater with no error handling
- **File:** `packages/client/src/hooks/useProjectEditor.ts:98-113`
- **Bug:** `handleDeleteChapter` calls `api.chapters.get(first.id).then(...)` inside a `setProject` callback. This is an anti-pattern (state updaters should be pure). The `.then()` has no `.catch()`. In React strict mode, the updater may fire twice, duplicating API calls.
- **Impact:** Unhandled promise rejection on network error. Potential duplicate API calls in development.
- **Suggested fix:** Move the chapter-fetch logic outside the state updater.
- **Confidence:** High
- **Found by:** Logic, Concurrency, Contract & Integration, Error Handling (4 agents)

### [I6] `apiFetch` crashes on non-JSON error responses
- **File:** `packages/client/src/api/client.ts:18-20`
- **Bug:** On `!res.ok`, calls `await res.json()`. If the server returns HTML (proxy 502, nginx error page), `res.json()` throws SyntaxError, masking the actual HTTP error.
- **Impact:** Network errors behind a reverse proxy show confusing JSON parse errors instead of meaningful messages.
- **Suggested fix:** Wrap `res.json()` in try/catch, fall back to `Request failed: ${res.status}`.
- **Confidence:** High
- **Found by:** Error Handling

### [I7] Project creation not transactional
- **File:** `packages/server/src/routes/projects.ts:26-43`
- **Bug:** Inserts into `projects` and `chapters` as two separate operations. If the chapter insert fails, a project with zero chapters exists -- the editor shows infinite "Loading...".
- **Impact:** Data integrity violation on partial failure.
- **Suggested fix:** Wrap both inserts in `db.transaction()`.
- **Confidence:** High
- **Found by:** Error Handling, Security

### [I8] Chapter reorder not transactional
- **File:** `packages/server/src/routes/projects.ts:200-202`
- **Bug:** Individual UPDATE statements in a loop with no transaction. Partial failure leaves inconsistent sort_order values.
- **Impact:** Data integrity risk on server error.
- **Suggested fix:** Wrap in `db.transaction()`.
- **Confidence:** High
- **Found by:** Concurrency, Error Handling, Security (3 agents)

### [I9] Multiple async callbacks without error handling
- **File:** `packages/client/src/hooks/useProjectEditor.ts` (multiple), `packages/client/src/pages/EditorPage.tsx:60-88`
- **Bug:** `handleCreateChapter`, `handleSelectChapter`, `handleReorderChapters`, `handleUpdateProjectTitle`, `handleRenameChapter`, `openTrash`, `handleRestore`, `confirmDeleteChapter` -- all async with no try/catch. Network failures produce unhandled rejections and no user feedback.
- **Impact:** Silent failures throughout the app with no error indication.
- **Suggested fix:** Add try/catch to each, set appropriate error state.
- **Confidence:** High
- **Found by:** Error Handling

### [I10] Missing Zod validation on PATCH /api/projects/:id
- **File:** `packages/server/src/routes/projects.ts:70`
- **Bug:** Uses `req.body as { title?: string }` type assertion instead of Zod validation. Other endpoints (POST projects, PATCH chapters) correctly use Zod. If `title` is not a string (e.g., number), `title.trim()` throws a 500 instead of returning 400.
- **Impact:** Unvalidated input, inconsistency with other endpoints.
- **Suggested fix:** Create `UpdateProjectSchema` in shared schemas and use `safeParse`.
- **Confidence:** High
- **Found by:** Security, Contract & Integration

### [I11] Drag handle uses `<span>` instead of `<button>` (spec violation)
- **File:** `packages/client/src/components/Sidebar.tsx:81-88`
- **Bug:** The drag handle is a `<span>` with dnd-kit attributes spread. The spec explicitly states "no `<div>`/`<span>` as interactive elements". While dnd-kit adds `role="button"` via attributes, the underlying element should be a `<button>` for proper semantic HTML.
- **Impact:** Accessibility spec violation.
- **Suggested fix:** Change `<span>` to `<button>` with appropriate styling (`type="button"`, remove default button styles).
- **Confidence:** Medium
- **Found by:** Plan Alignment

### [I12] No focus trapping in modal dialogs
- **File:** `packages/client/src/pages/EditorPage.tsx:333-364`, `packages/client/src/pages/HomePage.tsx:103-131`
- **Bug:** Delete confirmation and shortcut help dialogs have `aria-modal="true"` but no focus trap. Tab can escape the overlay to elements behind it.
- **Impact:** WCAG 2.1 AA violation -- keyboard users can interact with obscured content.
- **Suggested fix:** Use a focus trap library or native `<dialog>` with `showModal()` for the delete confirmation.
- **Confidence:** Medium
- **Found by:** Plan Alignment

### [I13] Purge may orphan chapters
- **File:** `packages/server/src/db/purge.ts:3-13`
- **Bug:** Purge deletes chapters and projects independently by their own `deleted_at` timestamps. A project deleted 31 days ago with a chapter deleted only 5 days ago: the project is purged, the chapter is orphaned with a broken `project_id` FK reference.
- **Impact:** Orphaned data, potential constraint errors.
- **Suggested fix:** When purging a project, also purge all its chapters regardless of their `deleted_at` age.
- **Confidence:** Medium
- **Found by:** Logic

## Plan Alignment

**Implemented:**
- Chapter CRUD, project CRUD, soft delete with 30-day trash
- Auto-save with 1.5s debounce, save on blur, flush on chapter switch
- Drag-and-drop reordering + keyboard Alt+Up/Down with live region
- Word count with `Intl.Segmenter`, shared between client and server
- Preview mode with TOC, IntersectionObserver, and chapter navigation
- Accessibility: semantic HTML, ARIA landmarks, `aria-pressed`, `aria-current`, visible focus indicators
- API error envelope format with machine-readable codes
- Background purge of old trash items

**Not yet implemented (neutral):**
- TOC in preview mode is not collapsible
- `prefers-reduced-motion` handling not explicitly present
- Hard delete from trash (`DELETE /api/trash/{id}`)
- Global trash listing (`GET /api/trash`)
- Server-side preview endpoint (`GET /api/projects/{id}/preview`)
- Client-side content cache for unsaved data

**Deviations:**
- Trash restore uses `POST /api/chapters/{id}/restore` instead of spec's `POST /api/trash/{id}/restore`
- Trash is per-project rather than global
- Error code `INVALID_CONTENT` not used; generic `VALIDATION_ERROR` used instead
- Save retry logic, `beforeunload` guard missing (spec core requirements -- see C1, I1, I2)
