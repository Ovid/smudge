## Executive Summary

This is a full MVP implementation of the Smudge writing application. The architecture is solid and the code is well-structured. However, three critical bugs were found in the client-side save/preview pipeline that could lead to data loss or a broken UI. The most impactful finding is that preview mode shows stale content because `handleSave` never updates the `content` field in project state. Additionally, navigating between chapters from preview mode skips the save flush, and deleting all chapters traps the user on an infinite "Loading..." screen.

## Critical Issues

### [C1] Preview mode shows stale content
- **File:** `packages/client/src/hooks/useProjectEditor.ts:62-70`
- **Bug:** After a successful save, `setProject` only updates `word_count` in `project.chapters`, not the `content` field. `PreviewMode` renders `project.chapters[].content`, so it always shows the version loaded at page init, not the user's latest edits.
- **Impact:** Users open preview expecting to see their current work but see an outdated version. Core user-facing bug.
- **Suggested fix:** Also update `content` in the `setProject` callback after save:
  ```ts
  chapters: prev.chapters.map((c) =>
    c.id === savingChapterId
      ? { ...c, word_count: updated.word_count, content }
      : c,
  ),
  ```
- **Confidence:** High
- **Found by:** Frontend Logic

### [C2] Preview navigation skips save flush -- data loss risk
- **File:** `packages/client/src/pages/EditorPage.tsx:413-414`
- **Bug:** `onNavigateToChapter` calls `handleSelectChapter(chapterId)` directly instead of `handleSelectChapterWithFlush(chapterId)`. The sidebar correctly uses the flush wrapper (line 218), but preview does not. If the user has a pending debounced save and navigates from preview to a different chapter, the save is skipped.
- **Impact:** Data loss. Contradicts spec: "On chapter switch, immediate save bypasses debounce."
- **Suggested fix:** Replace `handleSelectChapter(chapterId)` with `handleSelectChapterWithFlush(chapterId)`.
- **Confidence:** High
- **Found by:** Frontend Logic, Error Handling, Contract & Integration, Concurrency

### [C3] Deleting all chapters shows permanent "Loading..." with no escape
- **File:** `packages/client/src/pages/EditorPage.tsx:204-209`
- **Bug:** The guard `if (!project || !activeChapter)` renders "Loading..." and returns early, hiding the sidebar and all controls. When the last chapter is deleted, `activeChapter` becomes `null`, trapping the user with no way to add a chapter or navigate.
- **Impact:** Dead-end UI state. User must use browser back button to escape.
- **Suggested fix:** Differentiate "loading" from "no chapters". When `project` exists but `activeChapter` is null and `project.chapters` is empty, render an empty state with the sidebar and "Add Chapter" button.
- **Confidence:** High
- **Found by:** Frontend Logic

## Important Issues

### [I1] handleDeleteChapter uses stale closure for chapter list
- **File:** `packages/client/src/hooks/useProjectEditor.ts:119`
- **Bug:** `remaining` is computed from the closure-captured `project?.chapters` instead of inside the `setProject` functional updater using `prev.chapters`. A concurrent state update (e.g., save updating word count) could be overwritten.
- **Impact:** Potential data loss of concurrent chapter list changes.
- **Suggested fix:** Compute `remaining` inside the `setProject` updater from `prev.chapters`.
- **Confidence:** High
- **Found by:** Frontend Logic, Error Handling, Contract & Integration, Concurrency

### [I2] handleSelectChapter blocks when activeChapter is null
- **File:** `packages/client/src/hooks/useProjectEditor.ts:103`
- **Bug:** `if (!activeChapter || chapterId === activeChapter.id) return;` prevents chapter selection when `activeChapter` is null. After deleting all chapters and restoring one from trash, the restored chapter cannot be selected.
- **Impact:** Combined with C3, user is stuck after delete-all-then-restore flow.
- **Suggested fix:** Change guard to `if (activeChapter && chapterId === activeChapter.id) return;`
- **Confidence:** High
- **Found by:** Frontend Logic

### [I3] dirtyRef cleared before save completes
- **File:** `packages/client/src/components/Editor.tsx:36-38`
- **Bug:** `dirtyRef.current = false` runs synchronously when the save is initiated, not when it succeeds. If the save fails after all retries and the user has stopped typing, `beforeunload` won't warn about unsaved changes.
- **Impact:** Silent data loss on tab close after failed save.
- **Suggested fix:** Only clear `dirtyRef` after confirmed save success, or re-set it to `true` in the save error path.
- **Confidence:** Medium
- **Found by:** Frontend Logic

### [I4] HomePage has no error handling
- **File:** `packages/client/src/pages/HomePage.tsx:18, 29-39`
- **Bug:** `loadProjects`, `handleCreate`, and `handleDelete` have no try/catch. API failures produce unhandled promise rejections. Failed loads show an empty project list; failed creates/deletes give no user feedback.
- **Impact:** Misleading UI on network failure.
- **Suggested fix:** Add try/catch with error state and user-facing error messages.
- **Confidence:** High
- **Found by:** Error Handling

### [I5] No cancellation of in-flight save retries on chapter switch
- **File:** `packages/client/src/hooks/useProjectEditor.ts:50-83`
- **Bug:** No `AbortController` or cancellation token exists. If a save is mid-retry with backoff and the user switches chapters, the old retry loop continues, potentially setting `saveStatus` to "error" for content that was already saved by the flush.
- **Impact:** Misleading save status indicator.
- **Suggested fix:** Use an AbortController per save operation; abort on chapter switch.
- **Confidence:** Medium
- **Found by:** Concurrency

### [I6] No client-side cache for unsaved content (spec requires it)
- **File:** CLAUDE.md vs. codebase
- **Bug:** CLAUDE.md specifies "client-side cache holds unsaved content until server confirms" but no such cache exists. If the browser crashes during a failed save cycle, content is lost.
- **Impact:** Missing safety net for the core trust promise.
- **Suggested fix:** Implement localStorage/IndexedDB cache keyed by chapter ID; clear on confirmed save.
- **Confidence:** High
- **Found by:** Concurrency

### [I7] New chapter response skips parseChapterContent
- **File:** `packages/server/src/routes/projects.ts:181`
- **Bug:** `POST /:id/chapters` returns the raw DB row without calling `parseChapterContent()`. Currently benign (content is null for new chapters), but inconsistent with all other chapter endpoints. Will break if default content is ever added.
- **Impact:** Latent API contract violation.
- **Suggested fix:** Apply `parseChapterContent()` to the response.
- **Confidence:** Medium
- **Found by:** Backend Logic, Contract & Integration

### [I8] JSON.parse without try/catch on DB content
- **File:** `packages/server/src/routes/chapters.ts:10`, `packages/server/src/routes/projects.ts:137`
- **Bug:** If the DB contains malformed JSON (corruption, manual edit), `JSON.parse` throws. The global error handler catches it as a 500, but one corrupted chapter makes the entire project inaccessible.
- **Impact:** Data availability -- one bad chapter blocks all chapters in the project.
- **Suggested fix:** Wrap in try/catch; return `null` for corrupted content so other chapters remain accessible.
- **Confidence:** Medium
- **Found by:** Error Handling, Security

## Suggestions

- **[S1]** CLAUDE.md specifies `POST /api/trash/{id}/restore` but code implements `POST /api/chapters/:id/restore`. Update CLAUDE.md. *(Backend Logic, Contract & Integration, Security)*
- **[S2]** No project restore endpoint exists. Projects can only be restored as a side effect of chapter restore. *(Backend Logic)*
- **[S3]** Chapter reorder (`PUT /:id/chapters/order`) doesn't update project `updated_at`, unlike other mutations. *(Backend Logic)*
- **[S4]** Save retry doesn't distinguish retryable (5xx/network) from non-retryable (4xx) errors. 14 seconds wasted on permanent failures. *(Error Handling)*
- **[S5]** `TrashView.tsx:36` uses `.replace("Edited", "Deleted")` -- fragile coupling to string content. Add a dedicated string. *(Error Handling)*
- **[S6]** `handleUpdateChapterTitle` and `handleRenameChapter` are near-duplicates. Consolidate. *(Contract & Integration)*
- **[S7]** `GET /api/projects/:id/trash` returns raw chapters without parsing content. *(Contract & Integration)*
- **[S8]** `parseChapterContent` is duplicated between `chapters.ts:6-12` and `projects.ts:134-138`. Extract to shared utility. *(Contract & Integration)*
- **[S9]** Delete confirmation dialogs use `<div role="alertdialog">` instead of `<dialog>` element. `NewProjectDialog` correctly uses `<dialog>`, making this inconsistent. No focus trapping implemented. *(Plan Alignment, Contract & Integration)*
- **[S10]** Delete confirmation dialog markup is duplicated between `EditorPage.tsx` and `HomePage.tsx`. Extract shared component. *(Contract & Integration)*
- **[S11]** `PUT /api/projects/:id/chapters/order` doesn't validate that `chapter_ids` elements are strings/UUIDs (uses type assertion instead of Zod). *(Security)*
