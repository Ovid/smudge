## Executive Summary

The Smudge MVP is a well-structured greenfield implementation with solid architecture decisions (TipTap JSON storage, shared word count, string externalization). The most critical finding is a data loss bug where renaming a chapter silently discards unsaved editor content. Several important issues exist around missing transactions on multi-table writes, stale React closures, and spec deviations in trash management and accessibility. Overall confidence is high -- the save pipeline is well-designed with retry logic and content caching, though a few edge cases in the Editor component's error handling need attention.

## Critical Issues

### [C1] handleRenameChapter overwrites activeChapter, silently discarding unsaved content
- **File:** `packages/client/src/hooks/useProjectEditor.ts:193-195`
- **Bug:** `setActiveChapter(updated)` replaces the active chapter with the server response, which contains the *last saved* content. Any content typed within the 1.5s debounce window that hasn't been auto-saved is silently replaced. This triggers the Editor's content-sync effect (`Editor.tsx:127-136`), which overwrites the editor buffer and sets `dirtyRef=false`, losing the data with no warning.
- **Impact:** Silent data loss during a common user action (renaming a chapter while typing). The `handleSave` function explicitly avoids this pattern with the comment "Don't call setActiveChapter -- the editor holds the current truth" (line 64), but `handleRenameChapter` doesn't follow the same discipline.
- **Suggested fix:** Only update the title field: `setActiveChapter(prev => prev && prev.id === chapterId ? { ...prev, title } : prev)`
- **Confidence:** High
- **Found by:** Logic & Correctness, Contract & Integration, Concurrency & State

## Important Issues

### [I1] Preview shows stale content -- no save flush before opening
- **File:** `packages/client/src/pages/EditorPage.tsx:299,424`
- **Bug:** When the user opens Preview (via button click on line 299 or Ctrl+Shift+P on line 123), `PreviewMode` receives `project.chapters` which contains the last server-confirmed content. No `editorRef.current?.flushSave()` is called before opening preview. If a user types content and immediately opens preview, their recent edits are missing.
- **Impact:** Confusing UX -- user expects to see what they just wrote. The codebase already handles this for chapter switching (`handleSelectChapterWithFlush`), showing awareness of the pattern.
- **Suggested fix:** Call `editorRef.current?.flushSave()` before `setPreviewOpen(true)` in both the button handler and keyboard shortcut.
- **Confidence:** High
- **Found by:** Contract & Integration

### [I2] Project DELETE performs two writes without a transaction
- **File:** `packages/server/src/routes/projects.ts:281-287`
- **Bug:** Soft-deleting chapters (line 282-285) and the project (line 287) are two separate queries with no transaction. A crash between them orphans chapters: they appear deleted but the project is still active. Other routes in the same file (`POST /` at line 30, `PUT /order` at line 229) correctly use `db.transaction()`.
- **Impact:** Data integrity issue -- user would see a project with no chapters and no way to recover them.
- **Suggested fix:** Wrap both operations in `db.transaction()`.
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling, Concurrency & State

### [I3] Chapter PATCH performs two writes without a transaction
- **File:** `packages/server/src/routes/chapters.ts:68-73`
- **Bug:** Updating the chapter (line 68) and the parent project's `updated_at` (lines 71-73) are separate queries with no transaction. If the second fails, the project timestamp is stale.
- **Impact:** Project list sort order becomes incorrect (sorted by `updated_at`). Lower severity than I2 since no data is lost.
- **Suggested fix:** Wrap both operations in `db.transaction()`.
- **Confidence:** Medium
- **Found by:** Concurrency & State

### [I4] handleDeleteChapter uses stale `project` closure
- **File:** `packages/client/src/hooks/useProjectEditor.ts:137`
- **Bug:** `remaining` is computed from `project?.chapters` in the closure, but `setProject(prev => ...)` has already been called on line 133. The `project` variable still holds the pre-update state. If multiple rapid deletes occur, `remaining` could reference stale data, causing the editor to switch to a wrong or deleted chapter.
- **Impact:** Wrong chapter selected after rapid deletion. Mitigated by confirm dialog preventing rapid-fire deletes.
- **Suggested fix:** Compute `remaining` from within the `setProject` updater or derive it from current state via a ref.
- **Confidence:** Medium
- **Found by:** Logic & Correctness, Error Handling, Contract & Integration, Concurrency & State

### [I5] confirmDeleteChapter has unhandled promise rejection on trash refresh
- **File:** `packages/client/src/pages/EditorPage.tsx:92-95`
- **Bug:** After successful deletion, if `trashOpen && projectId`, `api.projects.trash(projectId)` is called with no try/catch. A network failure produces an unhandled promise rejection.
- **Impact:** Console error, potentially crashing the component in strict mode.
- **Suggested fix:** Wrap lines 92-95 in a try/catch.
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling

### [I6] Heading levels H3/H4/H5 -- toolbar labels say H1/H2/H3
- **File:** `packages/client/src/components/Editor.tsx:170-201`
- **Bug:** Toolbar buttons labeled "H1", "H2", "H3" (`STRINGS.toolbar.heading1/2/3`) actually toggle TipTap heading levels 3, 4, and 5. The heading offset may be intentional (chapter title is H2, so content starts at H3), but the toolbar labels are misleading. Screen readers announce "heading level 3" when the user expects "heading level 1".
- **Impact:** Accessibility confusion; semantic HTML mismatch between UI and output.
- **Suggested fix:** Either update labels to "H3"/"H4"/"H5" to match reality, or change heading levels to 1/2/3 to match labels.
- **Confidence:** High
- **Found by:** Plan Alignment

### [I7] ConfirmDialog uses `<dialog open>` instead of `showModal()`, no focus trapping
- **File:** `packages/client/src/components/ConfirmDialog.tsx:38-39`
- **Bug:** The dialog uses the `open` attribute directly rather than the `showModal()` API. This means no native focus trapping, no native `::backdrop`, and no top-layer behavior. Tab can escape to elements behind the dialog. Also missing `role="alertdialog"` for destructive confirmations.
- **Impact:** Keyboard users can tab behind the dialog. Screen readers don't announce it as an alert dialog. The spec requires "proper modal dialogs with focus trapping."
- **Suggested fix:** Use `useEffect` with a ref to call `dialogRef.current.showModal()`. Add `role="alertdialog"`.
- **Confidence:** High
- **Found by:** Plan Alignment

### [I8] "Unsaved changes" save status never displayed
- **File:** `packages/client/src/pages/EditorPage.tsx:366-372`
- **Bug:** The `SaveStatus` type is `"idle" | "saving" | "saved" | "error"` -- there is no `"unsaved"` state. `STRINGS.editor.unsaved` ("Unsaved changes") exists in strings.ts but is never referenced. After typing but before the debounce fires, the status shows empty string.
- **Impact:** The spec (W3) requires four visible states including "Unsaved changes" to support the "trust the save" design principle. Users can't tell when work hasn't been persisted yet.
- **Suggested fix:** Add `"unsaved"` to `SaveStatus`. Set it in `handleContentChange`. Display `STRINGS.editor.unsaved` in the footer.
- **Confidence:** High
- **Found by:** Plan Alignment

### [I9] Trash view only shows chapters, not projects; no permanent delete
- **File:** `packages/client/src/components/TrashView.tsx`, `packages/server/src/routes/projects.ts`
- **Bug:** TrashView only displays chapters from a specific project. The spec (section 4.5, C6) requires: (a) a unified trash view showing both projects and chapters, (b) permanent delete capability, and (c) a `GET /api/trash` endpoint. None of these exist. Trashed projects are invisible and unrestorable via the UI (except as a side effect of restoring one of their chapters).
- **Impact:** Users cannot see, restore, or permanently delete trashed projects.
- **Suggested fix:** Implement unified trash view with project support and permanent delete.
- **Confidence:** High
- **Found by:** Plan Alignment (3 specialist reports)

### [I10] No `prefers-reduced-motion` support
- **File:** (entire client codebase)
- **Bug:** No `prefers-reduced-motion` media query exists anywhere in the codebase. The Sidebar uses CSS transitions via @dnd-kit for drag-and-drop. CLAUDE.md explicitly requires "`prefers-reduced-motion` respected."
- **Impact:** WCAG 2.1 AA accessibility requirement not met.
- **Suggested fix:** Add `@media (prefers-reduced-motion: reduce) { *, *::before, *::after { transition-duration: 0.01ms !important; animation-duration: 0.01ms !important; } }` or use Tailwind's `motion-reduce:` variants.
- **Confidence:** High
- **Found by:** Plan Alignment

### [I11] Purge operation not wrapped in a transaction
- **File:** `packages/server/src/db/purge.ts:7-18`
- **Bug:** The purge function performs 3-4 separate database operations without a transaction. A crash mid-purge could orphan chapters or partially purge projects.
- **Impact:** Data integrity issue on crash during startup purge. Lower risk since it runs once at startup.
- **Suggested fix:** Wrap all operations in `db.transaction()`.
- **Confidence:** Medium
- **Found by:** Error Handling, Concurrency & State

### [I12] Restore succeeds even if parent project was hard-purged
- **File:** `packages/server/src/routes/chapters.ts:118-125`
- **Bug:** Chapter restore (line 119) succeeds, then lines 122-125 attempt to restore the parent project. If the parent was already hard-deleted by `purgeOldTrash`, the UPDATE silently affects 0 rows. The chapter is now orphaned -- it belongs to a non-existent project.
- **Impact:** Orphaned chapter accessible via API but invisible in the UI.
- **Suggested fix:** Verify the parent project exists before restoring. Return an error if it was purged.
- **Confidence:** Medium
- **Found by:** Error Handling

### [I13] flushSave is fire-and-forget -- save may not complete before chapter switch
- **File:** `packages/client/src/pages/EditorPage.tsx:52-53`, `packages/client/src/components/Editor.tsx:113-121`
- **Bug:** `flushSave()` calls `onSaveRef.current(...).then(...)` internally but returns `void`. The chapter switch proceeds immediately without waiting for the save to complete. If the save fails, `dirtyRef` gets set to `true` after the new chapter has loaded, creating a stale dirty signal. The `.then()` also has no `.catch()`.
- **Impact:** On slow networks, recent edits may not reach the server before chapter switch completes. Content cache mitigates this partially.
- **Suggested fix:** Make `flushSave` return the Promise so callers can await it. Add `.catch()`.
- **Confidence:** Medium
- **Found by:** Contract & Integration, Concurrency & State

### [I14] `total_word_count` from SQLite raw expression returns string, not number
- **File:** `packages/server/src/routes/projects.ts:71`
- **Bug:** `db.raw("COALESCE(SUM(chapters.word_count), 0) as total_word_count")` -- SQLite + Knex returns raw expression results as strings. The client type `ProjectListItem.total_word_count` expects `number`. While `"0".toLocaleString()` works in JS, this is a type contract violation that could cause subtle issues in comparisons or arithmetic.
- **Impact:** Type mismatch that currently works by accident but could break on strict comparisons.
- **Suggested fix:** Cast in the route: `result.map(r => ({ ...r, total_word_count: Number(r.total_word_count) }))`, or use Knex's `.sum()` method instead of `db.raw`.
- **Confidence:** Medium
- **Found by:** Contract & Integration

### [I15] Editor onBlur save has no `.catch()` handler
- **File:** `packages/client/src/components/Editor.tsx:90-92`
- **Bug:** `onSaveRef.current(...).then((ok) => { dirtyRef.current = !ok; })` has no `.catch()`. If the promise rejects, this is an unhandled rejection. The `handleSave` function catches errors internally and returns `false`, so this is unlikely in practice but fragile.
- **Impact:** Potential unhandled promise rejection.
- **Suggested fix:** Add `.catch(() => { dirtyRef.current = true; })`.
- **Confidence:** Medium
- **Found by:** Error Handling

## Suggestions

- `packages/client/src/components/Editor.tsx:58-66` -- Unmount flush sets `dirtyRef=false` before the async save completes. Consider removing the eager `dirtyRef=false` or awaiting the save.
- `packages/client/src/components/PreviewMode.tsx:93` -- `dangerouslySetInnerHTML` without sanitization. Low risk for single-user, but consider DOMPurify if content import is ever added.
- `packages/server/src/app.ts:18` -- `express.json()` has no explicit `limit`. The default 100KB may be too small for long chapters. Consider `express.json({ limit: '5mb' })`.
- `packages/client/src/components/Editor.tsx:147-239` -- Nine near-identical toolbar buttons. Consider extracting a `ToolbarButton` component to reduce duplication.
- `packages/shared/src/wordcount.ts:10` -- `extractText` joins nodes with spaces. If TipTap ever produces adjacent text nodes without whitespace (e.g., inline marks on adjacent characters), this could inflate word count by 1.
