## Suggestions

### [S1] `reorderChapters` creates timestamp inside transaction, inconsistent with other methods
- **File:** `packages/server/src/projects/projects.service.ts:248`
- **Bug:** `reorderChapters` calls `new Date().toISOString()` inside the transaction callback, while all other methods (`createProject`, `deleteProject`, `createChapter`, `updateProject`) create `now` before the transaction. Commit `9992668` explicitly refactored to "pass explicit now to updateProjectTimestamp for transactional consistency," but `reorderChapters` was missed.
- **Impact:** Low -- SQLite transactions are fast. If transactions are ever retried, the timestamp would change on each attempt.
- **Suggested fix:** Extract `const now = new Date().toISOString()` before `store.transaction()` and pass it into the callback.
- **Confidence:** High
- **Found by:** Logic & Correctness

### [S2] `restoreChapter()` uses thrown Error messages as control flow instead of typed error classes
- **File:** `packages/server/src/chapters/chapters.service.ts:155-207`
- **Bug:** Throws `new Error("PARENT_PURGED")` and `new Error("CHAPTER_PURGED")` inside the transaction, then catches by string-matching `err.message`. The codebase already has the typed error class pattern (`ProjectTitleExistsError` in `projects.service.ts:37-42`).
- **Impact:** Fragile -- any change to error message strings breaks control flow. String matching can accidentally match unrelated errors.
- **Suggested fix:** Create `ParentPurgedError` and `ChapterPurgedError` classes and use `instanceof` checks.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [S3] `getVelocityBySlug` performs non-transactional reads via dual DB access paths
- **File:** `packages/server/src/velocity/velocity.service.ts:87-106`
- **Bug:** Five separate reads using two access paths (`getDb()` for VelocityRepo, `getProjectStore()` for project/chapter data) with no shared transaction. Each read could observe a different database state.
- **Impact:** Low for single-user SQLite app. Comment at lines 1-6 documents the dual-path design. A concurrent save during velocity page view could briefly show slightly stale numbers.
- **Suggested fix:** Acceptable as-is for MVP. If ever a concern, wrap reads in a single transaction.
- **Confidence:** Medium
- **Found by:** Logic & Correctness, Contract & Integration, Concurrency & State

### [S4] Shutdown race -- `closeDb()` can destroy connection while in-flight requests still use it
- **File:** `packages/server/src/index.ts:71-78`
- **Bug:** `server.close()` stops new connections but keep-alive connections may persist. The callback calls `resetProjectStore()` then `closeDb()`, which could race with mid-execution requests on keep-alive connections.
- **Impact:** Low -- single-user app where shutdown is intentional. 10-second force-exit timeout is a safety net.
- **Suggested fix:** Consider `server.closeAllConnections()` (Node 18.2+) or a shutdown-aware middleware returning 503.
- **Confidence:** Medium
- **Found by:** Error Handling & Edge Cases, Concurrency & State

### [S5] No UUID validation on chapter ID route parameters
- **File:** `packages/server/src/chapters/chapters.routes.ts:11,35,75,89`
- **Bug:** `req.params.id` passed directly to service functions without UUID format validation. Knex prevents injection, but invalid IDs cause unnecessary DB round-trips.
- **Impact:** Low -- non-UUID IDs return 404 correctly. Only cost is wasted DB queries.
- **Suggested fix:** Add Zod `z.string().uuid()` validation or a small middleware to reject malformed IDs with 400.
- **Confidence:** Medium
- **Found by:** Security
