## Executive Summary

The slug implementation is well-structured and follows the design document closely. All major plan items are implemented with justified deviations. However, the review found 1 critical bug (data exposure via the trash endpoint), 5 important issues (state management and error handling in both client and server), and 3 suggestions. The most impactful finding is a missing `whereNull("deleted_at")` filter on the trash endpoint, confirmed by 4 of 5 specialist agents independently.

## Critical Issues

### [C1] GET /:slug/trash exposes soft-deleted project data
- **File:** `packages/server/src/routes/projects.ts:291`
- **Bug:** The trash endpoint queries `db("projects").where({ slug }).first()` without `.whereNull("deleted_at")`. Every other project endpoint filters by `deleted_at IS NULL`. A soft-deleted project's trashed chapters remain accessible via the API.
- **Impact:** Data exposure — after a user deletes a project, its trashed chapters are still retrievable by anyone who knows the slug. Semantically inconsistent with all other endpoints.
- **Suggested fix:** Add `.whereNull("deleted_at")` to the project query:
  ```typescript
  const project = await db("projects")
    .where({ slug: req.params.slug })
    .whereNull("deleted_at")
    .first();
  ```
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling, Contract & Integration, Security

## Important Issues

### [I1] saveProjectTitle discards user's draft on rename failure
- **File:** `packages/client/src/pages/EditorPage.tsx:188-189`
- **Bug:** `setEditingProjectTitle(false)` runs unconditionally after `handleUpdateProjectTitle`, even when the rename fails (returns `undefined`). The user's typed title is discarded and the input reverts to the old title. The user sees an error banner but must re-type the title from scratch.
- **Impact:** Poor UX on rename failure (network error, duplicate title). Particularly frustrating for long titles.
- **Suggested fix:** Only exit edit mode on success. On failure (`newSlug === undefined`), keep `editingProjectTitle` true so the user can retry or press Escape to cancel.
- **Confidence:** High
- **Found by:** Logic & Correctness

### [I2] openTrash and handleRestore silently swallow all errors
- **File:** `packages/client/src/pages/EditorPage.tsx:66-68` and `83-85`
- **Bug:** Both functions have empty `catch` blocks. If the trash endpoint fails or a restore fails, the user gets no feedback — the trash view silently doesn't open, or the chapter silently stays in the trash list.
- **Impact:** For a writing app where data recovery is trust-critical, silent restore failure is problematic. Contradicts the app's error-reporting contract (CLAUDE.md specifies `aria-live="polite"` for status messages).
- **Suggested fix:** Call `setError(...)` with appropriate messages from `STRINGS` in both catch blocks.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [I3] Enter+blur double-fires saveProjectTitle
- **File:** `packages/client/src/pages/EditorPage.tsx:286-294`
- **Bug:** The project title input wires both `onKeyDown` (fires `saveProjectTitle` on Enter) and `onBlur` (fires `saveProjectTitle`). When the user presses Enter, `saveProjectTitle` runs, which calls `setEditingProjectTitle(false)`, which removes the input from the DOM, which fires `onBlur`, which calls `saveProjectTitle` again. The second invocation sends a duplicate PATCH request using the potentially stale slug from the closure.
- **Impact:** Two PATCH requests sent. The second may 404 if the first changed the slug. User may see a flash of error state. Same pattern exists in the chapter title `saveTitle` function.
- **Suggested fix:** Add a guard ref (`isSavingRef`) to prevent re-entry, or set `editingProjectTitle` to false before the async call so the second invocation short-circuits on `!project`.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [I4] TOCTOU race between title uniqueness check and insert
- **File:** `packages/server/src/routes/projects.ts:33-45` (POST) and `133-148` (PATCH)
- **Bug:** Title uniqueness check and `resolveUniqueSlug` run as separate queries outside the transaction. Two concurrent requests with the same title can both pass the check. The DB-level partial unique index prevents data corruption, but the second request gets an unhandled SQLite constraint error surfaced as a 500 instead of a clean 400 with `PROJECT_TITLE_EXISTS`.
- **Impact:** Under concurrent submissions (possible with browser tab duplication or rapid double-submit), user sees a generic 500 error instead of the expected duplicate-title message. Data integrity is preserved by the DB index.
- **Suggested fix:** Wrap uniqueness check + slug resolution + insert inside `db.transaction()`. Catch `SQLITE_CONSTRAINT` errors and map to 400 `PROJECT_TITLE_EXISTS`.
- **Confidence:** Medium
- **Found by:** Logic & Correctness, Error Handling, Concurrency & State

### [I5] handleUpdateProjectTitle closes over stale project slug
- **File:** `packages/client/src/hooks/useProjectEditor.ts:189-202`
- **Bug:** `handleUpdateProjectTitle` is memoized with `[project]` and reads `project.slug` from the closure. Unlike `handleCreateChapter` and `handleReorderChapters` (which read from `projectSlugRef.current`), this callback can hold a stale slug after a rename until the next render. Combined with the double-fire bug (I3), the second call sends a PATCH to the old slug.
- **Impact:** In double-fire scenarios, the second PATCH targets the wrong (now-replaced) slug and 404s.
- **Suggested fix:** Read from `projectSlugRef.current` instead of the closed-over `project.slug`, consistent with the pattern already established in `handleCreateChapter` and `handleReorderChapters`.
- **Confidence:** Medium
- **Found by:** Concurrency & State, Contract & Integration

## Suggestions

### [S1] resolveUniqueSlug has no iteration cap
- **File:** `packages/server/src/routes/resolve-slug.ts:11`
- **Bug:** The `while (true)` loop issues one DB query per iteration with no maximum. Pathological data (many same-base slugs) causes O(N) queries.
- **Suggested fix:** Add a cap (e.g., 100 iterations) and throw a descriptive error if exceeded.
- **Found by:** Error Handling, Security

### [S2] CJK/symbol-only titles silently fall back to "untitled" slug
- **File:** `packages/shared/src/slugify.ts:11`
- **Bug:** Titles composed entirely of non-Latin characters (CJK, Arabic, emoji) produce an empty string after stripping, falling back to `"untitled"`. Multiple such projects get `untitled`, `untitled-2`, `untitled-3`, etc. with no feedback.
- **Suggested fix:** Consider using a short hash or UUID fragment instead of bare `"untitled"` as the fallback discriminator.
- **Found by:** Error Handling & Edge Cases

### [S3] No max title length in Zod schemas
- **File:** `packages/shared/src/schemas.ts:6-12`
- **Bug:** `CreateProjectSchema` and `UpdateProjectSchema` accept arbitrarily long titles. A 5MB title is stored, slugified, and indexed without issue but wastes resources.
- **Suggested fix:** Add `.max(500, "Title is too long")` to both schemas.
- **Found by:** Security
