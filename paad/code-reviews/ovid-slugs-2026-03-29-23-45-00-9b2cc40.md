## Executive Summary

The slug feature is well-implemented overall, with correct slug generation, migration backfill, and client/server integration. Two critical issues stand out: (1) after a chapter restore that also restores its parent project with a new slug, the client session silently breaks because the stale slug is never updated; (2) a transient error during project rename triggers the full-page error overlay, ejecting the user from the editor. Four important issues relate to TOCTOU races in slug resolution and a missing `updated_at` on project restore.

## Critical Issues

### [C1] Stale project slug after chapter restore that also restores parent project
- **File:** `packages/client/src/pages/EditorPage.tsx:73-88`
- **Bug:** `handleRestore` calls `api.chapters.restore()` which may silently re-slug the parent project (via the restore-collision path in `chapters.ts:134-141`). The restore response is a `Chapter` object — it does not include the project's updated slug. The client never refreshes `project.slug`, so all subsequent slug-based API calls (`handleCreateChapter`, `handleReorderChapters`, `openTrash`, etc.) use the stale slug and receive 404 responses.
- **Impact:** After restoring a chapter whose parent project was also restored with a changed slug, the entire editing session silently breaks. The user must manually navigate away and back.
- **Suggested fix:** After a successful restore, re-fetch the project (`api.projects.get(project.slug)`) to pick up any slug change. Alternatively, modify the restore endpoint to return `{ chapter, project }` when a project restore occurs.
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling & Edge Cases

### [C2] Title-update failure triggers full-page error overlay, ejecting user from editor
- **File:** `packages/client/src/hooks/useProjectEditor.ts:199` + `packages/client/src/pages/EditorPage.tsx:207-225`
- **Bug:** `handleUpdateProjectTitle` calls `setError(...)` on failure (line 199 in useProjectEditor). In `EditorPage`, the `error` state is checked on line 207 and renders a full-page "project not found" error overlay, replacing the entire editor. Any transient network error during a project rename destroys the editing session. The `saveProjectTitle` guard (`if (newSlug === undefined) return` on line 196) keeps edit mode open, but `setError` has already fired and will flip the UI on the next render.
- **Impact:** A momentary server hiccup or network glitch during a title rename kicks the user out of the editor to a bare error page. The save pipeline contract says failures should be surfaced inline without disrupting the session.
- **Suggested fix:** Split the `error` state into fatal load errors (which warrant the full-page treatment) and operational errors (which should be shown as an inline toast or indicator). `handleUpdateProjectTitle` should not trigger the full-page error.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

## Important Issues

### [I1] TOCTOU: slug resolved outside transaction on project CREATE — wrong error code on race
- **File:** `packages/server/src/routes/projects.ts:45,51-83`
- **Bug:** `resolveUniqueSlug` is called on line 45 outside the transaction (line 51). Under concurrent requests, two requests can resolve the same slug. The partial unique index catches the collision, but the catch block on line 73 returns `PROJECT_TITLE_EXISTS` — the wrong error code for a slug collision. The user sees "A project with that title already exists" when the title is actually unique.
- **Impact:** Misleading error message on slug collision. Single-user app mitigates the race likelihood.
- **Suggested fix:** Move `resolveUniqueSlug` inside the transaction (pass `trx` instead of `db`), or distinguish slug vs. title constraint errors in the catch.
- **Confidence:** High
- **Found by:** Logic & Correctness, Contract & Integration, Concurrency & State

### [I2] TOCTOU: same race on project PATCH/rename
- **File:** `packages/server/src/routes/projects.ts:161-178`
- **Bug:** Same pattern as I1. `resolveUniqueSlug` on line 161 runs outside the update (which is not even wrapped in a transaction). Slug collision returns `PROJECT_TITLE_EXISTS`.
- **Impact:** Same as I1 — misleading error on slug collision during rename.
- **Suggested fix:** Wrap the title check, slug resolution, and update in a single transaction.
- **Confidence:** High
- **Found by:** Logic & Correctness, Contract & Integration, Concurrency & State

### [I3] TOCTOU: slug resolution outside transaction on chapter restore — unhandled 500
- **File:** `packages/server/src/routes/chapters.ts:134-151`
- **Bug:** `resolveUniqueSlug` is called on line 134 before the transaction on line 145. Unlike the project create/rename handlers, the restore handler has **no catch block** for unique constraint errors. A slug race here becomes an unhandled 500 to the client.
- **Impact:** The most dangerous TOCTOU variant — the server returns a raw 500 with no user-facing explanation, and the chapter is not restored.
- **Suggested fix:** Move `resolveUniqueSlug` inside the transaction, and add a catch for constraint errors.
- **Confidence:** High
- **Found by:** Contract & Integration, Concurrency & State

### [I4] Restored project missing `updated_at` update
- **File:** `packages/server/src/routes/chapters.ts:132`
- **Bug:** The `projectUpdate` object is initialized as `{ deleted_at: null }` and may have `slug` added, but `updated_at` is never set. Every other write operation that touches the project updates `updated_at`. The project list endpoint orders by `updated_at DESC`.
- **Impact:** A restored project appears at its original pre-deletion position in the project list rather than at the top, which is inconsistent with all other write paths.
- **Suggested fix:** Add `updated_at: new Date().toISOString()` to `projectUpdate`.
- **Confidence:** High
- **Found by:** Contract & Integration

## Suggestions

- `packages/server/src/routes/resolve-slug.ts:12` — Off-by-two loop bound: runs 101 iterations (suffix 2–102), error message says "100 attempts". Five specialists agreed. Fix: change `suffix <= MAX_SUFFIX + 2` to `suffix <= MAX_SUFFIX + 1` and correct the error message.
- `packages/server/src/routes/resolve-slug.ts:7-24` — N+1 sequential DB queries per collision. Could be replaced with a single `WHERE slug LIKE ?` query to find all taken slugs at once.
- `packages/client/src/pages/EditorPage.tsx:62-71` — `openTrash` failure is silent (`console.error` only). No user-facing feedback when trash load fails.
- `packages/client/src/pages/EditorPage.tsx:85-87` — `handleRestore` failure is also silent (`console.error` only).
- `packages/server/src/db/migrations/002_add_project_slugs.js:22` — `slug` column is nullable (SQLite ALTER TABLE limitation, documented in comment). No DB-level NOT NULL constraint.
- `packages/client/src/hooks/useProjectEditor.ts:17,195` — Dual write to `projectSlugRef` (useEffect + direct assignment in `handleUpdateProjectTitle`). Necessary for timing reasons but creates a fragile dual-maintenance pattern.
- `packages/client/src/pages/EditorPage.tsx:181-205` — `saveProjectTitle` blur-retry after failure: when `handleUpdateProjectTitle` returns `undefined`, `isSavingProjectTitleRef` is cleared in `finally`, and `onBlur` fires `saveProjectTitle` again — auto-retrying the failed save.
