## Executive Summary

The slug feature is well-implemented and closely follows the design document. Three important issues were found: non-Latin titles all collapse to "untitled" with a hard cap at 100, project title rename errors are silently swallowed with no user feedback, and the chapter restore endpoint has a potential null dereference. The remaining findings are low-severity suggestions around consistency and defense-in-depth.

## Important Issues

### [I1] Non-Latin titles all map to "untitled" slug, hard crash after ~100
- **File:** `packages/shared/src/slugify.ts:1-12`
- **Bug:** `generateSlug()` strips all non-ASCII characters via NFD + regex. A title consisting entirely of CJK, Arabic, Hebrew, Thai, or emoji characters produces an empty string, falling back to `"untitled"`. Every such project gets base slug `"untitled"`, forcing `resolveUniqueSlug` to iterate. After 100 such projects, `resolveUniqueSlug` throws an unrecoverable error (`resolve-slug.ts:27`).
- **Impact:** Hard crash on project creation for users with many non-Latin-titled projects. Even before the cap, each creation does up to N sequential DB queries.
- **Suggested fix:** Either transliterate non-Latin characters (e.g., using a library), incorporate a hash/UUID fragment for the "untitled" fallback, or significantly raise the cap. Consider truncating long slugs while at it.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [I2] handleUpdateProjectTitle swallows all errors with no user feedback
- **File:** `packages/client/src/hooks/useProjectEditor.ts:197-204`
- **Bug:** The `catch` block at line 198 catches all errors and returns `undefined`. The comment explains this avoids the full-page error overlay, but no alternative feedback is provided. A `PROJECT_TITLE_EXISTS` error, network failure, or any other error is completely invisible to the user — the title input simply stays open with no indication of what went wrong.
- **Impact:** User types a duplicate title, hits Enter, and nothing visible happens. They may think the app froze or keep retrying without understanding the problem.
- **Suggested fix:** Return the error message or error code alongside `undefined`, or introduce a local error state specific to the title input (e.g., inline "Title already exists" message below the input).
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [I3] Null dereference on updatedProject.slug in restore endpoint
- **File:** `packages/server/src/routes/chapters.ts:164-165`
- **Bug:** After the restore transaction, `updatedProject` is fetched via `.first()` and `.slug` is immediately accessed without a null check. If the project were hard-purged between the transaction commit and this query, `updatedProject` would be `undefined`, causing `TypeError: Cannot read properties of undefined (reading 'slug')`.
- **Impact:** Unhandled crash producing a generic 500 error, even though the restore itself succeeded. The client never learns the new slug.
- **Suggested fix:** Add a null guard: `project_slug: updatedProject?.slug` or fetch the project inside the transaction.
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling & Edge Cases, Contract & Integration

## Suggestions

- **TOCTOU on title uniqueness** (`projects.ts:34` and `:147`): Title uniqueness check runs outside the transaction with no DB-level unique index on title. Two concurrent creates with the same title could both succeed. Low risk in single-user app, but the pattern is unsound. Consider adding a partial unique index on title or moving the check inside the transaction. *(Found by: Logic & Correctness, Error Handling, Concurrency & State)*
- **Save retries update stale saveStatus** (`useProjectEditor.ts:59-95`): `saveStatus` is global. If the user switches chapters during a save retry (up to 14s backoff), the status bar shows "saved"/"error" for the wrong chapter. Cosmetic, not data-corrupting. *(Found by: Logic & Correctness)*
- **No slug length truncation** (`slugify.ts` + `schemas.ts:6`): Title max is 500 chars; `generateSlug` doesn't truncate. A long ASCII title produces a ~500-char slug in URLs. Consider truncating at ~100-150 chars at a word boundary. *(Found by: Error Handling, Security)*
- **Duplicated generateSlug in migration** (`002_add_project_slugs.js:4-14` vs `slugify.ts`): Inline copy for ESM compatibility. Currently identical but could diverge. Add a cross-reference comment or conformance test. *(Found by: Contract & Integration)*
- **Duplicated collision resolution in migration** (`002_add_project_slugs.js:29-56` vs `resolve-slug.ts`): Same class of duplication risk. *(Found by: Contract & Integration)*
- **Chapter restore missing updated_at** (`chapters.ts:134`): Restore only sets `deleted_at: null`, doesn't update `updated_at`. Project restore does update `updated_at`. Minor inconsistency. *(Found by: Logic & Correctness)*
- **project_slug typed as optional but always returned** (`client.ts:88`): Server always returns `project_slug` in restore response, client types it as optional. Type imprecision, not a bug. *(Found by: Contract & Integration)*
- **Extra project_slug leaks into Chapter[]** (`EditorPage.tsx:81`): Restored chapter with `project_slug` field is pushed into chapters array. Harmless but sloppy — consider destructuring out `project_slug` first. *(Found by: Contract & Integration)*
- **No slug format validation on URL params** (`projects.ts` routes): `req.params.slug` used directly without format validation. Safe due to Knex parameterization but violates defense-in-depth. *(Found by: Security)*
- **parentProject read outside transaction** (`chapters.ts:120`): State could change between the read and the transaction. Minimal risk in single-user app. *(Found by: Concurrency & State)*

## Plan Alignment

2. Slug column is nullable (SQLite limitation) rather than NOT NULL as designed. Documented in migration comments.

4. Chapter restore endpoint returns `project_slug` — necessary for slug system but not mentioned in design.
