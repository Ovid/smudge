## Critical Issues

### [C1] Save triggers editor content rollback -- data loss during typing

- **File:** `packages/client/src/hooks/useProjectEditor.ts:83-86` + `packages/client/src/components/Editor.tsx:140-149`
- **Bug:** After a successful save, `handleSave` calls `setActiveChapter` with the content that was sent to the server. This updates the `content` prop passed to `<Editor>` (via `EditorPage.tsx:587`). The Editor's `useEffect` at line 140 compares the incoming prop (stale saved content) against the editor's internal state (which may contain newer keystrokes). When they differ, it calls `editor.commands.setContent()`, overwriting the editor and setting `dirtyRef.current = false`. Any keystrokes typed during the save round-trip (1.5s debounce + network latency) are silently lost and will not be re-saved.
- **Impact:** Data loss in the core save pipeline during normal editing. The previous code deliberately avoided this with the comment "Don't call setActiveChapter -- the editor holds the current truth."
- **Suggested fix:** Either (a) remove `content` from the `setActiveChapter` call in `handleSave` (only sync `word_count`), or (b) make the Editor's content-sync `useEffect` ignore content changes that originated from saves (e.g., by only firing `setContent` when the chapter ID changes, not when content changes for the same chapter). The stale-content-on-preview-toggle problem that motivated this change should be solved by having the Editor read from the content cache on remount, or by only updating `activeChapter.content` when the editor is not mounted (i.e., `viewMode !== "editor"`).
