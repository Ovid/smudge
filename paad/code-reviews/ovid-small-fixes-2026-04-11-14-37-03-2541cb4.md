## Executive Summary

This branch consolidates timezone settings from a standalone `SettingsDialog` into `ProjectSettingsDialog`, moves the gear icon to be visible on all editor views, and tidies the homepage layout. The consolidation is architecturally sound, but the old `SettingsDialog` component was not deleted (leaving dead code), the new timezone logic has inconsistent error handling compared to sibling fields, and a `key`-prop issue causes the dialog to remount on every settings save. No critical issues; 8 important issues and 2 suggestions.

## Important Issues

### [I1] Timezone not reverted on save failure
- **File:** `packages/client/src/components/ProjectSettingsDialog.tsx:126-135`
- **Bug:** `handleTimezoneChange` sets the timezone optimistically but does not revert it in the `catch` block on failure. By contrast, `saveField` (lines 79-100) carefully reverts `wordCountTarget`, `deadline`, and `threshold` on failure. After a failed timezone save, the dropdown shows the new timezone while the server retains the old one.
- **Impact:** User sees incorrect timezone value; may not realize the save failed if the error banner is brief.
- **Suggested fix:** Capture the previous timezone before the optimistic set and revert in `catch`: `const prev = timezone; setTimezone(value); ... catch { setTimezone(prev); }`
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling & Edge Cases, Concurrency & State

### [I2] `key` prop on ProjectSettingsDialog causes full remount on every settings save
- **File:** `packages/client/src/pages/EditorPage.tsx:567`
- **Bug:** The `key` prop is `` `${project.slug}-${project.target_word_count}-${project.target_deadline}-${project.completion_threshold}` ``. When `onUpdate` re-fetches the project, these values change, causing React to unmount and remount the dialog. This resets `timezone` to `"UTC"` (the `useState` initial value), fires a redundant `GET /settings`, and destroys any visible `saveError`. The user sees a visible timezone flicker on every project field save.
- **Impact:** Jarring UX flicker on every settings save; redundant API calls; loss of error state.
- **Suggested fix:** Use only `project.slug` as the key (or remove the key entirely). The component already receives updated `project` props and doesn't need key-based remounting to resync.
- **Confidence:** High
- **Found by:** Logic & Correctness, Concurrency & State

- **Bug:** Every project's delete button uses `STRINGS.delete.buttonLabel` which resolves to `"Delete"`. The SVG icon is `aria-hidden="true"`, so the accessible name is just "Delete" with no project name. A screen reader user cannot distinguish between delete buttons. The codebase already has `STRINGS.delete.deleteChapterAriaLabel` which takes a title parameter â€” the same pattern should be used here.
- **Impact:** WCAG 2.1 AA violation (4.1.2 Name, Role, Value). Accessibility is a first-class constraint per CLAUDE.md.
- **Suggested fix:** Add `deleteProjectAriaLabel: (title: string) => \`Delete ${title}\`` to strings and use it on the button.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [I8] Race condition â€” timezone fetch can overwrite user selection
- **File:** `packages/client/src/components/ProjectSettingsDialog.tsx:41-56`
- **Bug:** When the dialog opens, `useEffect` fetches the timezone. If the user changes the dropdown before the fetch completes, the fetch response overwrites their selection (`setTimezone(settings.timezone || "UTC")` on line 47). The `cancelled` flag only fires on unmount/re-open, not on user interaction. On slow connections, this window is significant.
- **Impact:** User's timezone change appears to succeed but is visually reverted; server has the correct value but UI shows stale data.
- **Suggested fix:** Track whether the user has interacted with the dropdown (e.g., a `userChangedTimezone` ref) and skip `setTimezone` if they have.
- **Confidence:** High
- **Found by:** Concurrency & State
