## Important Issues

### [I1] Dashboard view-switch button missing error handling
- **File:** `packages/client/src/pages/EditorPage.tsx:414-416`
- **Bug:** The Editor and Preview view-switch buttons use `.catch(() => {})` on their `switchToView()` calls (lines 392, 403), but the Dashboard button uses `await switchToView("dashboard")` inside an async arrow function with no try/catch or `.catch()`. If `switchToView` rejects (e.g., if `flushSave()` implementation changes to propagate errors), this becomes an unhandled promise rejection.
- **Impact:** Inconsistent error handling across sibling buttons. Currently low-risk since `flushSave` swallows errors, but a latent bug if the save pipeline changes.
- **Suggested fix:** Wrap in try/catch or use the same `.catch(() => {})` pattern as the sibling buttons.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [I2] Empty-chapters `onUpdate` sets error state that is never rendered
- **File:** `packages/client/src/pages/EditorPage.tsx:301-331`
- **Bug:** The empty-chapters render path's `ProjectSettingsDialog` `onUpdate` callback calls `setActionError(msg)` on line 325 if the project re-fetch fails. However, the empty-chapters JSX (lines 249-331) does not render the `actionError` banner -- that banner only exists in the main editor view at lines 457-470. Errors are set in state but never displayed to the user.
- **Impact:** Silent failure -- the user sees no indication that the settings re-fetch failed after a successful save.
- **Suggested fix:** Add an error banner to the empty-chapters view, or extract the `onUpdate` callback into a shared `useCallback` that is used by both `ProjectSettingsDialog` instances (which would also fix I3).
- **Confidence:** High
- **Found by:** Contract & Integration, Error Handling & Edge Cases

### [I3] Duplicated `onUpdate` callback has diverged between empty-chapters and main views
- **File:** `packages/client/src/pages/EditorPage.tsx:306-328` vs `608-632`
- **Bug:** Two `ProjectSettingsDialog` instances have nearly identical `onUpdate` callbacks, but the main-view version (line 609) calls `setDashboardRefreshKey((k) => k + 1)` while the empty-chapters version does not. This is a copy-paste divergence that will worsen as more settings are added.
- **Impact:** Low immediate impact (empty-chapters view has no dashboard to refresh), but a maintenance hazard. Any future change to `onUpdate` must be applied in two places.
- **Suggested fix:** Extract the `onUpdate` callback into a shared `useCallback` above the early returns.
- **Confidence:** High
- **Found by:** Contract & Integration, Concurrency & State

## Suggestions

- **[S1]** `packages/client/src/pages/EditorPage.tsx:155` -- Velocity fetch throttle sets `lastVelocityFetch.current` inside `.then()` instead of before the fetch. Two rapid saves can both pass the throttle check and fire duplicate fetches. Move the timestamp update before the fetch call. *(Found by: Concurrency & State)*

- **[S2]** `packages/client/src/components/ProjectSettingsDialog.tsx:50,94,144` -- Shared `saveError` state between project fields and timezone. Changing timezone clears a pending project-field error and vice versa. Consider separate error states. *(Found by: Concurrency & State, Error Handling, also flagged in previous review as S1)*

- **[S3]** `packages/client/src/components/ProjectSettingsDialog.tsx:140-151` -- Timezone revert race on rapid changes. If user changes A->B->C and save B fails after save C succeeds, the revert sets timezone to A (the captured `previous` for the B call), overwriting the successfully-saved C. Low practical risk since timezone changes are rare. *(Found by: Error Handling & Edge Cases)*

- **[S4]** `packages/client/src/api/client.ts:140` -- `settings.get()` returns `Record<string, string>`, so accessing `.timezone` is untyped. A more specific return type would catch typos at compile time. *(Found by: Contract & Integration)*
