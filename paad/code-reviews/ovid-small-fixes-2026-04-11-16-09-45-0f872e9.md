## Important Issues

### [I1] Empty-chapters branch omits ShortcutHelpDialog -- Ctrl+/ permanently blocks all shortcuts
- **File:** `packages/client/src/pages/EditorPage.tsx:274-351`
- **Bug:** The early-return branch for empty chapters renders `ProjectSettingsDialog` but does **not** render `ShortcutHelpDialog`. The `useKeyboardShortcuts` hook (registered at line 229, before early returns) handles `Ctrl+/` by setting `shortcutHelpOpen = true`. Without the dialog in the tree, the state is set but no dialog appears -- and the guard at `useKeyboardShortcuts.ts:75` (`if (shortcutHelpOpenRef.current ...) return`) then suppresses all other shortcuts permanently.
- **Impact:** On a new project with no chapters, pressing `Ctrl+/` permanently blocks all keyboard shortcuts (new chapter, sidebar toggle, etc.) until page refresh.
- **Suggested fix:** Add `<ShortcutHelpDialog open={shortcutHelpOpen} onClose={() => setShortcutHelpOpen(false)} />` to the empty-chapters branch, after the existing `ProjectSettingsDialog`.
- **Confidence:** High
- **Found by:** Contract & Integration

### [I2] Velocity throttle TOCTOU -- `lastVelocityFetch` written after fetch completes, not before
- **File:** `packages/client/src/pages/EditorPage.tsx:147-155`
- **Bug:** The throttle check `if (now - lastVelocityFetch.current < VELOCITY_THROTTLE_MS) return` runs before the fetch, but `lastVelocityFetch.current = Date.now()` is assigned inside the `.then()` callback (line 155). If two rapid `saveStatus === "saved"` transitions trigger the effect before the first fetch resolves, both pass the throttle check and fire concurrent velocity requests.
- **Impact:** The 60-second throttle provides no protection under rapid-save scenarios. Redundant concurrent API calls to the velocity endpoint.
- **Suggested fix:** Set `lastVelocityFetch.current = Date.now()` immediately before launching the fetch, not in `.then()`.
- **Confidence:** High
- **Found by:** Concurrency & State

### [I3] Project settings fields not re-synced from prop on dialog reopen
- **File:** `packages/client/src/components/ProjectSettingsDialog.tsx:44-48`
- **Bug:** `wordCountTarget`, `deadline`, and `threshold` are initialized from `project` prop via `useState(...)` and never re-synced when `open` transitions to `true`. The timezone field has a `useEffect([open])` that re-fetches on every open, but the project fields do not. If the parent updates the `project` prop while the dialog is closed (e.g., via `handleProjectSettingsUpdate`), reopening the dialog shows stale values from the initial mount. The `key={project.slug}` on the dialog forces remount on slug changes but not on numeric/date field updates within the same project.
- **Impact:** After a successful settings save, closing and reopening the dialog could show stale field values (e.g., old word count target) until the component is remounted.
- **Suggested fix:** Add a `useEffect` that re-initializes `wordCountTarget`, `deadline`, and `threshold` from `project` prop when `open` becomes `true`.
- **Confidence:** Medium
- **Found by:** Logic & Correctness, Error Handling & Edge Cases

### [I4] `isSavingProjectTitleRef` not cleared on navigation-cancel effect
- **File:** `packages/client/src/hooks/useProjectTitleEditing.ts:22-28`
- **Bug:** The navigation-cancel effect sets `projectEscapePressedRef.current = true` and calls `setEditingProjectTitle(false)`, but does not reset `isSavingProjectTitleRef.current`. If a title save is in-flight (`isSavingProjectTitleRef.current = true`, line 49) when the user navigates to a different project, the ref stays `true`. On the new project, `saveProjectTitle()` immediately returns at line 40, silently refusing to save any title change.
- **Impact:** Silent failure -- the user can enter edit mode for the project title but saving does nothing. No error is displayed. Only a page refresh recovers.
- **Suggested fix:** Add `isSavingProjectTitleRef.current = false` to the navigation-cancel effect.
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling & Edge Cases, Concurrency & State
