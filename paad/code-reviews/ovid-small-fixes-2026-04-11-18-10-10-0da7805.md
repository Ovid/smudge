## Important Issues

### [I1] Lost update: concurrent timezone saves can revert a successful save
- **File:** `packages/client/src/components/ProjectSettingsDialog.tsx:156-168`
- **Bug:** `handleTimezoneChange` captures `const previous = timezone` before the optimistic update. If the user changes timezone A->B->C rapidly, three concurrent async calls run. If call B fails after call C already succeeded, B's catch block does `setTimezone(previous)` using B's captured `previous` (which is A), overwriting C's successfully-saved value. The UI now shows A while the server has C.
- **Impact:** User sees an incorrect timezone after a concurrent save failure. No indication the displayed value is wrong; the error banner they dismiss belonged to the failed request, not the successful one.
- **Suggested fix:** Track which value is currently being saved (e.g., `savingTimezoneRef.current = value` before the await). In the catch, only revert if `savingTimezoneRef.current === value` (no newer save has started). Alternatively, use an AbortController to cancel in-flight requests before starting new ones.
- **Confidence:** High
- **Found by:** Concurrency & State, Error Handling & Edge Cases

### [I2] `handleProjectSettingsUpdate` never clears `actionError` on success
- **File:** `packages/client/src/pages/EditorPage.tsx:204-226`
- **Bug:** The `.catch()` path sets `actionError` (line 223), but the `.then()` path does not call `setActionError(null)`. If a previous operation (trash, rename, status change) set an action error, then the user opens project settings, makes a change, and successfully saves, the stale error banner remains visible.
- **Impact:** Stale error banners erode trust. The user sees an error that no longer applies after a successful operation.
- **Suggested fix:** Add `setActionError(null)` inside the `.then()` callback, or at the top of `handleProjectSettingsUpdate` before the API call.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [I3] Stale error persists after successful create/delete on HomePage
- **File:** `packages/client/src/pages/HomePage.tsx:38-58`
- **Bug:** Neither `handleCreate` nor `handleDelete` calls `setError(null)` at the start. After a failed delete, if the user then successfully deletes a different project, the error from the first failure remains visible because the success path does not clear `error`. `handleCreate` navigates away on success (hiding the error), but `handleDelete`'s success path never touches `setError`.
- **Impact:** A stale error banner stays on screen after a successful operation, confusing the user.
- **Suggested fix:** Add `setError(null)` at the top of both `handleCreate` and `handleDelete`.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [I4] Raw `err.message` rendered in UI across 15+ catch sites
- **Files:** `packages/client/src/pages/HomePage.tsx:26,44,55`, `packages/client/src/hooks/useProjectEditor.ts` (multiple lines), `packages/client/src/components/DashboardView.tsx`, `packages/client/src/hooks/useTrashManager.ts`, and others
- **Bug:** The pattern `err instanceof Error ? err.message : STRINGS.error.loadFailed` is used in at least 15 catch blocks across the client. `err.message` comes from `ApiRequestError`, which populates its message from the raw server JSON response (`body.error?.message`). This means whatever the server puts in `error.message` is displayed verbatim. While the server redacts 500-class errors, 4xx messages pass through directly. This contradicts the CLAUDE.md convention of externalizing all UI strings in `strings.ts`.
- **Impact:** Potential information disclosure if server error messages ever include implementation details. Inconsistent with the project's string externalization rule. `EditorPage.tsx:222` already uses the correct pattern (`STRINGS.error.loadProjectFailed`) but is the exception, not the rule.
- **Suggested fix:** Replace all `err.message` instances in UI-facing code with the appropriate `STRINGS.error.*` constant. Consider creating a shared error handler utility.
- **Confidence:** High
- **Found by:** Security, Contract & Integration

## Suggestions

- **[S1]** `packages/client/src/pages/EditorPage.tsx:205` -- `handleProjectSettingsUpdate` bumps `dashboardRefreshKey` even from the empty-chapters path, where no dashboard is rendered. Harmless but semantically incorrect; causes a redundant dashboard API call when the user later opens the dashboard after changing settings from an empty project. *(Found by: Logic & Correctness)*

- **[S2]** `packages/client/src/pages/EditorPage.tsx:117` -- Chapter status retry uses `if (attempts < 2)`, giving 2 retries with 2s/4s delays. The auto-save spec in CLAUDE.md describes 3 retries with 2s/4s/8s backoff. This retry is for chapter statuses (not the save pipeline), so the spec may not strictly apply, but the gap is worth noting. *(Found by: Error Handling & Edge Cases)*

- **[S3]** `packages/client/src/pages/EditorPage.tsx:299-312 vs 462-475` and `315-329 vs 479-493` -- Identical `<Sidebar>` JSX (12 props, differing only in `activeChapterId`) and identical `actionError` banner are duplicated between the empty-chapters and main editor branches. Prior reviews already found a divergence bug from this duplication pattern. Extract to shared variables or components above the early returns. *(Found by: Contract & Integration)*

- **[S4]** `packages/client/src/components/ProjectSettingsDialog.tsx:109-130` -- Concurrent `saveField` calls share `setSaveError` state. Each call starts with `setSaveError(null)`, which can clear an error from a concurrent in-flight save before the user sees it. Low practical risk since settings changes are infrequent, but the pattern could surprise users on slow connections. *(Found by: Concurrency & State)*

- **[S5]** `packages/client/src/pages/EditorPage.tsx:136-169` -- Two `useEffect`s for velocity share refs (`hasFetchedInitial`, `lastVelocityFetch`) and rely on React's source-order execution guarantee within the same commit. This implicit ordering dependency is fragile under refactoring; consider merging into a single effect. *(Found by: Concurrency & State)*

- **[S6]** `packages/client/src/__tests__/ProjectSettingsDialog.test.tsx:26,44` -- `as never` casts bypass interface type-checking. If `ProjectSettingsDialogProps["project"]` changes, the test will compile but silently use mismatched fixtures. Use the actual prop type for the fixture. *(Found by: Contract & Integration)*
