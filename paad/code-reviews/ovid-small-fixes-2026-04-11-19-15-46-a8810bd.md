## Important Issues

### [I1] Empty-chapters branch missing ConfirmDialog and TrashView
- **File:** `packages/client/src/pages/EditorPage.tsx:274-353`
- **Bug:** The empty-chapters early-return renders `Sidebar` with `onDeleteChapter={setDeleteTarget}` and `onOpenTrash={openTrash}`, but neither `ConfirmDialog` nor `TrashView` appears in this branch. The main return path (lines 608-617, 496-502) renders both. Clicking "Trash" in the sidebar silently sets `trashOpen=true` with no visible feedback. If the user then creates a chapter (exiting the empty branch), the stale `trashOpen=true` causes `TrashView` to appear instead of the editor. Similarly, delete actions set `deleteTarget` but no confirmation dialog appears.
- **Impact:** Broken trash functionality on empty projects; unexpected TrashView appearance after chapter creation; delete button silently fails.
- **Suggested fix:** Add `ConfirmDialog` and `TrashView` to the empty-chapters branch, or consolidate into a single return block where the empty-chapters content is rendered within the shared layout shell.
- **Confidence:** High
- **Found by:** Logic & Correctness, Contract & Integration

### [I2] Velocity fetch throttle timestamp blocks retries on failure
- **File:** `packages/client/src/pages/EditorPage.tsx:150`
- **Bug:** `lastVelocityFetch.current = Date.now()` is set before the API call starts (line 150). The `.catch()` block (line 163) is empty (best-effort). If the velocity fetch fails (network error, 500), the throttle timestamp is already recorded, blocking any retry for `VELOCITY_THROTTLE_MS` (60 seconds). Previously, the timestamp was set inside `.then()`, so failures allowed immediate retries on the next save.
- **Impact:** Transient network or server errors suppress velocity data for 60 seconds. The user sees stale or missing session info in the status bar.
- **Suggested fix:** Move `lastVelocityFetch.current = Date.now()` into the `.then()` callback, or reset it to `0` in `.catch()`.
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling & Edge Cases

### [I3] handleTimezoneChange does not call onUpdate() on success
- **File:** `packages/client/src/components/ProjectSettingsDialog.tsx:157-173`
- **Bug:** When `handleTimezoneChange` successfully saves the timezone via `api.settings.update`, it does not call `onUpdate()`. Compare with `saveField()` (line 114), which calls `onUpdate()` after every successful project field save. The `onUpdate` callback refreshes the project data and increments `dashboardRefreshKey`. Since timezone affects velocity calculations (daily snapshots are timezone-dependent), the dashboard displays stale data computed with the old timezone.
- **Impact:** After changing timezone, the velocity/dashboard view shows data computed with the old timezone until the user manually triggers a refresh.
- **Suggested fix:** Add `onUpdate()` after the successful `await api.settings.update(...)` call.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [I4] Raw server error message rendered in save failure UI
- **File:** `packages/client/src/hooks/useProjectEditor.ts:110-111`
- **Bug:** `handleSave` sets `setSaveErrorMessage(lastError instanceof ApiRequestError ? lastError.message : STRINGS.editor.saveFailed)`. The `ApiRequestError.message` is sourced from the server's error response body (`body.error?.message` in `apiFetch`). All other error handlers in this diff were converted to use static `STRINGS` constants, but this one was left unchanged. If the server returns detailed error text (SQL constraint names, internal paths, etc.), it is rendered verbatim in the footer status area (EditorPage.tsx:596).
- **Impact:** Potential information disclosure via raw server error text. Inconsistent with the project's string externalization convention.
- **Suggested fix:** Replace with `setSaveErrorMessage(STRINGS.editor.saveFailed)`, or map known 4xx error codes to specific static strings.
- **Confidence:** High
- **Found by:** Security

### [I5] saveField reverts to stale project props on failure
- **File:** `packages/client/src/components/ProjectSettingsDialog.tsx:118-129`
- **Bug:** When `saveField` fails, it reverts fields using `project.target_word_count`, `project.target_deadline`, and `project.completion_threshold` from the component's props. After a successful save, `onUpdate()` is called which triggers a parent re-fetch, but the `project` prop may not have updated yet by the time a second save attempt fails. The revert then uses the original prop values rather than the last successfully saved values.
- **Impact:** After a successful settings change followed by a failed change, the field reverts to the original value rather than the last saved value, showing incorrect data.
- **Suggested fix:** Track the last successfully saved value for each field (e.g., in refs updated after `onUpdate()` completes), and revert to those on error.
- **Confidence:** Medium
- **Found by:** Logic & Correctness

## Suggestions

- **[S1]** `packages/client/src/components/ProjectSettingsDialog.tsx:111,162` -- Shared `saveError` state between project fields and timezone. Starting a timezone save clears a pending project-field error and vice versa, which can hide genuine failures from the user. Consider separate error states. *(Found by: Concurrency & State)*

- **[S2]** `packages/client/src/__tests__/EditorPageFeatures.test.tsx:14-55` -- Mock missing `api.settings`. The `ProjectSettingsDialog` component (now rendered inside `EditorPage`) calls `api.settings.get()` when opened. Any future test that opens the settings dialog will crash. Add `settings: { get: vi.fn().mockResolvedValue({ timezone: "UTC" }), update: vi.fn().mockResolvedValue({ message: "ok" }) }` to the mock. *(Found by: Contract & Integration)*

- **[S3]** `packages/client/src/components/ProjectSettingsDialog.tsx:157-173` -- If user changes timezone twice rapidly and both API calls fail, the UI reverts to the first (unpersisted) timezone value instead of the original server value. The `previous` variable captures optimistic state rather than the last confirmed server value. Low practical risk since rapid consecutive timezone failures are unlikely. *(Found by: Logic & Correctness, Error Handling & Edge Cases, Concurrency & State)*
