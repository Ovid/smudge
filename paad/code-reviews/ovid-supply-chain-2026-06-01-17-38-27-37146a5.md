## Important Issues

### [I1] Non-object cache file crashes the gate, violating the documented "must not crash" contract
- **File:** `scripts/dep-cooldown.mjs:120-128` (read) and `:141` / `:143` (use)
- **Bug:** The corrupt-cache `try/catch` only catches `JSON.parse` *throwing*. A
  `.dep-cooldown-cache.json` containing valid JSON that is **not a plain object** —
  `null`, a number, a string, `true` — parses successfully, so `cache` becomes that
  value. Then `group.some((g) => !(g.id in cache))` at line 141 throws
  `TypeError: Cannot use 'in' operator to search for '…' in null`. The throw is
  **outside any try/catch**, propagates out of `main()` (bare `await main()` at :198),
  and crashes with a raw stack trace. This directly contradicts the inline comment at
  lines 109-111: *"A corrupt/partial cache file … must not crash the gate — fall back
  to an empty cache and re-fetch."* (`[]` is benign — `'id' in []` is `false` — only
  `null`/number/string/boolean throw.)
- **Impact:** Fails **closed** (exit 1), so it is not a security hole — but it turns a
  recoverable cache-corruption case (a prior run killed mid-write, an `actions/cache`
  restore of an oddly-shaped file, a hand-edit) into a hard crash the developer cannot
  self-diagnose from the message, and it misclassifies an infra-class problem (bad cache)
  as a policy-violation exit code (1, not the infra exit 3).
- **Suggested fix:** After parsing, coerce to a plain object before use:
  ```js
  const parsed = readJson(CACHE_PATH);
  cache = parsed && typeof parsed === "object" && !Array.isArray(parsed) ? parsed : {};
  ```
  emitting the same "ignoring cache, will re-fetch" note otherwise. Consider extracting a
  tiny `sanitizeCache(parsed)` helper into `dep-cooldown-core.mjs` so it is unit-testable
  (matching the design's "logic in core, IO in shell" split). This same fix should also
  drop any cache *entry* whose value is not a string (see [S1]).
- **Confidence:** High (empirically reproduced; exit code confirmed)
- **Found by:** Logic & Correctness, Error Handling, Concurrency & State (3 specialists) + Verifier

### [I2] Degenerate / old-format lockfile silently passes the gate (fail-open)
- **File:** `scripts/dep-cooldown-core.mjs:108-135` (`collectRegistryVersions`) + `scripts/dep-cooldown.mjs:95,191-195`
- **Bug:** `collectRegistryVersions` does `const packages = lockfile.packages ?? {}` then
  `Object.entries(packages)`. For every degenerate-but-valid-JSON lockfile shape it returns
  `{ versions: [], skipped: 0 }`, and the shell treats zero versions as a clean pass —
  printing `Dependency cooldown: OK — all 0 checked version(s) …` and exiting **0**.
  Empirically confirmed for: `{}` (no `packages` key), an **npm v1/v2 lockfile** (uses
  `dependencies`, not `packages`), `{packages: 5}`, `{packages: []}`, and a bare primitive
  like `5`. The real Smudge lockfile is v3 with 892 `packages` entries, so "0 checked" is
  always an anomaly, yet nothing guards against it.
- **Impact:** **Fail-open** — the worst failure direction for a security gate. A lockfile
  that was corrupted, truncated-but-still-valid-JSON by a bad merge, regenerated under an
  old npm, or downgraded would silently *disable* the gate rather than failing closed. This
  defeats design decision #4 ("check evaluates the WHOLE lockfile every run"): a zero-count
  run reads nothing but reports success. Severity is held below Critical because it is not
  reachable in normal operation with a git-tracked v3 lockfile — it requires a corrupted
  commit, manual downgrade, or deliberate tamper.
- **Sub-case (lower severity):** a lockfile that parses to **`null`** crashes at
  `collectRegistryVersions` (`lockfile.packages` on `null`) at `dep-cooldown.mjs:95`, which
  is **outside** the read try/catch (that only wraps `readJson` at :88-94) → raw stack
  trace, exit 1. A primitive like `5` does *not* crash (it fails open at exit 0); only
  `null` crashes. (This matches Logic's conf-62 note.)
- **Suggested fix:** Fail closed when the parsed value is not a recognizable v3 `packages`
  lockfile, e.g. after parsing:
  ```js
  if (!lockfile || typeof lockfile !== "object" ||
      typeof lockfile.packages !== "object" || lockfile.packages === null ||
      Array.isArray(lockfile.packages)) {
    console.error(`✗ ${LOCKFILE_PATH}: not a recognizable npm v3 lockfile (missing/invalid "packages")`);
    process.exitCode = 1;
    return;
  }
  ```
  A core-level guard would be unit-testable. Optionally also treat `versions.length === 0`
  as fail-closed, since the real lockfile can never legitimately yield zero registry
  versions.
- **Confidence:** High (every shape reproduced empirically)
- **Found by:** Error Handling (primary), Logic & Correctness (null sub-case) + Verifier

## Suggestions

- **[S1]** `scripts/dep-cooldown.mjs:143` (+ `dep-cooldown-core.mjs:352-353`) — A cached
  value that is a **non-string** is passed unguarded into `classify`, where
  `Date.parse(published)` coerces it: a tampered cache entry of `0` or `1` becomes
  `Date.parse("0")` → year 2000 → reads as ~9600 days old → the young package **passes**
  (exit 0). This lives strictly *within* the already-accepted, documented S5 tampered-cache
  trust boundary (the shell only ever writes ISO strings), so it adds no new attack surface
  in *kind* — but a `typeof === "string"` guard makes forgery require a syntactically-valid
  ISO string rather than a bare number, and folds neatly into the [I1] cache-shape fix.
  (Found by: Error Handling; verified)
