## Important Issues

### [I1] Age-check identity (`name`/`version`) is decoupled from the `resolved` tarball — possible fail-open bypass
- **File:** `scripts/dep-cooldown-core.mjs:143-151` (`collectRegistryVersions`)
- **Bug:** The gate builds the identity it age-checks from the self-declared `entry.name` (preferred via the alias branch at line 143) and `entry.version` (line 148). `entry.resolved` is used only as a boolean gate (`isRegistryResolved`, line 144); the code never cross-checks that `name`/`version` agree with the `<name>/-/<name>-<version>.tgz` embedded in `resolved`. So the gate ages the identity the lockfile *declares*, not the identity of the tarball `npm ci` installs.
- **Impact:** An adversary who can influence `package-lock.json` (the gate's stated threat actor — a malicious transitive dep, a poisoned PR) could craft a self-consistent entry whose `name`/`version` name an *aged, innocent* package (passes the cooldown) while `resolved` points at a *young/malicious* tarball. The lockfile `integrity` hash binds to tarball **content**, not to the `name`/`version` fields, so it does not independently close this gap. The spec's threat model addresses tampered artifacts of a *pinned* version but not name/version decoupled from `resolved`. The alias-preference at line 143 is what widens the gap.
- **Honest caveat (why Important, not Critical):** Exploitability has an unverified precondition — whether `npm ci` actually installs a registry entry whose `resolved` filename disagrees with its key/name/version, or rejects the mismatch at install time. That could not be executed in-environment, so the end-to-end bypass is asserted, not proven. The **design-level** point stands regardless: a security gate should age the artifact it installs, not a sibling field an attacker controls independently.
- **Suggested fix:** Derive `name` and `version` from the `resolved` tarball path (scope-aware) and use those for the fetch + `time[version]` lookup; or assert they equal `entry.name`/`entry.version` and **fail closed** on mismatch. Keep the alias handling but validate `entry.name` against the `resolved`-derived name before trusting it.
- **Confidence:** Medium
- **Found by:** Security (`Opus 4.8 (1M context)`)

## Suggestions

- **[S1] Null per-entry value in `packages` crashes the scan with a bare stack trace, misrouted to the violation exit code.** `scripts/dep-cooldown-core.mjs:139` — `isV3Lockfile` (lines 109-113) validates only the top-level `packages` type, not each entry; a lockfile like `{"packages":{"node_modules/x":null}}` passes the guard then throws `Cannot read properties of null (reading 'link')`. `await main()` (`dep-cooldown.mjs:216`) has no try/catch, so Node prints a stack and exits **1** (the *violation* code), bypassing the curated unreadable-lockfile message the sibling guard at `dep-cooldown.mjs:102-108` deliberately emits. Fail-closed (CI still blocks), so this is hygiene, not a hole. Fix: skip non-object entries in `collectRegistryVersions` (`if (entry === null || typeof entry !== "object") { skipped++; continue; }`), or wrap `main()` to map unexpected throws to a curated message. Confidence Medium. (Error Handling, `Opus 4.8 (1M context)`)
- **[S2] Partial-group refetch can discard a known-good cached date and produce a spurious "absent" (blocking) violation.** `scripts/dep-cooldown.mjs:159-181` — `needFetch = group.some(g => !(g.id in cache))` forces a full fetch when *any* group member is uncached, then the post-fetch loop overwrites `publishDates` for *every* member, including cached ones, with `publishDateFromTime(times, g.version)` — which is null if the freshly-fetched `time` doc transiently lacks that exact version key. The good cached date is thrown away and the version becomes a false "absent" violation. Fail-closed (false block, never a silent pass) and self-healing (only positive dates persist to disk). Fix: `const iso = publishDateFromTime(times, g.version) ?? (g.id in cache ? cache[g.id] : null);`. Confidence Medium. (Logic & Correctness, `Opus 4.8 (1M context)`)
- **[S3] CI publish-time cache does not accumulate on unchanged-lockfile runs — the inline "ACCUMULATE" comment overstates the guarantee.** `.github/workflows/ci.yml:96-103` — `restore-keys: dep-cooldown-` is present and restore-side accumulation works, but `actions/cache@v4` skips the **save** when the exact `key` (`dep-cooldown-${hashFiles('package-lock.json')}`) had a restore hit. On the common stable-lockfile run the key hits → save is skipped → dates fetched that run are never persisted → the next run re-fetches them. Correctness/fail-closed unaffected (entries are immutable; re-fetch always corrects); cost is more cold fetches → more exposure to the exit-3 infra-failure path the cache exists to minimize. Fix: per-run-unique key (`...-${{ github.run_id }}`) keeping `restore-keys: dep-cooldown-`, or correct the comment to admit accumulation only happens on lockfile-change runs. (Folds in the benign cache-growth / cross-ref last-writer-wins / non-atomic local `writeFileSync` notes — all verified harmless given immutable entries + fail-closed semantics + `sanitizeCache`.) Confidence Medium. (Contract & Integration + Concurrency & State, `Opus 4.8 (1M context)`)
- **[S4] Spec promises a named constant `COOLDOWN_DAYS = 7` that the code never declares.** `scripts/dep-cooldown-core.mjs:221` — the value lives as the inline string default `"7"` inside `parseCooldownDays` (`Number(raw ?? "7")`); the spec (§Components, lines 110-113) names a `COOLDOWN_DAYS = 7` constant. Behavior (default 7, env-overridable) is preserved and arguably improved by the fail-closed validation, so this is doc-contract drift, not a behavior bug. Fix: either extract `const COOLDOWN_DAYS = 7` and use it as the default, or update the spec to ratify the parser. Confidence High. (Spec Compliance, `Opus 4.8 (1M context)`)
