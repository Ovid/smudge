## Critical Issues

### [C1] The gate trusts a cache file from the checked-out working tree; the `needFetch` short-circuit lets a committed cache skip the registry entirely (fail-open)
- **File:** `scripts/dep-cooldown.mjs:138-163` (cache read + `needFetch` short-circuit), with `.github/workflows/ci.yml:67-107` (CI job) and `.gitignore:168`
- **Bug:** The CI `dep-cooldown` job runs on `pull_request: branches: [main]` and checks out the PR tree via `actions/checkout@v4`. `main()` reads the publish-time cache from the repo-root `.dep-cooldown-cache.json` if it exists (`dep-cooldown.mjs:36, 138-140`) with no integrity/trust check, then short-circuits the registry fetch:
  ```js
  const needFetch = group.some((g) => !(g.id in cache));
  if (!needFetch) {
    for (const g of group) publishDates.set(g.id, cache[g.id] ?? null);
    continue; // never contacts the registry
  }
  ```
  `.gitignore` lists the cache file, but that does **not** prevent committing it via `git add -f`. A PR author can commit a `.dep-cooldown-cache.json` containing a forged old ISO date for their malicious `name@version` (and any sibling versions sharing that package name, so the whole group is cached). `needFetch` is then `false` for that group, the registry is never queried, `classify` ages the forged date as old (`core.mjs:505-515`), and the run exits 0 — a young/malicious package passes the gate. `sanitizeCache` does not help: it only drops non-string values and prototype keys (`core.mjs:447-457`); a well-formed forged date string survives intact.
- **Impact:** A complete bypass of the supply-chain control by anyone who can open a PR — the gate's stated threat actor. The committed cache file would appear as a diff (a reviewer *might* notice an unexpected `.dep-cooldown-cache.json`), but the entire point of the gate is automated defense that does not rely on a human spotting it.
- **Honest skepticism — why this may be less than fully exploitable in steady state:** In CI the `actions/cache` *restore* step (`ci.yml:100-104`) runs **after** checkout and, on a cache hit, extracts its tarball over the working tree — overwriting the attacker's committed file with the legitimately-accumulated `main` cache. That restored cache will **not** contain a brand-new malicious version's id (no prior run fetched it), so after the overwrite `needFetch` becomes `true`, the registry is queried, and the young package is caught. So on a well-established repo where `main` keeps a warm cache, the attack is defeated *incidentally*. The reliable exploit window is a **cold cache**: the first run before any `dep-cooldown-*` cache exists, after the 7-day no-access eviction, or after the 10 GB-repo eviction — and the `make dep-cooldown` local path, which has no `actions/cache` at all and reads the file directly. The reason this stays **Critical** rather than Important: a security control must not depend on an incidental, unreliable side-effect of an unrelated caching action for its integrity. The code path itself trusts attacker-controllable checked-out data with zero verification — a textbook fail-open — and the fix is cheap.
- **Suggested fix:** Make the defense explicit and not dependent on cache-restore ordering. Either (a) in the CI job, `rm -f .dep-cooldown-cache.json` immediately after checkout and **before** the `actions/cache` restore, and/or (b) have `dep-cooldown.mjs` refuse to start if `.dep-cooldown-cache.json` is git-tracked (e.g. `git ls-files --error-unmatch`), and/or (c) drop the `needFetch` short-circuit so the registry is always queried for a fresh date and the on-disk cache is used **only** as the documented S2 *fallback* when a live `time` doc omits a version — then a forged cache entry can never author a date, only (harmlessly) rescue an already-fetched real one. Option (b)+(c) is the most robust. Update the CLAUDE.md / CI trust-boundary notes to add the committed-file vector.
- **Confidence:** High (the code path is confirmed; the residual is exploitability reliability, which is cold-cache-dependent)
- **Found by:** Security; corroborated by Concurrency & State (cache-trust angle)

## Suggestions

- **[S1] Unhandled throw past `await main()` exits code 1 (the *violation* code), not the curated infra exit 3.** `scripts/dep-cooldown.mjs:226` is a bare `await main()` with no outer try/catch; `collectRegistryVersions` (109), `groupVersionsByName` (151), `classify` (194), `buildReport` (201) are unwrapped. An unexpected throw there exits Node with code 1 and a raw stack trace — indistinguishable from a real policy violation — instead of the distinct exit-3 "registry unreachable, re-run" message the design promises for infra-class failures. Fail-closed (still blocks the merge); diagnostics only. Fix: wrap `main()`'s body (or the call) in a try/catch that maps an otherwise-unhandled error to a curated message + `process.exitCode = 3`. Confidence 90. (Error Handling)
- **[S2] `tarballMatchesIdentity` false-mismatch on a percent-encoded scoped `resolved` URL → spurious block (availability, not security).** `scripts/dep-cooldown-core.mjs:114` compares `url.pathname` (which WHATWG `URL` leaves percent-encoded — `%40`/`%2F` are not decoded) against an expected path built from the *unencoded* name. A non-canonical/hand-edited lockfile whose scoped `resolved` is `/%40types%2Fnode/-/...` would be diverted to `mismatched` and block a legitimate dependency. Low likelihood — npm's canonical lockfile form is unencoded. Fix: normalize both sides (e.g. compare against `new URL(expected, REGISTRY).pathname`, or also accept a `decodeURIComponent(url.pathname)` form). Confidence 70. (Logic & Correctness)
- **[S3] CI cache "per-branch scoping protects main" comment overstates GitHub's guarantee.** `.github/workflows/ci.yml:90-99` presents per-branch cache scoping as the security justification for trusting cached dates as-is. The direction is roughly right (fork PRs get read-only restricted scope; reads are limited to the current ref + base/default branch), but the bare `restore-keys: dep-cooldown-` prefix maximizes restore surface within the allowed scopes, and the comment reads as a hard guarantee. Largely moot relative to C1 (the committed-file vector bypasses the cache mechanism entirely). Fix: soften the comment to the actual scoping rules and lean on the `npm ci` integrity-hash pillar. Confidence 65. (Concurrency & State)
- **[S4] `entry.version` is truthy-checked but never type-checked as a string (asymmetric with `name`).** `scripts/dep-cooldown-core.mjs:204` — `name` is `typeof === "string"` guarded (line 203) but `version` only `!entry.version`. A non-string truthy version (`5`, `true`, `{}`) flows into `versionId`/`tarballMatchesIdentity`. Fail-closed today (stringified version won't match the canonical tarball path → diverted to `mismatched` → block), but the safety is a stringification coincidence and the asymmetry could trip a future refactor. Fix: `if (typeof entry.version !== "string" || !entry.version || !isRegistryResolved(entry.resolved))`. Confidence 80. (Error Handling / Security)
- **[S5] `resolvePublishDate` cache fallback can resurrect a yanked version during a partial-group refetch (residual of S2).** `scripts/dep-cooldown-core.mjs:427-428` returns `publishDateFromTime(times, version) ?? cache[id] ?? null`; an already-cached version the fresh registry doc no longer lists (a genuine yank/unpublish) falls back to the old cached date and passes instead of being flagged "absent". The S2 justification (transient single-key omission) is weakened because `fetchPublishTimes` already throws when the whole `time` object is missing/malformed (`core.mjs:372-379`), so a successful return implies a usable doc — a missing single key is more likely a real yank. Narrow window (a sibling version must force the refetch). Deliberate, documented tradeoff; flagging for confirmation. Confidence 62. (Logic & Correctness)
- **[S6] `parseAllowlist` reads `e.reason` without re-guarding that `e` is an object.** `scripts/dep-cooldown-core.mjs:270` — safe only because the `!pkg`/`!version` throws (266-269) fire first for a null/non-object `e`. A fragile ordering dependency: reorder or relax those guards and `e.reason` on a primitive throws a raw `TypeError` instead of the curated message. Fix: guard `e` once at the top of the loop body. Confidence 75. (Contract & Integration)
- **[S7] The "skipped … (git/file/link)" report wording does not match what is actually counted.** `scripts/dep-cooldown-core.mjs:582-584` — `link: true` entries are `continue`d **without** incrementing `skipped` (line 199), while malformed/non-object entries (191-194) and registry entries on a non-`registry.npmjs.org` host (204-207) **are** counted. So the message names a category that isn't counted (link) and omits ones that are. Purely cosmetic — the count never blocks. Fix: reword to "non-registry / unrecognized entries" or align the enumeration. Confidence 70. (Plan Alignment)
