## Notes (informational — not bugs, not introduced by this PR)

- **Search match-cap test message ambiguity.** `search.service.test.ts:1060` asserts `/too many matches/i`. Two distinct code paths (`MatchCapExceededError` from the matcher vs. the cross-chapter running-total check at `search.service.ts:179-180`) emit the *same* message+code. The test mocks `searchInDoc` to *return* 6000 matches (not throw), so it does correctly drive the running-total path it names — but because the two paths are textually indistinguishable, a future regression that flipped which path fires would not be caught. Below the confidence bar; noted for awareness only. (Found by: Error-Handling, Logic)
- **F-4 cross-project image bleed remains open (pre-existing).** The new EPUB tests add coverage for the *cover* ownership guard (`epub.renderer.ts:87`, `row.project_id === project.id`), but the inline-image resolvers `resolveImage`/`resolveImagesForEpub` (`packages/server/src/export/image-resolver.ts:33,160`) still call `findImageById(id)` with no `project_id` check. So the added coverage is meaningful for the cover path but does **not** close the F-4 cross-project inline-image bleed, and no test asserts cross-project exclusion. This is a pre-existing code gap (documented as F-4 in the same architecture report), not introduced by this test PR. Out of scope for a test-coverage PR; flagged so the coverage's boundaries are clear. (Found by: Security)
